# 제목

## 1. 프로젝트 한 줄 요약

- MQTT 기반 센서 데이터를 수집하여 농가 환경을 실시간으로 모니터링하는 스마트팜 관리 시스템

---

## 2. 프로젝트 배경

- 기존 회사에서 사용하던 스마트 공장 및 스마트 팩토리를 관리하던 사이트와는 별도로 스마트팜만을 관리하는 사이트의 필요성 대두.
- 사용자 친화적이며, 모바일 대응이 가능한 관리 사이트의 필요.

---

## 3. 나의 역할 (책임 범위)

- 프론트엔드 전반 설계 및 구현
- 백엔드 API 및 DB 구조 설계
- MQTT 기반 센서 데이터 수집·저장 및 웹 제어 API 연동 로직 설계
- Docker 기반 배포 환경 구성 및 운영

---

## 4. 핵심 기술 스택 (대표 3~5개)

- Frontend: Next.js, TypeScript
- Backend: Spring Boot
- DB: PostgreSQL
- Infra: Docker, Nginx
- ETC: MQTT

---

## 6. 문제 상황 & 해결 경험

- 문제: 백엔드 재배포 시 알림이 중복 발송되는 문제 발생  
  - 원인: 메모리 기반 상태 관리로 인해 재기동 시 상태 초기화  
  - 해결: 알림 발송 상태를 DB에 저장하도록 구조 개선  
  - 결과: 문자 발송 비용 감소 및 알림 안정성 확보

- 문제: MQTT 데이터 수집 시 웹 제어 API 응답 지연 발생  
  - 대안 검토: 동일 프로세스 내 비동기 처리 vs 백엔드 분리
  - 판단 이유: 웹 제어 API의 응답 안정성을 최우선으로 고려
  - 원인: MQTT 데이터 수집 로직이 동기적으로 처리되어 웹 제어 API가 후순위로 밀림  
  - 해결: MQTT 데이터 수집 전용 백엔드를 분리하여 비동기 처리  
  - 결과: 웹 제어 API 응답 속도 개선 및 시스템 책임 분리

- 문제: 실시간 데이터가 아닌 과거 데이터가 우선 표시되는 현상  
  - 원인: MQTT retain 설정으로 인해 이전 데이터가 우선 수신  
  - 해결: 일정 시간 이상 차이가 나는 데이터는 비동기 저장 처리  
  - 결과: 실시간 데이터 정상 표시

- 문제: HMI 기준의 센서 설정 값이 사용자에게 그대로 노출되어 혼란 발생
  - 원인: HMI에 적용되는 형식이 그대로 사용자에게 전달되어 이해하기 어려움.
  - 해결: 사용자 실수를 줄이기 위해 입력 단위 검증, 가이드 문구, 기본값 설계 적용
  - 결과: 사용자 경험 개선 및 사용자 이해도 향상

---

## 7. 결과 및 성과

- 실제 스마트팜 운영을 목표로 시스템을 설계·구현하며,
  MQTT 기반 데이터 수집, 알림, 웹 제어 영역에서 발생 가능한 주요 운영 리스크를 사전에 식별하고 구조적으로 대응

- 현장별로 상이한 센서 구성과 데이터 포맷이 표준화되어 있지 않은 환경에서,
  단일 시스템으로의 즉각적인 실운영보다는
  구조적 안정성과 확장 가능성 검증에 우선순위를 두고 개발 진행

- 센서 표준화 및 운영 기준 합의가 선행되어야
  실운영 단계로 자연스럽게 확장될 수 있음을 기술적으로 검증

---

## 8. 아쉬운 점 & 개선 아이디어 (선택)

- 실시간 시스템에서는 동기/비동기 경계를 명확히 나누지 않으면
  사용자 제어 API 안정성에 직접적인 영향을 준다는 점을 체감
- 향후 앱 형태로 확장 시 사용자 경험 개선 가능성 인식
