## 1. 프로젝트 한 줄 요약

- Spring Boot 3.2 + Vue.js 2.7 기반으로 중소 식품 제조기업의 자재·제품 입출고 및 재고 관리 프로세스를 전산화하고, 실사용자 피드백을 반영해 안정화한 풀스택 내부 관리 시스템

---

## 2. 프로젝트 배경

- 기존에는 입출고·재고·생산·주문 이력을 수기 대장으로 관리하여 데이터 누락과 이전 자료 추적이 어려웠다.
- 2인 개발 팀에서 사수의 기존 백엔드 코드베이스를 분석하여 일관성을 유지하면서, 신규 도메인 API 및 화면 전체를 단독 설계·구현해야 하는 상황이었다.

---

## 3. 나의 역할 (책임 범위)

**백엔드**
- `입출고(io)` · `원부자재(matr)` · `제품/BOM(prod)` · `주문(ordr)` 도메인의 레이어드 아키텍처(Controller → Service → Repository/Mapper) 전 구간 설계 및 구현
- JPA(Hibernate 6.4.4)와 MyBatis 3.5.15 혼용 전략 직접 결정: 단순 엔티티 CRUD는 JPA Repository, 다중 테이블 조인·집계 쿼리는 MyBatis XML Mapper로 역할 분리
- `JwtInterceptor` 기반 JWT 인증 체계 설계 — 로그인/회원가입 경로를 제외한 모든 API 요청에 토큰 검증 적용
- `@BasicLog` 커스텀 어노테이션 + `LogAspect`(AOP) 구조로 비즈니스 코드와 로깅 로직 분리
- WebSocket(STOMP) + SockJS 기반 실시간 알림 서비스(`NotificationService`) 구현 — 재고 변동·주문 상태 변경 시 연결된 모든 클라이언트에 브로드캐스트

**프론트엔드**
- Vue.js 2.7 + Vuex + Vue Router SPA 전체 화면 구현 (7개 통계 뷰 포함 30여 개 페이지)
- AG Grid Enterprise 31.0을 활용한 입출고·통계 그리드 구현 — 그룹핑, 필터링, Excel 내보내기 기능 적용
- Chart.js 2.9 기반 재고·생산·주문 현황 시각화 컴포넌트 구현
- SockJS + webstomp-client로 백엔드 STOMP 엔드포인트 구독, 실시간 알림 수신 UI 연동
- Apache POI와 연동하는 Excel 대량 업로드 진행률 UI(`PopProgress.vue`) 구현

**인프라**
- Nginx를 이용한 Vue SPA 정적 빌드 서빙 및 Spring Boot API 리버스 프록시 구성 — 모든 경로를 `index.html`로 fallback 처리하는 SPA 라우팅 설정 포함

## 5. 기술적 판단 & 설계 포인트 ⭐

### 5-1. JPA와 MyBatis 혼용 전략 — 쿼리 복잡도 기준으로 도구 분리

MES 도메인은 단순 CRUD(거래처 등록, 원부자재 추가 등)와 복잡한 집계 쿼리(월별 입출고 추이, BOM 기준 소요 원료 계산, 주문-출고 매핑)가 공존한다. 모든 쿼리를 JPA JPQL로 처리하면 `@Query`가 장황해지고 동적 조건 처리가 까다로워진다. 반대로 모든 쿼리를 MyBatis로 작성하면 단순 save/findById에도 XML을 작성하는 비용이 생긴다.

**결정**: 엔티티 생명주기를 관리하는 단순 조작은 JPA Repository(`MatrRepository`, `ProdBomRepository` 등), 다중 테이블 조인·집계·동적 조건이 필요한 통계 쿼리는 MyBatis XML Mapper(`IoMapper.xml`, `ProdMapper.xml`, `OrdrMapper.xml`)로 역할을 명확히 나눴다.

**효과**: 서비스 레이어에서 `IoService`는 단건 입출고 저장에는 JPA를 호출하고, 기간별 집계 조회에는 `IoMapper`를 호출하는 구조가 되어, 쿼리 유지보수 범위가 명확하게 분리됐다.

### 5-2. JWT + JwtInterceptor 패턴 — 무상태 인증으로 API 서버 분리 대응

내부 시스템이지만 Spring Boot API 서버와 Vue SPA가 완전히 분리된 구조이므로 세션 기반 인증은 적합하지 않았다. 서버가 세션 상태를 유지하면 서버 재시작 시 모든 로그인 상태가 초기화되고, 수평 확장 시에도 세션 공유 문제가 생긴다.

**결정**: 로그인 성공 시 JWT 토큰을 발급하고, `JwtInterceptor`가 로그인·회원가입 경로를 제외한 모든 API 요청 헤더의 `Authorization: Bearer {token}`을 검증하도록 `WebConfig`에서 인터셉터를 등록했다. 비밀번호는 BCrypt로 해시 저장하여 평문 노출을 방지했다.

**효과**: API 서버 재시작 이후에도 클라이언트의 토큰 기반 재인증이 가능하고, 프론트엔드에서 Axios 인스턴스에 요청 인터셉터를 추가하는 것만으로 모든 API 호출에 토큰이 자동 첨부된다.

### 5-3. WebSocket(STOMP) 실시간 알림 — 폴링 없이 재고 변동 즉시 전파

재고 변동(입출고 등록)이나 주문 상태 변경은 담당자 외 다른 사용자 화면에도 즉시 반영되어야 한다. 폴링으로 구현하면 불필요한 API 요청이 반복되고, 갱신 간격만큼 정보 지연이 생긴다.

**결정**: 백엔드에 STOMP 브로커를 구성하고(`WebSocketConfig`), 입출고/주문 서비스에서 데이터 변경 후 `NotificationService.broadcast()`를 호출해 `/topic/notifications` 채널로 이벤트를 전송한다. 프론트엔드는 컴포넌트 마운트 시 SockJS + webstomp-client로 구독하여 알림을 수신 즉시 UI에 반영한다.

**효과**: 한 사용자가 입고 처리를 완료하면, 같은 화면을 열어 둔 다른 사용자에게 별도 새로고침 없이 변경 내역이 전달된다.

### 5-4. AOP 기반 공통 로깅 — 비즈니스 로직과 로깅 로직의 완전 분리

서비스 메서드마다 `logger.info()`를 수동으로 삽입하면 비즈니스 코드가 로깅 코드로 오염되고, 로깅 포맷 변경 시 전체 서비스를 수정해야 한다.

**결정**: `@BasicLog` 커스텀 어노테이션을 정의하고, `LogAspect`의 `@Around` advice에서 어노테이션이 붙은 메서드의 호출 전후를 가로채 요청 파라미터·실행 시간·반환값을 일관된 포맷으로 기록하도록 설계했다.

**효과**: 서비스 코드는 `@BasicLog`만 선언하면 자동으로 로그가 남고, 로깅 포맷 변경은 `LogAspect` 한 곳만 수정하면 전체에 반영된다.

## 6. 문제 상황 & 해결 경험

### 통합 현황 화면 — 복잡한 다중 테이블 조인을 도메인 단위 API로 분리

**현상**

배포 초기, 사용자들이 "입출고·생산·주문 현황을 한 화면에서 한눈에 보고 싶다"는 피드백을 줬다. 이를 단일 화면에서 구현하려 했더니, 아래 테이블들을 동시에 조인해야 하는 쿼리가 필요했다.

| 테이블 | 역할 |
|---|---|
| `matr_iwh` / `matr_owh` | 원부자재 입·출고 내역 |
| `prod_iwh` / `prod_owh` | 제품 생산(입고) · 출고 내역 |
| `prod_bom` | 제품 BOM (생산 시 소요 원료 정의) |
| `ordr_info` / `ordr_prod` | 주문 헤더 · 주문 제품 상세 |
| `matr_info` / `prod_info` | 원부자재 · 제품 마스터 |

이 10개 이상의 테이블을 단일 쿼리에서 처리하면 SQL이 수백 줄에 달하고, 조인 조건 하나의 오류가 전체 집계를 오염시키는 구조가 된다. 실제로 초기 구현에서 BOM 소요량 계산 쿼리가 생산 내역과 잘못 조인되어 재고 수치가 틀리게 표시되는 장애가 발생했다.

**원인 분석**

- 데이터 도메인(원부자재 입출고, 제품 생산, 주문)이 본질적으로 독립적인 흐름임에도, 이를 하나의 쿼리로 통합하려 한 것이 근본 원인이었다.
- 사용자가 "한 화면에서 보고 싶다"는 요구사항은 실제로는 "각 업무 목적에 맞게 빠르게 조회하고 싶다"는 의미였고, 하나의 쿼리로 모든 데이터를 합산하는 것과는 다른 요구임을 뒤늦게 파악했다.

**백엔드 대응**

- 집계 목적별로 MyBatis XML Mapper를 도메인 단위로 분리했다: `IoMapper.xml`은 입출고 이력, `ProdMapper.xml`은 생산 현황, `OrdrMapper.xml`은 주문·납품 현황을 각각 독립적으로 처리한다.
- 통계 API를 `/api/stats/matr/in`, `/api/stats/prod/in`, `/api/stats/warehouse` 등으로 책임 단위별로 분리 설계하여, 각 쿼리가 단일 도메인에만 집중하도록 했다.
- MyBatis XML의 `<if>`, `<choose>` 태그로 일별·월별·연간 조회 조건을 동적으로 처리 — 조건마다 별도 쿼리를 만들지 않고 단일 Mapper에서 동적 SQL로 커버했다.

**프론트엔드 대응**

- `80_stats` 모듈을 7개의 독립 뷰로 분리: `InMatrStat`, `OutMatrStat`, `MatrStat`, `InProdStat`, `OutProdStat`, `ProdStat`, `OrdrStat`.
- 각 뷰에서 AG Grid Enterprise의 그룹핑·필터링 기능을 활용해 사용자가 날짜·거래처·품목 기준으로 직접 데이터를 재구성할 수 있게 했다. 이를 통해 "여러 조건 조합"마다 별도 API를 만들지 않고도 사용자 요구를 충족할 수 있었다.
- `SearchPeriodWithSelect` 공통 컴포넌트로 모든 통계 뷰에서 기간 조건을 일관된 방식으로 처리했다.

**결과**

- 쿼리 오류 범위가 도메인 단위로 좁혀져 장애 발생 시 원인을 빠르게 특정할 수 있게 됐다.
- 각 통계 API의 응답 시간이 단일 통합 쿼리 대비 명확하게 단축됐다 (다중 조인 제거).
- 사용자는 자신의 업무 목적에 맞는 뷰로 직접 이동하는 방식에 만족도가 높아졌고, 이후 유지보수 요청이 감소했다.

※ 엑셀 업로드 오류·UX 개선 등 운영 중 추가 안정화 작업도 병행했다.

## 8. 아쉬운 점 & 개선 아이디어

- 초기 도입 특성상 단일 서버 구조로 배포했다. 트래픽 증가 시 API 서버와 정적 파일 서버를 분리하고, WebSocket 브로커를 Redis Pub/Sub으로 교체하면 수평 확장이 가능한 구조가 된다.
- 현재 배치 스케줄러(`Scheduler.java`)가 단순 구조이므로, 정기 재고 마감 집계 같은 작업은 Spring Batch로 확장하면 실패 재시도와 이력 관리가 용이해진다.
- Vue 2.7 EOL(2023.12)을 고려하여, 신규 기능 추가 시에는 Vue 3 + Composition API 마이그레이션을 우선 검토할 필요가 있다.

## 9. 이 프로젝트에서 강조하고 싶은 한 가지

- 백엔드 아키텍처(JPA·MyBatis 혼용, JWT 인증, AOP 로깅, WebSocket STOMP) 설계부터 Vue SPA 화면 구현, Nginx 배포 환경 구성까지 풀스택 파이프라인 전체를 2인 팀에서 주도적으로 수행한 프로젝트. 단순 구현에 그치지 않고 복잡한 도메인 쿼리 구조를 직접 분석·재설계하여 운영 장애를 해결한 경험이 핵심이다.

> 개발 기간: 2024.01 ~ 2024.03 (3개월) / 2인 팀 / 풀스택

- Spring Boot 3.2 + Vue.js 2.7 기반으로 원부자재·제품 입출고, BOM 생산 관리, 주문·납품 관리 전 도메인의 백엔드 API 및 프론트엔드 화면을 단독 설계·구현
- JPA(단순 CRUD)와 MyBatis(복잡한 조인·집계 쿼리) 혼용 전략을 직접 결정하여 도메인별 쿼리 유지보수 범위 명확화
- JWT + JwtInterceptor 인증 체계, AOP 기반 공통 로깅(`@BasicLog`), WebSocket(STOMP) 실시간 재고·주문 알림 서비스 설계 및 구현
- 단일 통합 쿼리로 인한 집계 오류를 도메인 단위 API 분리와 AG Grid 기반 클라이언트 그룹핑으로 해결 — 유지보수 요청 빈도 감소 및 사용자 만족도 향상
- Nginx 기반 SPA 배포 환경(정적 파일 서빙 + API 리버스 프록시 + fallback 라우팅) 구성
