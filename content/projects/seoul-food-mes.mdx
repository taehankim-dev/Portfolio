> **프로젝트 개요**
>
> 재입사 후 첫 프로젝트로, Spring Boot 3.2 + Vue.js 2.7 기반으로 중소 식품 제조기업의 자재·제품 입출고 및 재고 관리 프로세스를 전산화한 풀스택 내부 관리 시스템입니다. 2인 개발 환경에서 동료의 기존 코드베이스를 분석하며 백엔드 API 전 구간 및 프론트엔드 SPA 화면 전반을 주도적으로 설계·구현했으며, 다중 테이블 집계 장애를 도메인 분리 아키텍처로 해결하여 실사용 환경에서 안정화했습니다.
>
> **프로젝트 성격**: 회사 레거시 관행을 존중하면서 풀스택 파이프라인 전체를 책임지고, 발생한 결합도 장애를 도메인 분리 API와 MyBatis 동적 SQL로 영리하게 해결한 내부 관리 시스템

## 문제 상황 & 해결

### 1. 통합 현황 화면 — 다중 테이블 단일 조인을 도메인 단위 API로 분리

**현상**

배포 초기, 사용자들이 "입출고·생산·주문 현황을 한 화면에서 한눈에 보고 싶다"는 피드백을 전달했다. 이를 단일 화면에서 구현하려 하자, 아래 데이터 도메인들을 단일 쿼리에서 동시에 조인해야 하는 상황이 됐다.

| 데이터 도메인 | 역할 |
|---|---|
| 원부자재 입고 내역 / 원부자재 출고 내역 | 원부자재 입·출고 이력 집계 |
| 제품 생산 입고 내역 / 제품 출고 내역 | 제품 생산(입고) · 출고 이력 집계 |
| 제품 BOM 소요 원료 정의 | 생산 시 소비되는 원부자재 소요량 |
| 주문 헤더 / 주문 제품 상세 | 고객 주문 정보 및 납품 상세 |
| 원부자재 마스터 / 제품 마스터 | 원부자재·제품 기준 정보 |

10개 이상의 데이터 도메인을 단일 쿼리에서 처리하면 SQL이 수백 줄에 달하고, 조인 조건 하나의 오류가 전체 집계를 오염시키는 구조가 된다. 실제로 초기 구현에서 BOM 소요량 계산 쿼리가 생산 내역과 잘못 조인되어 재고 수치가 틀리게 표시되는 심각한 장애가 발생했다.

**원인**:
- 원부자재 입출고·제품 생산·주문이라는 각 데이터 도메인이 본질적으로 독립적인 흐름임에도 이를 하나의 집계 쿼리로 통합하려 한 것이 근본 원인
- 사용자의 "한 화면에서 보고 싶다"는 요구사항을 분석하면, 실제 의미는 "각 업무 목적에 맞게 빠르게 조회하고 싶다"는 것이었다 — 단일 쿼리 통합이 아닌 도메인 단위 독립 조회 API로 요구사항을 재정의해야 하는 설계 문제임을 파악했다.

**해결 과정**:

*백엔드*: 집계 목적별로 MyBatis XML Mapper를 도메인 단위로 완전히 분리했다. 입출고 이력, 제품 생산 현황, 주문·납품 현황을 각각 전담하는 독립 Mapper로 재구성하고, 통계 API 엔드포인트도 책임 단위별로 설계하여 각 쿼리가 단일 도메인에만 집중하도록 했다. 일별·월별·연간 조회 조건은 MyBatis의 `<if>`, `<choose>` 동적 SQL 태그로 처리하여, 조건 조합마다 별도 쿼리를 파편화하지 않고 단일 Mapper 내에서 동적으로 커버했다.

*프론트엔드*: 통합 현황 화면을 7개의 독립 전용 통계 뷰로 분할했다. 각 뷰는 AG Grid로 데이터 테이블을 렌더링하며, 공통 조건 컴포넌트에서 날짜 범위·조회 조건을 입력받아 해당 도메인의 통계 API를 호출하는 방식으로 연결했다. AG Grid 내에서 사용자가 직접 데이터를 재조작하는 방식은 사용자 복잡도를 높인다고 판단하여 적용하지 않았다. 검색 조건을 백엔드로 전달하고 MyBatis 동적 쿼리에서 처리하여, 사용자는 조건 선택만으로 필요한 데이터를 명확하게 조회할 수 있게 했다.

**결과**:
- 쿼리 오류 범위가 도메인 단위로 좁혀져 장애 발생 시 원인을 빠르게 특정할 수 있게 됐다.
- 각 통계 API가 단일 도메인에 집중하면서 쿼리 유지보수가 용이해졌다.
- 사용자는 업무 목적에 맞는 뷰로 직접 이동하는 방식에 만족도가 높아졌고, 이후 유지보수 요청이 감소했다.

## 기술적 판단 & 설계 포인트

### 1. JPA와 MyBatis 혼용 — 기존 관행 준수 속 점진적 도입

재입사 후 첫 프로젝트였기에, 기존 코드베이스 전체가 MyBatis 중심으로 구성되어 있다는 사실을 먼저 파악하고 이를 따르는 방향으로 설계를 시작했다. 동료의 관행을 유지하면서 단순 엔티티 CRUD에 한해 JPA Repository를 점진적으로 추가하는 식으로 혼용 구조가 형성됐다.

| 영역 | 선택 기술 | 이유 |
|---|---|---|
| 단순 엔티티 CRUD | JPA Repository | 반복성 낮은 표준 저장소 인터페이스로 충분 |
| 복잡한 조인·집계·동적 조건 | MyBatis XML Mapper | 세밀한 SQL 제어, 기존 코드베이스 호환 |

프로젝트 완료 이후 JPA만으로도 동일 도메인을 설계할 수 있다는 것을 인식했고, 이는 이후 프로젝트에서 아키텍처를 다시 설계하는 기준점이 됐다.

### 2. 세션 + 쿠키 인증 — 회사 표준 적용과 현실적 판단

인증 방식으로는 회사가 기존부터 사용해 온 세션 + 쿠키 방식을 적용했다. JWT 기반 인증을 직접 구현해보려 시도했으나, 3개월이라는 짧은 기한과 당시 구현 난이도를 고려했을 때 프로젝트 내에서 안정적으로 완성하기 어렵다는 판단을 내렸다. 비밀번호는 BCrypt로 해시 저장하여 평문 노출을 방지했다.

기존 코드베이스에는 JWT 관련 인터셉터 코드가 잔재해 있었으나, 이는 동료가 이전 프로젝트 코드를 가져온 흔적으로 실제 인증 흐름에는 적용되지 않은 상태임을 개발 과정에서 파악했다. 미사용 코드와 실제 동작 흐름을 구분하여 혼선 없이 세션 인증 구조를 완성했다.

### 3. WebSocket(STOMP) 실시간 알림 — 폴링 없이 재고 변동 즉시 전파

재고 변동(입출고 등록)이나 주문 상태 변경은 담당자 외 다른 사용자 화면에도 즉시 반영되어야 한다. 폴링으로 구현하면 불필요한 API 요청이 반복되고, 갱신 주기만큼 정보 지연이 구조적으로 발생한다.

STOMP 브로커 채널을 구성하고, 입출고·주문 서비스에서 데이터 변경 후 알림 브로드캐스트 서비스를 호출해 구독 채널로 이벤트를 전송하도록 설계했다. 프론트엔드는 컴포넌트 마운트 시 SockJS + webstomp-client로 구독하여 알림을 수신 즉시 UI에 반영한다.

> 한 사용자가 입고 처리를 완료하면, 동일 화면을 열어 둔 다른 사용자에게 별도 새로고침 없이 변경 내역이 즉시 전달된다.

### 4. exceljs 선택 — 사용자 환경에 맞춘 클라이언트 사이드 처리

엑셀 업로드 처리를 서버가 아닌 프론트엔드 라이브러리(exceljs)로 구현했다. 주요 사용자가 1인이고 업로드 파일 규모가 크지 않아, 서버 왕복 없이 클라이언트에서 직접 파싱하는 방식이 응답 속도 면에서 더 유리했다. 서버 자원을 소비하지 않는다는 부수적인 이점도 있었다.

## 회고

### 잘한 점
- **도메인 분리 아키텍처**: 단일 집계 쿼리의 결합도 장애를 직접 경험하고, Mapper·API·화면을 도메인 단위로 재구성하여 유지보수성과 집계 정확성을 동시에 확보했다.
- **현실적 기술 판단**: 기한(3개월)과 숙련도를 감안해 JWT 대신 세션 인증을 선택하고, 기존 관행(MyBatis)을 준수하면서 JPA를 점진적으로 도입하는 균형감 있는 결정을 내렸다.
- **사용자 요구사항 재정의**: "한 화면에서 다 보고 싶다"는 피드백을 단일 쿼리 통합이 아닌 도메인별 분리 조회라는 올바른 아키텍처 해법으로 번역하고, 7개 독립 뷰로 구현했다.

### 아쉬운 점
- 개발·검증 단계에서 로그 출력 기반 수동 E2E 방식으로 API 동작을 확인했다. 초기에는 이 방식으로 충분하다고 판단했으나, 도메인 수가 늘어나고 쿼리 복잡도가 높아질수록 회귀 오류를 수동으로 잡아내는 비용이 급격히 커진다는 것을 뼈저리게 체감했다. 코드 변경 범위가 넓어질수록 안정성 보장이 개발자의 감(感)에 의존하게 되며, 단위 테스트와 통합 테스트 기반의 자동화된 검증 체계가 반드시 필요하다는 것을 이 프로젝트에서 처음으로 절실히 인식했다.
- 재입사 후 첫 프로젝트로 회사 관행(MyBatis, 세션+쿠키 인증)을 우선 따랐으나, 완료 시점에 JPA 단독 구성과 JWT 기반 인증이 API 서버 구조에 더 적합하다는 것을 인식했다.

### 배운 점
- **도메인 경계 설계**: 독립적인 데이터 흐름을 단일 쿼리로 묶으면 결합도 장애가 발생한다는 것을 실제 장애를 통해 체득한 경험은, 이후 모든 프로젝트에서 아키텍처 설계의 첫 번째 질문이 됐다.
- **사용자 요구사항의 본질 파악**: 사용자의 말 뒤에 있는 진짜 의도를 올바른 아키텍처 언어로 번역하는 능력 — "한 화면"이 "단일 쿼리"가 아님을 간파한 것이 이 프로젝트의 핵심 판단이었다.
- **한계를 성장의 나침반으로**: 이 프로젝트에서 체감한 한계 — 수동 E2E 검증의 회귀 비용, JPA 단독 구성 가능성, JWT 기반 인증의 적합성 — 는 모두 이후 프로젝트들의 아키텍처 설계 발전의 강력한 기준점이자 출발점이 됐다.
