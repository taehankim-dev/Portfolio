> **프로젝트 개요**
>
> 모니터링 시스템 개선 프로젝트 이후 투입된 대규모 MES(Manufacturing Execution System) 프로젝트입니다. 이전 프로젝트에서 처음으로 백엔드 DB 레이어에 발을 들인 경험이 자극이 되어, 이번 프로젝트에서는 프론트엔드 단독 개발을 메인으로 수행하면서 일부 백엔드 API 개발에도 더 적극적으로 참여했습니다. 다만 당시 백엔드 참여 수준은 기본적인 CRUD API를 작성하거나, 다른 백엔드 개발자분들의 기존 코드 구조를 도메인에 맞게 변환하여 사용하는 수준에 그쳤습니다. 백엔드 아키텍처를 스스로 설계하거나 레이어 구조를 이해하고 판단한 것이 아니라, 동작하는 코드를 빌려 맞춰 쓰는 방식이었습니다. 이 한계를 직접 겪으며, 진짜 풀스택 엔지니어가 되기 위해 필요한 것이 무엇인지를 처음으로 실감했습니다.
>
> **프로젝트 성격**: 프론트엔드에서 확립한 컴포넌트 설계 원칙과 처음으로 참여한 백엔드 개발의 한계를 동시에 체험한 프로젝트 — 이 한계가 이후 재입사 시점에 시스템 전체를 책임지는 풀스택 개발자로의 도약을 이끈 결정적 자극제가 됨

## 문제 상황 & 해결

### 1. MES 유사 화면 전반의 UI 중복 구현 — Atomic Design 계층 기반 컴포넌트 재사용 구조 확립

**현상**

입출고 화면, 생산 현황 화면, 불량 관리 화면이 모두 DataTable + SearchBar + StatusBadge 조합을 사용했는데, 이를 각 화면에 별도로 구현했다. 디자인 수정 한 건이 발생하면 동일 UI 패턴을 복사해 둔 모든 화면을 일일이 순회해야 하는 유지보수 구조였다.

| 구분 | 화면별 개별 구현 | Atomic Design 계층 구조 (선택) |
|---|---|---|
| 공통 UI 수정 시 영향 범위 | 동일 패턴을 복사한 모든 화면 개별 수정 | atoms 수정 → 해당 Molecule·Organism 자동 반영 |
| 유사 화면 추가 비용 | 기존 화면에서 코드 복사·붙여넣기 반복 | 기존 컴포넌트 조합만으로 구현 완결 |
| 변경 전파 예측 가능성 | 어느 화면까지 영향을 받는지 추적 불가 | 계층 범위 내로 전파 경계 명확화 |

**원인**:
- 화면 단위로 UI 요소를 독립 구현하는 구조에서 공통 설계 계층이 없었고, 재사용 기준이 사전에 정의되지 않아 화면마다 유사 코드가 누적됐다.

**해결 과정**:

변경이 전파되는 범위를 계층으로 통제할 수 있다는 Atomic Design 원칙을 실무에 적용했다. Button 색상을 바꾸면 Button을 사용하는 모든 Molecule·Organism이 자동으로 반영되고, SearchBar 레이아웃 변경은 SearchBar를 사용하는 화면에만 영향을 주며 Button에는 영향을 주지 않는다.

```
src/components/
├── atoms/           # 단독으로 의미를 갖는 최소 단위 (변경 시 전체 전파)
│   ├── Button.vue
│   ├── Input.vue
│   └── Badge.vue
├── molecules/       # Atom의 조합 — 단일 기능 단위
│   ├── SearchBar.vue
│   ├── FormField.vue
│   └── StatusCard.vue
├── organisms/       # 화면의 독립적인 구역 단위
│   ├── DataTable.vue
│   ├── ProductionChart.vue
│   └── Header.vue
└── templates/       # 페이지 레이아웃 골격만 담당
    ├── DashboardLayout.vue
    └── FormLayout.vue
```

처음에는 모든 요소를 Atomic Design 계층에 맞추려 했지만, 화면 전용 UI는 굳이 atoms/molecules로 분리하면 오히려 복잡도만 높아졌다. 재사용이 2회 이상 확인된 경우에만 상위 계층으로 분리하는 실용적 기준을 세웠다.

**결과**:
- 공통 컴포넌트 수정이 계층 범위 내 모든 사용처에 자동 반영되어 화면별 개별 수정이 불필요해졌다.
- SearchBar·DataTable·StatusCard 등 약 30개 공통 컴포넌트 구축으로 유사 화면 추가 시 조합만으로 구현이 완결됐다.

## 기술적 판단 & 설계 포인트

### 1. UI 컴포넌트의 비즈니스 로직 배제 — emit 경계 설계와 책임 분리 원칙의 체득

공통 SearchBar를 입출고·생산·불량 3개 화면에서 재사용하려면, SearchBar 자체가 어떤 화면의 조회 로직도 내부에 알아서는 안 됐다. SearchBar가 특정 화면의 API를 직접 호출하는 순간, 그 컴포넌트는 더 이상 다른 화면에서 쓸 수 없게 된다.

**결정**: SearchBar는 `search` 이벤트만 emit하고, 실제 조회 로직은 전적으로 부모 컴포넌트가 담당하도록 경계를 명확히 설계했다.

```vue
<template>
  <div class="search-bar">
    <v-text-field
      v-model="searchQuery"
      placeholder="검색..."
      prepend-inner-icon="mdi-magnify"
      variant="outlined"
      density="compact"
      @keyup.enter="handleSearch"
    />
    <v-btn color="primary" @click="handleSearch">검색</v-btn>
  </div>
</template>

<script setup lang="ts">
import { ref } from 'vue';

const searchQuery = ref('');
const emit = defineEmits<{ search: [query: string] }>();
const handleSearch = () => emit('search', searchQuery.value);
</script>
```

| 항목 | 로직 혼재 방식 | emit 경계 분리 방식 (선택) |
|---|---|---|
| 화면 간 재사용 가능성 | 특정 화면 조회 로직을 알기 때문에 다른 화면에 사용 불가 | 이벤트 emit만 담당 → 모든 화면에서 독립 재사용 가능 |
| 수정 시 영향 범위 | 컴포넌트 수정이 연결된 API 호출 로직까지 전파 | UI 수정이 부모의 조회 로직과 완전히 격리 |
| 책임 단위 명확성 | UI 렌더링과 비즈니스 조회가 동일 레이어에 혼재 | UI 컴포넌트는 UI만, 조회 로직은 부모만 담당 |

> SearchBar가 비즈니스 로직을 모른다는 사실이, 입출고·생산·불량 화면 모두에서 같은 컴포넌트를 재사용할 수 있게 만드는 핵심이었다. 이 "UI와 로직의 분리" 원칙은 이후 백엔드 레이어드 아키텍처 설계에서 Controller·Service·Repository 레이어 경계 분리 개념으로 자연스럽게 이어졌다.

## 결과

### 구조적 성과
- **변경 전파 범위 통제**: Atomic Design 계층 구조로 공통 컴포넌트 수정이 사용처 전체에 일관되게 반영 — 화면별 개별 수정 불필요
- **재사용 기반 확보**: SearchBar, DataTable, StatusCard 등 약 30개 공통 컴포넌트 구축 — 유사 화면 추가 시 조합만으로 구현
- **설계 관점 전환**: UI 컴포넌트에서 로직을 분리하는 과정에서 "책임 분리"와 "경계 설계"의 중요성을 체감 — 이후 백엔드 레이어 설계로 이어진 관심의 시작점

## 회고

### 잘한 점
- **체계적인 컴포넌트 설계**: Atomic Design 개념을 실무에 처음 적용하여 변경 전파 범위를 계층으로 통제하는 구조를 직접 설계했다. 개념을 읽는 것과 실제 화면 수십 개에 적용하는 것은 다른 수준의 판단을 요구한다는 것을 체득했다.
- **재사용 기준의 실용적 설정**: 처음에는 모든 UI 요소를 Atomic Design에 맞추려 했지만, 재사용이 2회 이상 확인된 것만 상위 계층으로 올리는 실용적 기준을 세워 불필요한 복잡도 증가를 방지했다.

### 아쉬운 점
- **백엔드 참여의 기술적 한계**: 이 프로젝트에서 일부 백엔드 API 개발에 참여했지만, 수준은 기본적인 CRUD API를 작성하거나 다른 백엔드 개발자분들의 기존 소스 코드 구조를 그대로 복사해 도메인에 맞게 변환하는 수준이었다. 레이어드 아키텍처가 왜 그렇게 설계되어 있는지, Service와 Repository가 분리되는 이유가 무엇인지를 이해하고 판단한 것이 아니라, 동작하는 패턴을 빌려 쓰는 것에 그쳤다. 스스로 백엔드 엔지니어나 풀스택 개발자라고 부르기에는 기술적 깊이와 기반 지식이 명백히 부족했다.
- 컴포넌트 계층 기준이 문서화되지 않아 팀원들이 일관된 기준으로 컴포넌트를 배치하기 어려웠다. Atomic Design 도입의 효과가 팀 전체로 확산되려면 어느 단위를 atoms·molecules로 분류할지에 대한 명시적 기준 문서가 함께 있어야 했다.

### 배운 점
- **"책임 분리"가 풀스택 엔지니어 시각으로 전환된 이정표**: SearchBar가 비즈니스 로직을 모르게 설계한다는 결정이 컴포넌트 재사용성 이상의 의미를 가진다는 것을 이 프로젝트에서 처음 깨달았다. UI 레이어와 비즈니스 로직 레이어의 경계를 설계하는 과정에서 레이어드 아키텍처의 본질 — "각 레이어가 자신의 책임 범위 밖을 알지 못하게 설계하는 것" — 을 처음으로 실감했다.
- **한계를 정면으로 마주한 것이 도약의 자극제**: 프론트엔드 설계에서는 주도적으로 판단했지만, 백엔드 API 개발에서는 기존 코드를 복사·변환하는 수준을 벗어나지 못했다는 사실이 이 프로젝트가 끝날 무렵 명확하게 보였다. "동작하는 코드를 빌려 쓸 수 있는 것"과 "아키텍처를 이해하고 스스로 설계할 수 있는 것"은 전혀 다른 역량이었다. 이 한계를 직시했기 때문에, 이후 재입사 시점에는 백엔드 레이어드 아키텍처를 스스로 설계하고, 프론트엔드부터 DB까지 시스템 전체를 단독으로 책임질 수 있는 풀스택 개발자로의 도약을 구체적인 목표로 삼게 됐다. 이 프로젝트의 한계가 없었다면, 그 도약을 위한 동기도 없었을 것이다.
