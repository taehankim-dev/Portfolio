## 문제 상황 & 해결

### 1. 프로덕션 배포 시 이기종 도메인 CORS 충돌 — WebSocket 실시간 스트림 완전 차단

**현상**

로컬 개발 환경에서는 프론트엔드와 백엔드가 동일 호스트에서 구동되어 CORS 이슈 없이 WebSocket 연결이 정상 동작했다. 그러나 공식 프로덕션 환경으로 배포하자, 정적 파일을 서빙하는 프론트엔드 출처(Origin)와 API 서버 출처가 달라지면서 브라우저가 WebSocket 핸드셰이크 단계에서 Cross-Origin 정책 위반으로 연결 자체를 강제 차단했다. 외부 고객이 문의를 접수해도 관리자 대시보드에 실시간 알림이 전혀 도달하지 않는 상태였다.

| 요청 유형 | 차단 위치 | 차단 원인 |
|---|---|---|
| REST API 요청 | 브라우저 CORS 정책 | 출처(Origin)가 API 서버와 상이 |
| WebSocket 업그레이드 요청 | 브라우저 보안 정책 | 핸드셰이크 단계 Cross-Origin 차단 |

**원인**:
- 로컬 환경과 프로덕션 환경은 프론트엔드·백엔드의 호스트 구성이 근본적으로 다르다. 로컬에서는 동일 호스트로 동작하지만, 프로덕션에서는 각자의 출처를 갖게 되므로 브라우저의 교차 출처 보안 정책이 발동한다.
- WebSocket 프로토콜은 초기 HTTP 업그레이드 핸드셰이크 단계에서 CORS 정책의 적용을 받으므로, REST API의 CORS 설정과 별도로 WebSocket 전용 Origin 검증 처리가 필요하다는 점을 초기 설계에서 누락했다.

**해결 과정**:

*HTTP 레이어*: Express CORS 미들웨어 설정에서 허용 출처를 공식 프로덕션 도메인 호스트로 명시적으로 한정하여, REST API에 대한 Cross-Origin 요청을 허가된 출처에서만 수락하도록 통제했다.

*WebSocket 레이어*: HTTP → WebSocket 업그레이드 핸드셰이크 시점에서 요청 헤더의 `Origin` 값을 서버 측에서 직접 검사하는 검증 핸들러를 별도로 추가했다. 허용 목록에 등록되지 않은 출처의 연결 요청은 업그레이드 단계에서 즉시 거부하도록 구현하여, REST API 레이어와 WebSocket 레이어 각각에서 보안 정책이 독립적으로 작동하는 이중 방어 구조를 구성했다.

**결과**:
- 허가된 공식 프로덕션 출처에서의 WebSocket 연결이 정상 동작하며, 신규 문의 발생 즉시 관리자 대시보드에 실시간 알림이 도달했다.
- 비허가 출처의 연결 시도는 차단 상태를 유지하여 보안 정책과 서비스 기능이 동시에 충족되었다.
- CORS 통제가 REST API 레이어와 WebSocket 레이어에 각각 독립적으로 적용되어, 어느 한 레이어의 우회 시도도 다른 레이어에서 차단하는 이중 보안 구조가 완성되었다.

### 2. 서버 프로세스 비정상 종료 시 실시간 알림 파이프라인 단절 — PM2 자가 회복 체계 구축

**현상**

Node.js 프로세스가 예외 오류나 서버 리부팅으로 인해 예기치 않게 종료되면, PostgreSQL을 `LISTEN` 중이던 전용 DB 커넥션과 WebSocket 서버가 함께 내려간다. 이 상태에서 DB 트리거는 계속 `pg_notify()`를 발화하지만 수신할 리스너가 존재하지 않아, 고객이 문의를 접수해도 관리자에게 알림이 전혀 전달되지 않는 서비스 공백이 발생했다.

| 장애 시나리오 | 영향 | 복구 방법 |
|---|---|---|
| 예외 오류로 인한 프로세스 크래시 | LISTEN 커넥션 소실 → 알림 유실 | 수동 SSH 접속 후 프로세스 재기동 |
| 서버 리부팅 | 전체 파이프라인 비가동 상태 | 수동 SSH 접속 후 서비스 재시작 |

**원인**:
- 단일 Node.js 프로세스 위에서 API 서버, LISTEN 전용 커넥션, WebSocket 서버가 함께 동작하는 구조이므로, 프로세스가 종료되면 세 구성 요소가 동시에 내려가는 단일 장애점이 존재했다.
- 개발 단계에서는 로컬 환경에서 수동으로 서버를 재기동하는 것이 가능했으나, 프로덕션 환경에서 수동 개입 의존은 서비스 연속성을 보장하지 못하는 구조라는 점을 프로세스 크래시 발생 후 인식했다.

**해결 과정**:

*프로세스 매니저 도입*: PM2를 프로덕션 환경에 적용하여, 프로세스 비정상 종료를 감지하는 즉시 자동으로 재기동하는 자가 회복 구조를 구성했다. 크래시 발생 후 수 초 이내에 LISTEN 커넥션과 WebSocket 서버가 함께 복구된다.

*부팅 시퀀스 등록*: `pm2 startup` 명령으로 시스템 부팅 시퀀스에 PM2 자체를 데몬으로 등록하고, `pm2 save`로 현재 프로세스 목록을 저장하여 서버 리부팅 후에도 자동으로 서비스가 구동되도록 구성했다. PM2의 로그 집계 기능을 통해 프로세스 생명주기 이력이 보존되어 사후 장애 분석도 지원한다.

**결과**:
- 프로세스 크래시 발생 시 PM2가 자동으로 재기동하여 LISTEN 커넥션과 WebSocket 서버가 수 초 이내에 복구된다.
- 서버 리부팅 이후에도 부팅 시퀀스 자동 구동이 보장되어 실시간 알림 파이프라인의 상시 가동이 유지된다.
- 관리자 수동 개입 없이 서비스 연속성을 유지하는 자가 회복 인프라가 확립되었다.

## 기술적 판단 & 설계 포인트

### 1. PostgreSQL LISTEN/NOTIFY — 폴링 없는 DB 레이어 Pub/Sub 실시간 파이프라인

실시간 알림을 구현하는 가장 흔한 방식은 일정 주기로 DB를 SELECT하는 폴링(Polling)이다. 그러나 폴링은 이벤트가 없어도 주기적으로 쿼리를 실행하며 DB 커넥션을 소비하고, 폴링 주기만큼의 응답 지연이 구조적으로 발생한다.

PostgreSQL 내장 `LISTEN/NOTIFY` 메커니즘은 이 문제를 DB 레이어에서 원천적으로 해결한다. INSERT/UPDATE 이벤트가 발생하는 순간 DB 트리거가 `pg_notify()`를 호출하여 채널에 이벤트를 발행하고, 해당 채널을 `LISTEN` 중인 전용 커넥션이 즉시 이벤트를 수신한다. 외부 메시지 브로커나 폴링 레이어 없이, DB에 데이터가 저장되는 순간과 관리자 알림이 도달하는 순간이 원자적으로 결합된다.

```
[외부 고객 — 랜딩 페이지 문의 폼 제출]
              │ REST API 요청
              ↓
[Node.js/Express API 서버]
              │ INSERT 실행
              ↓
[PostgreSQL DB]
              │ AFTER INSERT 트리거 발동
              │ → pg_notify(채널명, JSON 페이로드)
              ↓
[Node.js — LISTEN 전용 DB 커넥션]
              │ 이벤트 수신 → JSON 파싱
              ↓
[WebSocket 브로드캐스트 레이어]
              │ 연결된 모든 관리자 클라이언트에 PUSH
              ↓
[관리자 Vue 3 대시보드]
  실시간 신규 문의 알림 수신 및 목록 갱신
```

구독 채널은 이벤트 성격별로 독립 설계하여, 관리자 화면이 이벤트 타입에 따라 서로 다른 UI 반응을 정밀하게 제어할 수 있는 구조를 구성했다.

| 채널 | 발화 조건 | 관리자 반응 |
|---|---|---|
| 신규 일반 문의 알림 채널 | 일반 도입 문의 폼 INSERT 시 | 신규 문의 알림 팝업 + 목록 갱신 |
| 청년농업인 신규 문의 알림 채널 | 청년농업인 지원 문의 접수 INSERT 시 | 전용 알림 팝업 + 청년농업인 목록 갱신 |

### 2. 2-도메인 문의 데이터 분리 & 전화번호 기반 중복 고객 자동 병합

서비스 초기에는 일반 도입 문의 채널 하나로 출발했다. 이후 청년농업인 지원 사업 시즌이 도래하면서 청년농업인 전용 문의 채널이 추가되었다. 두 채널은 수집하는 고객 정보 필드의 구성이 다르고, 관리자 관제 화면에서도 각 채널의 문의를 분리하여 다루어야 하므로 독립적인 엔티티로 설계했다.

| 문의 테이블 | 역할 | 유입 경로 |
|---|---|---|
| 일반 도입 문의 | 스마트팜 설비 도입 상담 접수 | 메인 문의 폼 |
| 청년농업인 지원 문의 | 청년농업인 지원 사업 참여 신청 접수 | 청년농업인 전용 지원 페이지 |

동료 개발자가 청년농업인 페이지 화면 개발을 담당하는 일정에 맞춰, 청년농업인 문의 채널의 백엔드 API 엔드포인트와 DB 테이블 스키마를 선행 설계하여 인터페이스 명세를 먼저 전달했다. 프론트엔드 구현과 백엔드 구현이 의존성 없이 병렬로 진행될 수 있는 협업 흐름을 만든 것이다.

**중복 고객 자동 병합**: 새 문의가 접수될 때 백엔드는 전화번호를 식별 키로 기존 이력을 먼저 조회한다. 동일 전화번호의 고객이 이미 존재하면 신규 문의를 기존 레코드에 인라인으로 병합하여, 동일 고객의 모든 접촉 이력이 단일 레코드에 통합 관리되는 구조를 유지한다. 데이터 삭제는 소프트 삭제(논리적 비활성화) 방식으로 처리하여 실제 레코드를 보존하고, 상담 진행 상태는 `접수 → 진행중 → 완료 / 보류`의 생명주기로 정의하여 담당자 배정·상담 메모·처리 일시를 단계별로 누적 기록하는 구조를 구성했다.

> 고객이 다른 날에 다시 문의를 접수해도, 관리자는 단일 고객 카드에서 모든 상담 이력을 시간순으로 파악할 수 있다.

### 3. Nginx 최전방 HTTP→HTTPS 301 영구 리다이렉트 — 전 구간 암호화 통신 강제 및 SPA 라우팅 무결성

프로덕션 서비스에서 리버스 프록시는 단순한 포워딩 레이어가 아니다. 사용자 요청이 애플리케이션 서버에 도달하기 전 통과하는 보안 최전방으로서, 전 구간 암호화 통신 강제라는 인프라 책임을 담당한다.

비보안 HTTP 채널로 유입되는 모든 요청을 HTTPS 보안 채널로 강제 전환하는 301 영구 리다이렉트(Permanent Redirect) 규칙을 Nginx 설정에 구성했다. `return 301`은 임시 우회(302 Temporary Redirect)와 달리, HTTP 프로토콜 수준에서 해당 리소스가 영구적으로 보안 채널로 이동했음을 브라우저에 명시한다. 브라우저는 이 신호를 수신한 이후 동일 호스트에 대한 요청을 자동으로 HTTPS 채널 우선으로 처리하므로, 단 한 번의 설정으로 비보안 평문 채널 접근을 구조적으로 억제한다. 데이터 전송 구간에서 발생할 수 있는 도청·중간자 공격(MITM) 위협을 리버스 프록시 최전방에서 원천 차단했다.

```nginx
# HTTP → HTTPS 301 영구 리다이렉트 (비보안 채널 원천 차단)
server {
    listen 80;
    return 301 https://$host$request_uri;
}

# HTTPS + SPA 라우팅 포워딩
server {
    listen 443 ssl;
    # SSL 인증서 설정

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

Vue Router 기반 SPA는 클라이언트 사이드에서 라우팅을 처리하므로, 브라우저가 특정 경로로 직접 접근하거나 새로고침하면 해당 경로에 실제 파일이 없어 서버가 404를 반환하는 구조적 문제가 있다. `try_files` 지시어로 요청된 경로에 파일이 없을 때 진입점 HTML로 안전하게 포워딩하여 SPA 라우팅 무결성을 확보했다. 공개 소개 페이지와 관리자 전용 관제 페이지 모두 HTTPS 보안 채널 위에서 일관되게 서빙되며, 어떤 경로로 직접 접근하거나 새로고침해도 Vue Router가 정상적으로 라우팅을 처리한다.

### 4. CORS 화이트리스트 + WebSocket Origin 검증 — 이중 보안 레이어 설계

Node.js/Express 서버는 REST API 요청과 WebSocket 연결 요청을 함께 처리하는 구조이므로, 두 채널 각각에 독립적인 출처 검증 레이어가 필요하다.

| 보안 레이어 | 적용 위치 | 역할 |
|---|---|---|
| CORS 화이트리스트 | Express 미들웨어 레이어 | REST API Cross-Origin 요청 허용 출처 제한 |
| Origin 검증 핸들러 | WebSocket 업그레이드 핸드셰이크 단계 | 비허가 출처의 WebSocket 연결 요청 즉시 거부 |

HTTP CORS 설정만으로는 WebSocket 연결까지 보호되지 않는다. WebSocket 프로토콜은 초기 HTTP 업그레이드 요청의 형식을 취하지만, 이후 독립적인 연결로 전환되는 특성상 별도의 Origin 검증 핸들러가 서버 측에 명시적으로 구현되어야 한다. REST API 레이어와 WebSocket 레이어가 각각 독립적으로 출처를 검증하므로, 한 레이어를 우회하더라도 다른 레이어에서 차단하는 구조가 완성된다.

## 회고

### 잘한 점
- **이벤트 드리븐 아키텍처 선택**: 폴링 방식 대신 PostgreSQL 네이티브 Pub/Sub을 DB 레이어에서 직접 활용하는 구조를 선택한 것은, 불필요한 외부 의존성 없이 실시간 이벤트 파이프라인을 간결하게 완성한 핵심 판단이었다.
- **협업 병렬 개발 리드**: 동료 개발자의 청년농업인 페이지 화면 개발 일정에 맞춰 백엔드 API 명세와 DB 스키마를 선행 제공함으로써, 프론트엔드·백엔드 구현이 의존성 없이 동시에 진행되는 협업 흐름을 설계하고 실행했다.
- **디자이너 시안 기술 검토**: 단순히 디자인 시안을 구현하는 수동적 역할에 그치지 않고, 기술 구현 가능성과 성능·접근성 관점에서 시안을 검토하고 대안을 제안하는 기술 협업 파트너로서의 역할을 수행했다.
- **보안 인프라 일괄 완결**: Nginx 301 영구 리다이렉트, Express CORS 화이트리스트, WebSocket Origin 검증을 배포 완료 시점에 함께 구성하여, 보안 설정이 후속 작업이 아닌 배포 요건 그 자체가 되도록 했다.

### 아쉬운 점
- 로컬 개발 환경과 프로덕션 환경 간의 출처(Origin) 차이를 개발 단계에서 사전에 시뮬레이션하지 않았다. CORS 충돌은 프로덕션 배포 후에야 발견되었는데, 로컬 환경에서 이기종 출처 구조를 재현하는 테스트 환경을 미리 구성했다면 배포 전에 발견하고 해결할 수 있었던 문제였다. 개발 환경과 프로덕션 환경의 동등성(Environment Parity) 확보가 배포 안정성의 선결 조건이라는 것을 이 경험으로 체득했다.
- PM2 도입 전 단계에서 프로세스 비정상 종료 시의 서비스 공백을 고려하지 않았다. 프로세스 생명주기 관리 전략은 처음부터 인프라 설계 요건에 포함되어야 하는 항목임을 프로세스 크래시 발생 후에야 인식했다.

### 배운 점
- **DB 레이어 Pub/Sub 아키텍처**: 외부 메시지 브로커 없이 PostgreSQL 네이티브 `LISTEN/NOTIFY` 만으로 이벤트 드리븐 파이프라인을 구성하는 방식을 이 프로젝트에서 처음 설계·구현했다. 기술 스택을 추가하지 않고 이미 사용 중인 DB의 네이티브 기능으로 실시간 파이프라인을 완성할 수 있다는 설계 관점은, 이후 프로젝트에서 외부 의존성 도입 여부를 판단하는 기준이 됐다.
- **환경 동등성과 배포 전 검증**: 로컬에서 정상 작동했다고 프로덕션에서도 동일하게 작동한다고 가정하면 안 된다. 특히 CORS처럼 환경 구성에 따라 전혀 다른 결과가 나타나는 보안 정책 관련 항목은, 반드시 프로덕션과 동일한 출처 구조를 재현하는 환경에서 사전 검증이 이루어져야 한다.
- **보안 레이어 설계는 배포 요건**: 보안 설정을 배포 이후 별도 작업으로 미루지 않고, 인프라 구성과 배포를 하나의 완결된 단위로 묶어 처리하는 방식이 배포 품질을 실질적으로 보장한다. Nginx 리다이렉트·CORS·WebSocket Origin 검증을 배포 시점에 함께 완결했던 경험은, 이후 모든 배포 작업에서 보안 체크리스트를 배포 요건의 일부로 다루게 되는 출발점이 됐다.
