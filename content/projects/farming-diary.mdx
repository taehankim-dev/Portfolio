> **프로젝트 개요**
>
> 사라팜 스마트팜 고객이 매일 수행하는 파종·수확·질병 처치 등의 현장 작업을 체계적으로 기록하고, 축적된 데이터를 향후 AI 분석 기반으로 활용할 수 있도록 전산화한 풀스택 모바일 영농 기록 서비스입니다. 단순 기록 도구를 넘어 위치·작업 유형·질병 이력·알림 패턴 데이터가 머신러닝 피처로 연결될 수 있는 군집 구조 스키마를 설계 단계부터 고려하며, Spring Boot 백엔드 전 구간·Vue 3/Capacitor Android 앱·Docker Compose 인프라를 단독으로 설계·구현했습니다. 사내 기존 공유 Nginx 자산을 외부 진입점으로 재활용하여 추가 인프라 비용 없이 배포 구조를 완성했으며, Compressor.js 클라이언트 사이드 이미지 압축으로 현장 고용량 이미지의 서버 부하를 업로드 전 차단했습니다.
>
> **프로젝트 성격**: 사내 인프라 자산을 실리적으로 재활용하고 AI 확장 기반 데이터 구조를 선제 설계하면서, 현장 검증을 통해 기능 완성과 현장 사용성이 별개 차원임을 직접 확인하고 차세대 이터레이션 방향을 확보한 모바일 풀스택 서비스

## 문제 상황 & 해결

### 1. 내부 테스트 현장 검증을 통한 물리적 작업 환경과 UX 제어 불협화음 식별

**현상**

전 기능이 완벽하게 가동되는 프로토타입 시스템을 내부 테스트 서버에 배포하고 실제 농장 현장에서 사용성 검증을 진행하던 중, 작업자가 흙이 묻은 작업 장갑을 탈착해가며 모바일 화면을 수동 터치하여 일지를 기록해야 하는 현장 프로세스 충돌 문제가 발견됐다.

| 구분 | 개발 단계 가정 | 현장 검증 결과 |
|---|---|---|
| 입력 환경 | 손이 자유로운 상태에서 화면 터치 | 흙 묻은 장갑 착용으로 터치 불가 |
| 기록 가용 시간 | 작업 전후 충분한 입력 여유 | 작업 흐름 중 극도로 제한된 기록 시간 |
| 조작 방식 | 멀티 탭·폼 필드 순차 입력 | 장갑 탈착 없이 처리 가능한 최소 조작 요구 |

**원인**:
- 브라우저 화면 설계와 데이터 바인딩의 완성도에만 집중하다 보니, 물리적 작업 현장 도메인의 환경적 제약 — 손이 장갑으로 묶여 있고 기록 시간이 극도로 제한된다는 조건 — 을 설계 단계에서 선제적으로 인지하지 못했다.
- 기능의 완성도와 현장 가용성이 완전히 분리된 별개의 차원임을 내부 테스트 환경에서는 확인할 수 없는 구조적 한계였다.

**해결 과정**:

현장 피드백을 바탕으로 기능 완성도와 현장 사용성이 별개 차원임을 파악하고, 두 방향으로 대응 방향성을 구체화했다. 첫째, 폼 레이아웃의 극단적 간소화를 통해 최소 터치 조작으로 기록이 완결되는 구조로 이터레이션을 설계했다. 둘째, 음성 인식 기반 일지 입력 파이프라인을 차세대 개선 방향으로 확정했다. 이미 구축 완료된 AI 확장용 군집화 관계형 스키마는 음성으로 파싱된 구조적 데이터도 스키마 변경 없이 수용할 수 있어 설계 무결성이 유지됐다.

| 항목 | 기존 구현 (터치 폼 입력) | 차세대 이터레이션 방향 |
|---|---|---|
| 입력 방식 | 화면 터치 + 멀티 필드 폼 순차 입력 | 음성 인식 기반 자연어 입력 |
| 현장 가용성 | 장갑 탈착 없이는 실사용 불가 | 음성 명령만으로 기록 완결 |
| 데이터 스키마 호환성 | 구조적 데이터로 저장 | 음성 파싱 결과도 기존 스키마로 수용 가능 |

**결과**:
- 비즈니스 현장 검증을 통해 진짜 필요한 도메인 개선 요구사항을 조기에 발굴했다.
- 기 설계된 데이터 영속성 구조를 강력한 엔지니어링 자산으로 보존한 채 차기 고도화 규격을 선제 확보했다.

## 기술적 판단 & 설계 포인트

### 1. Android 전용 클라이언트 사이드 이미지 압축 — 현장 이미지의 서버 부하를 업로드 전 차단

사용자 그룹이 Android 기기를 사용하는 현장 작업자로 특정되어 있어, 핵심 과제는 농장 현장에서 촬영된 수 메가바이트 이상의 고용량 이미지를 서버에 도달하기 전에 클라이언트 단에서 먼저 처리하는 것이었다. 서버 사이드에서 수신 후 압축하는 방식은 고용량 파일의 네트워크 전송 자체를 막지 못한다.

**결정**: `Compressor.js`를 활용하여 화질 저하를 최소화하는 설정으로 파일 크기를 클라이언트 단에서 직접 압축 후 멀티파트 폼으로 전송하는 파이프라인을 구성했다.

| 처리 방식 | 서버 사이드 압축 | 클라이언트 사이드 압축 (Compressor.js) |
|---|---|---|
| 서버 수신 파일 크기 | 수 MB 원본 그대로 수신 | 이미 압축된 파일만 수신 |
| 현장 네트워크 부담 | 고용량 전송으로 업로드 실패 위험 | 전송량 감소 → 열악한 환경에서도 업로드 성공률 향상 |
| 서버 처리 부담 | 이미지 처리 API 응답 지연 발생 | 파일 저장 API의 응답 부담 구조적 제거 |

> 서버는 이미 압축된 파일만 수신한다. 현장의 열악한 모바일 네트워크에서도 업로드 성공률이 높아지고, 서버 스토리지와 파일 처리 API의 응답 지연 부담이 구조적으로 제거됐다.

### 2. 사내 기존 Nginx 자산 재활용 — 2-컨테이너로 인프라를 완결

서비스 배포 환경을 검토하는 과정에서 사내 호스트 서버에는 이미 공유 Nginx가 운영 중이었다. 별도의 Nginx 컨테이너를 추가로 구성하는 것은 이미 존재하는 자산을 중복 투자하는 셈이었다. Docker Compose의 역할을 애플리케이션 컨테이너 관리로 한정하고, 사내 Nginx를 외부 진입점으로 재활용하는 구조가 현실적이라고 판단했다.

**결정**: Docker Compose는 FE 컨테이너(Vue 빌드 파일 서빙)와 BE API 컨테이너(Spring Boot API) 2개만 관리하고, 사내 공유 Nginx가 외부 트래픽을 수신하여 두 컨테이너로 프록시하는 구조로 설계했다.

| 구분 | Nginx 컨테이너 추가 방식 | 사내 공유 Nginx 재활용 방식 |
|---|---|---|
| 관리 컨테이너 수 | 3개 (FE + BE + Nginx) | 2개 (FE + BE) |
| 인프라 관리 비용 | Nginx 컨테이너 설정·유지보수 추가 | 기존 운영 중인 자산 그대로 활용 |
| 중복 투자 여부 | 이미 존재하는 기능을 새로 구성 | 사내 자산 재활용으로 배포 프로세스 슬림화 |

> 관리 대상 컨테이너가 최소화되어 배포·재시작 절차가 단순해졌다. 사내 Nginx의 기존 설정과 운영 이력을 그대로 활용하면서, 새로운 인프라 도입 비용 없이 안정적인 서빙 구조를 확보했다.

### 3. AI 확장을 고려한 관계형 데이터 스키마 설계

영농일지 서비스의 최종 목표는 단순 기록을 넘어 축적된 데이터를 AI 분석에 활용하는 것이다. 기록 데이터가 AI 모델의 유의미한 피처가 되려면, 자유 텍스트가 아닌 구조화된 코드성 데이터로 누적되어야 한다.

**결정**: 기초 정보 마스터 테이블들을 영농일지 기록 테이블과 외래키로 연결하는 관계형 스키마를 설계했다.

| 마스터 테이블 | 영농일지 연결 방식 | AI 분석 활용 방향 |
|---|---|---|
| 농장 구역 위치 | 외래키로 작업 위치 코드화 | 구역별 작업 분포 및 질병 발생 패턴 분석 |
| 작업 유형 | 외래키로 작업 패턴 코드화 | 파종·수확·방제 시계열 작업 패턴 분석 |
| 질병 정보 + 발생 이력 | 발생 일시·위치와 군집화 | 질병 발생 빈도·위치·시기 기반 예방 모델 학습 |
| 알림 설정 + 완료 이력 | 알림 주기·완료율 누적 | 자동화 알림 스케줄링 최적화 모델 학습 피처 |

> 자유 텍스트 메모가 아닌 코드화된 구조적 데이터로 기록이 쌓인다. 향후 작업 패턴 분석·질병 예측·자동화 알림 최적화 등 머신러닝 파이프라인을 붙일 수 있는 데이터 기반이 확보됐다. 음성 인식으로 파싱된 구조적 데이터도 이 스키마 안으로 그대로 수용된다.

### 4. BCrypt + AES 이중 보안 체계 — 저장 시 단방향, 전송 시 양방향

보안 위협 유형에 따라 요구되는 암호화 방향이 다르다. 비밀번호는 원래 값으로 복원할 필요가 없으므로 단방향 해싱이 적합하고, 사용자 식별자처럼 서비스 로직에서 다시 활용해야 하는 데이터는 복호화가 가능한 양방향 암호화가 필요하다.

**결정**: 암호화 유형을 위협 유형별로 분리했다.

| 보안 위협 경로 | 적용 암호화 방식 | 보호 효과 |
|---|---|---|
| DB 유출 (저장 단계) | BCrypt 단방향 해싱 — 비밀번호 | DB 유출 시 salt 포함 비가역 해시로 평문 비밀번호 복원 불가 |
| 네트워크 탈취 (전송 단계) | AES 대칭키 양방향 암호화 — 사용자 식별자 | API 요청·응답 전 구간 암호화 — 트래픽 탈취 시에도 원문 사용자 정보 노출 차단 |

> 저장 단계의 위협(DB 유출)과 전송 단계의 위협(네트워크 탈취) 두 경로를 각각의 적합한 암호화 방식으로 독립 방어하는 체계가 됐다.

### 5. 세션+쿠키 인증 — 폐쇄망 스마트팜 시스템에서 JWT의 명분이 없다.

설계 초기에는 JWT를 기본 인증 방식으로 검토했다. 그러나 아키텍처 검토 과정에서 이 시스템의 성격을 재평가했다. JWT의 핵심 장점인 무상태 확장성은 소수의 내부 사용자가 사내 폐쇄망 환경에서 접속하는 이 서비스에서 실질적인 이득을 제공하지 않았다. 오히려 토큰 갱신 로직·블랙리스트 관리·클라이언트 저장소 보안 처리 등 JWT 운영에 따르는 복잡도가 비용으로만 남았다.

| 항목 | JWT (무상태 토큰) | 세션+쿠키 (선택) |
|---|---|---|
| 강점이 실현되는 환경 | 다중 서버 수평 확장, 마이크로서비스 간 인증 전파 | 소수 내부 사용자 단일 서버 폐쇄망 접속 |
| 운영 복잡도 | 토큰 갱신·블랙리스트·클라이언트 저장소 보안 처리 필요 | 서버 측 세션 생명주기 관리로 단순화 |
| 이 시스템에서의 JWT 실익 | 실질적 이득 없음 — 복잡도만 증가 | 보안 요건 충족 + 구현 복잡도 최소화 |

> 폐쇄망 내부 시스템의 성격에 맞는 검증된 인증 방식을 선택하여, 인증 구현 복잡도를 낮추면서 보안 요건을 충족했다.

- Compressor.js 클라이언트 사이드 압축 파이프라인 + Capacitor Android 하이브리드 앱 구현 및 내부 테스트 환경 가동 완수

- 수기 기록 방식에서 벗어나 코드성 구조적 데이터로 기록이 쌓이는 전산화 기반 확보 — 향후 머신러닝 파이프라인 연결을 위한 데이터 기반 구축
- 현장 검증에서 장갑 착용 작업자의 터치 입력 제약이라는 UX 한계를 조기에 발굴 → 음성 인식 기반 일지 입력·폼 간소화를 차세대 이터레이션으로 구체화
- 서비스가 중단된 것이 아니라, 더 정밀하게 다듬어야 할 지점을 조기에 발견했다. — 기 설계된 AI 확장 스키마는 차세대 이터레이션의 음성 파싱 데이터도 스키마 변경 없이 수용하는 강력한 엔지니어링 자산으로 보존됐다.

## 회고

### 잘한 점
- **사내 인프라 자산 재활용 판단**: 사내 공유 Nginx가 이미 운영 중이라는 사실을 파악하고, 불필요한 Nginx 컨테이너 추가 없이 2-컨테이너 경량 구조로 인프라를 설계했다. 이는 주어진 환경적 제약을 실리적으로 분석하여 최소 비용으로 안정적 배포 구조를 확보한 결정이었다.
- **AI 확장 기반 스키마 선제 설계**: 단순 기록 도구를 넘어 향후 머신러닝 파이프라인 연결을 고려한 군집화 관계형 스키마를 설계 단계부터 내재화했다. 이 선제 설계 덕분에 음성 인식 기반 차세대 이터레이션으로 전환하더라도 스키마를 변경할 필요가 없다.
- **현장 사용성 검증 수행**: 기능이 동작하는 것으로 개발을 종료하지 않고, 내부 테스트 서버 배포 후 실제 농장 현장에서 사용성 검증을 진행했다. 이 과정에서 코드로는 절대 발견할 수 없는 물리적 환경 충돌 문제를 조기에 발굴할 수 있었다.

### 아쉬운 점
- 현장 작업자의 손이 장갑으로 묶여 있고 기록 가능한 시간이 극도로 제한된다는 물리적 환경 제약을 UX 설계 이전에 도메인 요구사항으로 먼저 수집하지 못했다. 실제 사용자가 어떤 물리적 상황에서 이 화면을 조작하는지를 기능 설계의 출발점으로 삼았어야 했다.
- 개발·검증 단계에서 로그 기반 수동 E2E 방식으로 동작을 확인했다. 도메인별 비즈니스 로직(작업 유형·질병 이력·알림 스케줄 연동)의 정합성을 핵심 기능 단위로 검증하는 구조가 없었기 때문에, 회귀 오류가 발생해도 빠르게 포착하기 어려운 구조였다.

### 배운 점
- **기능 완성과 현장 사용성은 완전히 다른 차원의 도메인이다.**: 모든 기능이 정상 가동되는 프로토타입을 배포했음에도 현장에서의 첫 번째 피드백은 "쓸 수가 없다"였다. 코드의 완성도는 기능의 동작을 보장하지만, 현장 사용성은 물리적 환경 제약과 작업 흐름의 도메인 지식이 설계 단계에서 선행되어야 보장된다. 이후 서비스 설계에서 "이 화면을 사용하는 사람의 손이 자유로운가"를 첫 번째 UX 질문으로 삼게 됐다.
- **선행 프로젝트 인프라 경험이 현 프로젝트 설계 결정을 가속화했다.**: 이전 프로젝트들에서 사내 공유 Nginx 자산의 존재와 운영 방식을 파악했기 때문에, 이 프로젝트에서 별도 Nginx 컨테이너 없이 2-컨테이너 경량 인프라로 설계를 즉시 결정할 수 있었다. 이전 경험에서 체득한 인프라 지식이 다음 프로젝트의 아키텍처 결정을 직접 가속화하는 성장 사이클을 이 프로젝트에서 명확히 인식했다.
