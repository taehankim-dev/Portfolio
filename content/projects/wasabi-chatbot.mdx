## 문제 상황 & 해결

### 1. 이기종 서버 간 이미지 유실 — 멀티파트 중계 파이프라인 파일 버퍼 참조 무결성 확보

**현상**

Node.js 중간 계층이 클라이언트로부터 이미지를 수신한 후 Python AI 코어 서버로 중계할 때, 이미지 분석이 실패하고 파이프라인이 중단되는 현상이 발생했다.

| 장애 유형 | 증상 | 원인 |
|---|---|---|
| 임시 저장 디렉토리 누락 | Multer 이미지 저장 실패 → 중계 파일 참조 불가 | 서버 기동 시 디렉토리 존재 여부 미검증 |
| 멀티파트 필드명 불일치 | Python 서버에서 수신 파일 미인식 → 객체 탐지 미실행 | 이기종 런타임 간 수신 규격 미동기화 |

**원인**:
- Node.js에서 Multer가 이미지를 임시 디렉토리에 저장한 후 해당 경로를 참조하여 form-data 객체를 재조립하는 과정에서 두 가지 원인이 동시에 작용했다.
- 두 런타임이 각자 독립적으로 멀티파트 수신 규격을 정의하다 보니 필드명 불일치가 발생했고, 이를 사전에 검증하는 인터페이스 계약이 없었다.

**해결 과정**:

두 방향을 동시에 수정했다. 서버 기동 시 임시 저장 디렉토리 존재 여부를 검증하는 초기화 로직을 추가하여 디렉토리 누락으로 인한 저장 실패를 원천 차단했다. 동시에 Python 서버 엔드포인트의 멀티파트 수신 필드명 규격과 Node.js의 form-data 전송 필드명을 명시적으로 동기화하여 이기종 런타임 간 인터페이스 불일치를 해소했다.

| 수정 사항 | 방법 | 효과 |
|---|---|---|
| 임시 저장 경로 초기화 검증 | 서버 기동 시 경로 존재 여부 확인 및 생성 로직 추가 | 디렉토리 누락으로 인한 저장 실패 원천 차단 |
| 멀티파트 필드명 동기화 | Python 수신 규격에 맞춰 Node.js 전송 필드명 명시적 정렬 | 이기종 런타임 간 인터페이스 불일치 해소 |

**결과**:
- 이미지 업로드 → Node.js 임시 격리 → form-data 재조립 → Python 서버 수신으로 이어지는 전 구간 이미지 중계 파이프라인이 안정적으로 동작하게 됐다.

### 2. 생성형 AI 예외 및 이기종 런타임 연결 장애 방어 — 환경 변수 격리와 DB 확장성 선제 설계

**현상**

개발 과정에서 Gemini API 호출 실패(API Key 권한 거부)와 Python 서버 연결 거부(이기종 런타임 간 커넥션 장애) 두 가지 유형의 외부 의존성 장애가 반복됐다.

| 장애 유형 | 증상 | 공통 원인 |
|---|---|---|
| Gemini API 호출 실패 | API Key 권한 거부 오류 | API 자격증명 코드 내 하드코딩 |
| Python 서버 연결 거부 | 이기종 런타임 간 커넥션 실패 | 연결 URL 정보 코드 내 하드코딩 |

**원인**:
- 두 장애 모두 외부 의존성 자격증명과 연결 정보를 코드에 직접 기재한 것이 공통 원인이었다.
- 배포 환경이 달라지거나 자격증명이 교체될 때마다 코드를 수정해야 하는 구조였으며, 장애 발생 시 원인 추적이 코드 전체에 걸쳐 분산됐다.

**해결 과정**:

Gemini API 키, Python 서버 연결 URL, PostgreSQL 접속 정보를 모두 `dotenv` 기반 환경 변수로 격리하여, 배포 환경이 바뀌어도 코드 변경 없이 자격증명만 교체할 수 있는 구조로 전환했다. 추가로 PostgreSQL 드라이버를 주석 처리된 예비 레이어로 미리 배치하여, 향후 대화 기록 저장 기능 확장 시 코드 구조 변경 없이 DB 연동 레이어를 활성화할 수 있는 확장 포인트를 선제적으로 설계에 반영했다.

| 처리 항목 | 격리 전 | 격리 후 |
|---|---|---|
| API 자격증명 | 코드 내 하드코딩 | dotenv 환경 변수 격리 |
| 런타임 간 연결 URL | 코드 내 하드코딩 | dotenv 환경 변수 격리 |
| DB 연동 레이어 | 미반영 | 주석 처리 예비 레이어 선제 배치 |

**결과**:
- 환경별 자격증명 관리가 코드에서 완전히 분리됐고, 런타임 간 연결 장애의 원인 추적과 복구 범위가 명확해졌다.
- 코드 구조 변경 없이 DB 연동 레이어를 활성화할 수 있는 확장 포인트를 확보했다.

## 기술적 판단 & 설계 포인트

### 1. Node.js와 Python의 역할 분리 — 런타임 강점을 서비스 구조에 그대로 반영

단일 서버에 모든 AI 로직을 밀어 넣는 방식도 가능했다. 그러나 Node.js와 Python은 처리 특성이 근본적으로 다르다는 판단에서 역할 분리 아키텍처를 설계했다.

**결정**: 클라이언트 요청 수신·라우팅·인증·외부 API 중계는 비동기 I/O 처리에 최적화된 Node.js에 전담시키고, 행렬 연산 기반 객체 탐지(YOLOv8)·자연어 생성(Gemini) 등 CPU 연산 집약 작업은 Python 서버로 격리했다.

| 런타임 | 처리 영역 | 선택 이유 |
|---|---|---|
| Node.js (중간 계층) | 클라이언트 요청 수신·라우팅, Multer 이미지 임시 격리, Gemini 텍스트 상담 직접 처리, form-data 재조립·Python 중계 | 비동기 I/O 처리 최적화 — 요청 수신·중계에 적합 |
| Python (AI 코어) | YOLOv8 객체 탐지(행렬 연산), 탐지 결과 컨텍스트 조립, Gemini 이미지 상담 답변 생성 | CPU 연산 집약 작업 격리 — AI 라이브러리 생태계 최적 |

> 각 런타임이 자신의 강점 범위 안에서만 동작한다. 향후 Python AI 코어의 연산 부하가 증가해도 Node.js 중간 계층의 응답성은 영향을 받지 않는 독립 확장 구조가 됐다.

### 2. 텍스트 API와 이미지 API의 물리 분리 — 런타임 안정성과 응답 지연 병목 제거

초기에는 하나의 엔드포인트에서 이미지 첨부 여부를 분기 처리하는 방식을 시도했다. 그러나 이기종 서버 간 파일 버퍼 수신 시 런타임 에러가 빈번하게 발생했고, 이미지 분석 코어까지 요청이 묶여 들어갈 때 단일 API 전체의 응답 지연(Timeout) 병목이 확인됐다.

**결정**: 예외 발생 확률을 구조적으로 차단하기 위해 '텍스트 전용 상담 API'와 '이미지 분석 결합형 상담 API'를 물리적으로 완전 분리했다. 프론트엔드도 이 구조에 맞춰 텍스트 입력 컴포넌트와 미디어 업로드 컴포넌트의 요청 경로를 각각 독립적으로 제어하도록 설계했다.

| 구분 | 단일 엔드포인트 분기 방식 | 물리 분리 방식 (선택) |
|---|---|---|
| 이기종 파일 버퍼 수신 | 첨부 여부 분기 처리 → 런타임 에러 빈발 | 이미지 전용 API로 예외 경로 완전 격리 |
| 응답 지연 (Timeout) | 텍스트 요청도 이미지 분석 파이프라인 대기 | 텍스트 API가 이미지 처리 지연과 무관하게 응답 |
| 장애 격리 범위 | 이미지 파이프라인 오류가 텍스트 상담 전체를 차단 | API별 단일 책임 범위 → 장애 원인 격리 명확화 |

> 텍스트 상담 요청이 이미지 분석 파이프라인의 처리 지연에 영향을 받지 않게 됐다. 각 API가 단일 책임 범위 안에서만 동작하므로 장애 발생 시 원인 격리와 복구 범위가 명확해졌다.

### 3. 컨텍스트 주입형 하이브리드 AI 프롬프트 파이프라인

YOLOv8 탐지 결과를 단순히 화면에 표시하는 것이 아니라, 생성형 AI의 입력 컨텍스트로 재활용하여 사용자가 체감하는 답변 품질을 높이는 것이 핵심 설계 과제였다. 정형 분석 결과(탐지 클래스명 + 신뢰도%)와 사용자의 자연어 질문을 하나의 구조화된 프롬프트로 결합하는 파이프라인을 설계했다.

| 파이프라인 단계 | 처리 내용 | 역할 |
|---|---|---|
| 1단계: 객체 탐지 | YOLOv8 커스텀 모델 → 탐지 클래스명 + 신뢰도(%) 추출 | 정형 분석 결과 생성 |
| 2단계: 컨텍스트 조립 | 탐지 결과 + 사용자 자연어 질문 결합 | 프롬프트 입력 구조화 |
| 3단계: 출력 규칙 제어 | 시스템 내부 탐지 용어 출력 차단 규칙 프롬프트 삽입 | 프롬프트 무결성 확보 |
| 4단계: 자연어 생성 | Gemini 2.0 Flash 기반 이미지 분석 근거 자연어 답변 생성 | 사용자 최종 응답 반환 |

**프롬프트 무결성 제어**: Gemini가 시스템 내부의 객체 탐지 용어를 사용자 응답에 그대로 출력하지 않도록 프롬프트 출력 규칙을 명시적으로 제어했다. 사용자는 AI가 이미지를 직접 해석한 자연어 응답만 수신하고, 내부 처리 구조는 노출되지 않는다.

> YOLOv8이 탐지한 정형 데이터가 Gemini의 답변 근거로 주입되어 사용자는 "이미지를 직접 분석한 전문 답변"을 체감하고, AI는 컨텍스트 없는 일반 답변이 아닌 현장 데이터 기반 응답을 생성한다.

## 결과

### 정량적 성과
- 회사 스마트팜에서 직접 촬영·수동 레이블링한 50여 장의 수제 데이터셋으로 YOLOv8 커스텀 모델 학습 및 내부 테스트 환경 가동 완수
- Vue 3 → Node.js → Python 3계층 AI 오케스트레이션 파이프라인 기술 타당성 단독 증명 — AI·Python 서비스 개발 지식 제로 베이스에서 출발하여 1개월 내 내부 테스트 가동

### 정성적 성과
- 텍스트 API와 이미지 API의 물리 분리 + 멀티파트 중계 안정화로 이기종 런타임 간 전 구간 파이프라인 신뢰성 확보
- YOLOv8 탐지 결과를 Gemini 2.0 Flash 프롬프트 컨텍스트로 주입하는 하이브리드 AI 파이프라인 구현 — 정형 분석 결과와 자연어 질의 통합 답변 생성 구조 실증
- 3계층 파이프라인이 기술적으로 동작한 이후에도 도메인 데이터 한계를 직접 확인하고 프로덕션 진입을 객관적으로 보류 — 비즈니스 리스크를 계량화하여 판단하는 엔지니어 시선 확립

## 회고

### 잘한 점
- **제로 베이스 독학 PoC 완수**: AI·Python 서비스 개발 지식이 전무한 상태에서 레퍼런스와 오픈소스 문서를 바닥부터 독학하며 3계층 AI 오케스트레이션 파이프라인 기술 타당성을 단독으로 증명했다. 결과가 불확실한 상황에서 무모한 과시 없이 증명 가능한 범위를 설정하고 단계적으로 검증한 접근이 PoC를 완수하게 했다.
- **역할 분리 아키텍처 설계**: Node.js와 Python을 각 런타임의 강점 범위로 분리한 판단은, AI 서비스 구조를 처음 설계하는 상황에서도 각 기술의 처리 특성을 먼저 파악하고 아키텍처에 반영한 결과였다. 단일 서버로 모든 것을 처리하는 단순한 선택이 아닌, 독립 확장과 장애 격리를 고려한 구조를 선택했다.
- **프로덕션 보류 판단**: 파이프라인이 기술적으로 동작하는 상태에서도 내부 테스트를 통해 확인한 도메인 데이터 한계를 근거로 프로덕션 진입을 스스로 보류했다. 기술적 완성과 서비스 안전성을 분리하여 판단할 수 있었던 것은 이 프로젝트의 가장 중요한 성과 중 하나다.

### 아쉬운 점
- 50여 장의 수제 데이터셋은 파이프라인 연결성 검증용으로는 충분했지만, 실제 와사비 현장의 다양한 상태 변수를 커버하기에는 절대적으로 부족했다. 와사비 잎 황화 현상 하나만 보더라도 '햇빛 부족', '과습', '특정 병해충' 등 환경적 원인이 전혀 다르게 나타나는데, 50장으로 이 복잡한 예외 변수를 모델이 구분하게 만드는 것은 구조적으로 불가능한 목표였다. 학습 데이터 규모와 다양성에 대한 현실적 기준을 프로젝트 기획 단계에서 먼저 설정했어야 했다.
- 챗봇 응답의 정합성 검증을 내부 수동 테스트에만 의존했다. AI 응답 품질을 정량적으로 평가할 수 있는 기준(예: 탐지 신뢰도 임계값 설계, 클래스별 오탐율 측정)이 없었기 때문에 "동작한다"와 "신뢰할 수 있다" 사이의 간극을 수치로 파악하기 어려웠다.
- 파이프라인의 기술적 연결성 확보에 우선순위를 두다 보니, 파운데이션 모델을 특정 비즈니스 도메인 전용 AI로 격리하는 시스템 프롬프트 하네스(Guardrail) 설계 지식이 부족했다. 실제로 사용자가 "당신은 누구인가요?"와 같은 정체성 질의를 던졌을 때, 서비스 주체인 "사라팜 와사비 전용 가이드 챗봇"으로 귀속되어 응답하도록 통제하지 못하고 기저 모델의 본래 명칭이 그대로 출력되는 예외 응답이 식별됐다. YOLOv8 탐지 결과를 Gemini 프롬프트 컨텍스트로 주입하는 파이프라인은 설계했으나, 모델의 페르소나 격리와 예외 출력 방어를 위한 시스템 프롬프트 및 가드레일 인프라 구축은 개발 범위에 포함되지 않았다. 향후 LLM을 서비스에 도입할 때는 컨텍스트 엔지니어링과 출력 가드레일 설계를 파이프라인 연결성과 동등한 우선순위로 다뤄야 함을 명확한 개선 과제로 도출했다.

### 배운 점
- **AI 서비스 실무화의 선결 조건은 코드가 아닌 도메인 데이터다**: 인공지능 서비스를 실제 환경에 안착시키기 위해서는 정교한 코드 설계보다 '오염되지 않은 압도적인 양의 도메인 데이터셋 확보'와 '하나의 현상 이면의 복잡한 도메인 원인 분석 모델 설계'가 반드시 선행되어야 함을 뼈저리게 체감했다. 이 PoC는 3계층 파이프라인의 기술 연결 가능성을 증명하는 동시에, AI 서비스 프로덕션 진입에 필요한 데이터 요건과 도메인 복잡성의 규모를 현실적으로 계량화한 값진 경험 자산이다.
- **이기종 런타임 설계의 인터페이스 계약 중요성**: Node.js와 Python이라는 서로 다른 런타임을 연결할 때 발생한 대부분의 장애는 두 시스템이 독립적으로 인터페이스를 정의하다 발생한 불일치에서 비롯됐다. 이기종 런타임 간 통신에서는 필드명·데이터 포맷·에러 응답 구조에 대한 명시적 인터페이스 계약을 설계 초기에 먼저 수립하는 것이 중요함을 실무 디버깅을 통해 직접 체득했다.
