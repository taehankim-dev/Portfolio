>
> 재입사 후 첫 프로젝트로, 중소 제조기업의 수기 자재·제품 입출고 및 재고 관리를 전산화한 내부 관리 시스템입니다. 2인 개발 환경에서 프론트엔드 주요 기능 및 백엔드 일부 API를 구현했으며, 실제 사용자와 소통을 통한 지속적인 개선을 진행했습니다.

> **프로젝트 성격**: 실사용 기업의 수기 관리 시스템을 전산화하고 사용자 피드백을 반영해 안정화시킨 내부 관리 시스템

## 문제 상황 & 해결 경험

**문제**: 수기로 관리되던 입출고 및 재고 관리의 작성 누락과 자료 관리 어려움, 배포 후 사용자의 업무 흐름에 맞지 않는 UI로 인한 효율 저하

**역할**: 프론트엔드 주요 기능 구현 / 백엔드 일부 API 개발 / Nginx 배포 환경 구성 / 실사용자와 소통 및 기능 보완

**결과**:
- 사용자 피드백 반영으로 데이터 조회 편의성 향상
- 엑셀 업로드 전처리 로직 추가로 입력 안정성 확보
- 실제 기업에서 안정적으로 사용 중

## 문제 상황 & 해결

### 1. 업무 흐름에 맞지 않는 UI

**현상**: 배포 초기, 입출고·생산·출고 데이터를 한 화면에서 확인하기 어려워 업무 효율 저하

**원인**:
- 개발자 관점에서 설계한 UI가 실제 업무 흐름과 불일치
- 조회 기준(일별/월별/연간)이 복잡하고 직관적이지 않음

**해결 과정**:
- 사용자와 직접 소통하며 실제 업무 프로세스와 데이터 조회 패턴 파악
- 조회 기준 단순화: 복잡한 옵션을 일별/월별/연간으로 단순화
- 화면 분리: 연관 데이터를 그룹화하여 화면을 명확히 분리
- 대안 제시: 기술적 제약 사항은 사용자에게 설명하고 대안 제시

**결과**:
- 데이터 조회 편의성 향상
- 사용자 이해도 및 만족도 증가

### 2. 엑셀 업로드 오류

**현상**: 엑셀 파일 업로드 시 사용자 입력 실수로 인한 오류 발생

**원인**:
- 사용자가 엑셀 파일에 공백 행을 포함하여 업로드
- 백엔드에서 공백 행을 데이터로 인식하여 처리 오류 발생

**해결**:

업로드 전에 공백 행과 필수 필드를 검증하여 잘못된 데이터가 DB에 진입하는 것을 원천 차단했다. 검증과 저장을 분리하고, 저장은 단일 트랜잭션으로 묶어 부분 저장 상태가 남지 않도록 설계했다.

```java
@PostMapping("/upload/excel")
public ResponseEntity<?> uploadExcel(@RequestParam("file") MultipartFile file) {
    List<RowData> rows = excelParser.parse(file);

    // 검증 단계: 트랜잭션 밖에서 수행 — DB 연결 없이 빠르게 실패
    List<RowData> validRows = rows.stream()
        .filter(row -> !row.isEmpty())          // 공백 행 제외
        .filter(row -> row.hasRequiredFields()) // 필수 필드 검증
        .collect(Collectors.toList());

    if (validRows.isEmpty()) {
        return ResponseEntity.badRequest().body("유효한 데이터가 없습니다");
    }

    // 저장 단계: 전체를 단일 트랜잭션으로 처리
    // 일부 행 저장 실패 시 전체 롤백 → 부분 저장 상태 방지
    inventoryService.saveAll(validRows);
    return ResponseEntity.ok(validRows.size() + "건 업로드 완료");
}
```

```java
@Service
public class InventoryService {

    @Transactional  // 전체 배치 저장을 단일 트랜잭션으로 묶음
    public void saveAll(List<RowData> rows) {
        // 중간에 예외 발생 시 이미 저장된 행도 함께 롤백
        // → 재고 데이터의 부분 반영 상태(inconsistent state) 방지
        rows.forEach(row -> inventoryRepository.save(InventoryItem.from(row)));
    }
}
```

**결과**:
- 잘못된 입력이 DB에 도달하기 전에 차단 — 엑셀 업로드 오류 제거
- 배치 저장 중 실패 시 전체 롤백으로 재고 데이터 정합성 보장

## 기술적 판단 & 설계 포인트

### 1. 기존 시스템 스타일 분석 우선

재입사 후 첫 프로젝트로, 새로운 구조 도입보다는 기존 시스템의 코드 컨벤션과 개발 패턴을 먼저 분석했다. 팀원과의 협업 비용을 줄이고, 운영 중인 시스템과 유사한 구조를 유지해 인수인계 부담을 낮추는 것이 이 시점에서 더 중요한 우선순위였다.

### 2. 입력 검증을 API 레이어가 아닌 서비스 진입 전에 배치

엑셀 업로드 처리에서 검증 로직을 컨트롤러 레이어에서 수행하고 저장은 서비스 레이어로 분리했다. 이 분리 덕분에 서비스 레이어는 "유효한 데이터만 들어온다"는 전제 하에 트랜잭션 처리에만 집중할 수 있으며, 검증 로직은 DB 연결 없이 단독 테스트 가능한 구조가 된다.

| 레이어 | 역할 | 트랜잭션 |
|---|---|---|
| Controller | 파일 파싱 + 유효성 검증 | 없음 |
| Service (`saveAll`) | 배치 저장 | `@Transactional` — 전체 롤백 |

### 3. 조회 기준 단순화 — 사용자 적응 비용 최소화

초기 배포 후 사용자와 직접 소통하며 발견한 핵심 문제는 조회 옵션이 너무 많다는 것이었다. "완전한 기능"보다 "사용자가 바로 쓸 수 있는 기능"이 더 중요한 시점이었고, 일별/월별/연간 3단계로 단순화하여 적응 비용을 낮추는 결정을 했다.

## 테스트 용이성 설계

### ExcelParser — 파일 없이 Row 데이터로 단위 테스트

검증과 저장을 레이어로 분리한 덕분에 엑셀 파싱·검증 로직을 실제 파일 없이 `RowData` 객체만으로 테스트할 수 있다. 공백 행 필터링과 필수 필드 검증이라는 두 가지 분기를 독립 검증하는 것이 핵심이다.

```java
class InventoryUploadValidationTest {

    @Test
    @DisplayName("공백 행이 포함된 목록에서 유효한 행만 필터링된다")
    void filter_out_empty_rows() {
        List<RowData> rows = List.of(
            RowData.of("자재A", 10, "2024-01-15"),  // 유효
            RowData.empty(),                         // 공백 행
            RowData.of("자재B", 5, "2024-01-15")   // 유효
        );

        List<RowData> validRows = rows.stream()
            .filter(row -> !row.isEmpty())
            .collect(Collectors.toList());

        assertThat(validRows).hasSize(2);
    }

    @Test
    @DisplayName("필수 필드 누락 행은 검증에서 제외된다")
    void exclude_rows_with_missing_required_fields() {
        List<RowData> rows = List.of(
            RowData.of("자재A", null, "2024-01-15"),  // 수량 누락
            RowData.of("자재B", 5, "2024-01-15")      // 유효
        );

        List<RowData> validRows = rows.stream()
            .filter(RowData::hasRequiredFields)
            .collect(Collectors.toList());

        assertThat(validRows).hasSize(1);
        assertThat(validRows.get(0).getName()).isEqualTo("자재B");
    }
}
```

### 설계 원칙: 레이어 분리가 테스트 격리를 자연스럽게 달성한다

컨트롤러에서 검증, 서비스에서 저장을 담당하는 구조는 각 책임을 독립적으로 테스트할 수 있게 한다. `InventoryService.saveAll()`은 유효한 데이터만 전달된다는 전제로 `@Transactional` 롤백 동작만 검증하면 되고, 검증 로직 테스트는 DB와 완전히 무관하게 실행된다.
