> **프로젝트 개요**
>
> 개발자로서 첫 실무 프로젝트입니다. 프론트엔드 개발자로 입사해 CJ 올리브네트웍스의 JSP 기반 레거시 모니터링 시스템을 Vue.js로 전환하는 작업을 맡았습니다. 과도한 API 폴링 호출로 인한 서버 부하와 유지보수 어려움 개선이 주 목표였으나, 프로젝트 중반에 대시보드 수치 정합성 문제가 발견되었습니다. 이 문제는 프론트엔드에서 숫자를 보정하는 방법으로 해결할 수 없었고 — 처음으로 백엔드 개발자분의 안내 아래 DB 집계 쿼리를 직접 들여다보고 수정 작업에 참여했습니다. "화면에서 발생한 문제의 원인이 DB 레이어에 있고, 해결도 그 레이어에서 이루어진다"는 사실을 처음으로 손으로 체감한 경험이었습니다.
>

## 문제 상황 & 해결

### 1. 1초 단위 API 폴링으로 인한 서버 부하 — 도메인 요구사항 기반 폴링 주기 재설계

**현상**

모니터링 대시보드에서 1~2초마다 API를 호출하면서 서버 부하가 증가했다. 폴링 간격을 짧게 설정한 이유는 "실시간에 가깝게 보여야 한다"는 요구사항이었으나, 실제로는 공장 설비 상태가 수 초 단위로 급변하지 않기 때문에 1초 갱신이 필요한 상황이 아니었다.

| 구분 | 1초 폴링 (기존) | 5초 폴링 (선택) |
|---|---|---|
| 분당 API 호출 횟수 | 60회 | 12회 (1/5 감소) |
| 서버 부하 | 도메인 요구와 무관한 불필요한 요청 지속 발생 | 도메인 허용 범위 내 제어 |
| 모니터링 품질 | 설비 변화 주기 대비 과도한 빈도 | 이상 감지 허용 지연 범위 내 유지 |

**원인**:
- "실시간" 요구사항이 설비 도메인 특성(수 초 단위 급변 없음)과 이상 감지 대응 구조(담당자 확인 후 대응)에 대한 검토 없이 폴링 간격에 그대로 반영됐다.
- WebSocket·SSE 등 진짜 실시간 기술이 아닌 폴링 구조에서 1초 간격은 도메인 요구와 무관한 서버 부하를 발생시키는 구조였다.

**해결 과정**:

WebSocket(진짜 실시간 필요 시 적합하나 서버 연결 유지 비용 발생), SSE(단방향 스트리밍이나 레거시 JSP 구조와 충돌)를 검토했으나 현재 요구사항에 과도하다고 판단했다. 기존 폴링 구조를 유지하면서 간격 기준만 업무 특성에 맞게 조정하는 방향을 선택했다. 5초 기준은 설비 이상 감지 허용 지연(담당자 확인 후 대응 구조)과 서버 부하 감소 사이의 균형점으로 현장 담당자와 합의하여 결정했다.

```typescript
import { throttle } from 'lodash-es';

// throttle: 연속 호출 시 마지막 호출 시점 기준 5초 대기
// setInterval과 함께 사용해 주기적 갱신도 보장
const fetchMonitoringData = throttle(async () => {
  try {
    const response = await fetch('/api/monitoring/status');
    monitoringData.value = await response.json();
  } catch (error) {
    console.error('모니터링 데이터 조회 실패:', error);
  }
}, 5000);

onMounted(() => {
  fetchMonitoringData();               // 즉시 1회 실행
  setInterval(fetchMonitoringData, 5000); // 이후 5초 주기 유지
});
```

**결과**:
- API 호출 횟수 1/5 감소 (1초 → 5초 기준, 분당 60회 → 12회)
- 서버 부하 감소 — 동일 모니터링 품질 유지하면서 불필요한 요청 제거
- 향후 간격 조정이 필요할 경우 상수 하나만 변경하면 되는 구조

### 2. 화면별 이기종 DB 집계 기준으로 인한 대시보드 수치 정합성 장애

**현상**

대시보드에 표시되는 집계 수치(설비별 가동률, 불량 건수 등)가 실제 데이터와 맞지 않는 경우가 발생했다. 같은 데이터를 여러 화면에서 각자 다른 기준으로 집계하고 있었기 때문이었다. 처음에는 프론트엔드 렌더링 버그를 의심했으나, 디버깅 과정에서 문제의 원인이 화면이 아닌 DB 집계 쿼리 로직에 있다는 것을 발견했다.

| 구분 | 화면별 개별 집계 쿼리 | 집계 기준 일원화 (선택) |
|---|---|---|
| 동일 지표의 집계 결과 | 화면마다 다른 수치 표시 | 모든 화면이 동일 기준 수치 표시 |
| 담당자 데이터 신뢰도 | 수치 불일치로 데이터 신뢰 불가 | 단일 기준 기반 수치 신뢰 확립 |
| 집계 기준 변경 시 | 모든 화면의 쿼리를 개별 수정 | 공통 집계 레이어 한 곳만 수정 |

**원인**:
- 여러 화면에서 동일한 데이터를 집계하는 쿼리가 제각각 작성되어 있었고, 집계 기준(시간 범위, 상태값 분류)이 쿼리마다 달랐다.

```sql
-- 문제: 화면 A와 화면 B가 서로 다른 기준으로 가동률을 집계
-- 화면 A: status = 'RUNNING'만 포함
SELECT COUNT(*) FROM equipment WHERE status = 'RUNNING'

-- 화면 B: status IN ('RUNNING', 'IDLE')을 가동으로 포함
SELECT COUNT(*) FROM equipment WHERE status IN ('RUNNING', 'IDLE')
```

**해결 과정**:

이 문제는 프론트엔드에서 숫자를 임의로 보정하는 방법으로는 해결할 수 없었다. 백엔드 개발자분의 도움을 받아 DB 구조를 함께 들여다보며, 집계 기준을 단일 뷰(View) 또는 공통 쿼리로 일원화하여 화면마다 다르게 해석되던 집계 로직을 통일했다. 백엔드 레이어를 주도적으로 설계한 것은 아니었지만, 처음으로 DB 집계 쿼리를 직접 수정하는 작업에 참여한 경험이었다. "화면에서 발생하는 증상의 원인이 항상 프론트엔드에 있지 않으며, 해결 위치를 레이어 단위로 판단해야 한다"는 사실을 처음으로 체감한 순간이었다.

**결과**:
- 대시보드 수치와 실제 DB 데이터 정합성이 확보됐다.
- 담당자가 수치를 신뢰할 수 있는 기준이 마련되어 화면 간 수치 불일치로 인한 혼란이 해소됐다.

### 3. JSP+jQuery 레거시 코드 — Vue.js 컴포넌트 구조 전환

**현상**

JSP + jQuery 기반 코드는 전역 스크립트 방식으로 작성되어 있었다. 기능 추가나 수정 시 어디까지 영향을 받는지 파악하기 어려운 구조였으며, UI 렌더링과 데이터 조작 로직이 동일 파일에 혼재했다.

| 구분 | JSP + jQuery (기존) | Vue.js 컴포넌트 구조 (선택) |
|---|---|---|
| 기능 수정 영향 범위 | 전역 스크립트 의존 — 수정 범위 파악 어려움 | 컴포넌트 단위로 경계 명확 |
| UI와 로직 분리 | 동일 파일에 DOM 조작과 비즈니스 로직 혼재 | 컴포넌트 단위 반응형 상태 관리 |
| 유지보수 진입 비용 | 전체 스크립트 맥락 파악 필요 | 수정 대상 컴포넌트만 이해하면 됨 |

**원인**:
- JSP 구조는 UI와 데이터 조작 로직이 전역 스크립트에 혼재하여 컴포넌트 단위의 기능 경계가 없었다.

**해결 과정**:

JSP + jQuery 기반 코드를 Vue.js 컴포넌트 기반 구조로 전환하여 기능별 수정 범위가 컴포넌트 단위로 명확해지도록 재구성했다.

```vue
<template>
  <div class="monitoring-panel">
    <StatusCard 
      v-for="item in monitoringData" 
      :key="item.id"
      :data="item"
    />
  </div>
</template>

<script setup lang="ts">
import { ref, onMounted } from 'vue';
import StatusCard from '@/components/StatusCard.vue';

interface MonitoringData {
  id: number;
  name: string;
  status: 'normal' | 'warning' | 'error';
}

const monitoringData = ref<MonitoringData[]>([]);

onMounted(async () => {
  await fetchMonitoringData();
});
</script>
```

**결과**:
- JSP 전역 스크립트에서 Vue.js 컴포넌트 경계로 전환되어 기능별 수정 범위가 컴포넌트 단위로 명확해졌다.
- 중복 코드가 제거되어 유지보수 진입 비용이 감소했다.

## 기술적 판단 & 설계 포인트

### 1. 폴링 로직 순수 함수 분리 — 간격 기준의 단위 검증 가능 구조 설계

`fetchMonitoringData` 자체는 API 호출을 포함하므로 단위 테스트가 어렵지만, throttle 간격 설정값과 실제 호출 횟수의 관계는 mock 타이머를 이용해 독립적으로 검증할 수 있다. Vue 컴포넌트 마운트 없이 순수 타이머 동작만 격리해 검증하는 것이 핵심이다.

```typescript
import { vi, describe, it, expect, beforeEach, afterEach } from 'vitest';
import { throttle } from 'lodash-es';

describe('모니터링 폴링 간격 검증', () => {
  beforeEach(() => vi.useFakeTimers());
  afterEach(() => vi.useRealTimers());

  it('5초 간격 throttle 적용 시 10초 동안 최대 2회만 호출된다', () => {
    const mockFetch = vi.fn();
    const throttled = throttle(mockFetch, 5000);

    // 1초마다 10번 호출 시도
    for (let i = 0; i < 10; i++) {
      throttled();
      vi.advanceTimersByTime(1000);
    }

    // throttle 적용으로 실제 호출은 2회 이하
    expect(mockFetch).toHaveBeenCalledTimes(2);
  });
});
```

> throttle 간격 상수를 비즈니스 요구(5초)와 1:1로 대응시키고, 그 동작을 컴포넌트 외부에서 독립적으로 검증할 수 있도록 설계했다. 간격 기준이 바뀌어도 이 검증 코드의 기대값 하나만 수정하면 되는 구조다.

## 회고

### 잘한 점
- **도메인 특성 기반 트레이드오프 분석**: "실시간이어야 한다"는 막연한 요구사항을 공장 설비 도메인의 실제 상태 변화 주기와 이상 감지 대응 구조에 비춰 검토하고, 5초라는 기준을 현장 담당자와 합의하여 결정했다. 주어진 요구사항을 그대로 구현하는 것과, 도메인을 이해하고 요구사항의 전제를 다시 검토하는 것의 차이를 신입 시절에 처음으로 경험했다.
- **문제 원인을 화면 너머로 추적**: 대시보드 수치 불일치를 처음에는 프론트엔드 렌더링 버그로 의심했지만, 포기하지 않고 원인을 파고들어 DB 집계 쿼리 기준의 불일치를 찾아냈다. 백엔드 개발자분의 도움을 받아 함께 해결했지만, 증상의 근본 원인을 끝까지 추적한 경험 자체가 커리어 초반에 중요한 학습이었다.

### 아쉬운 점
- Vue.js 전환 범위를 우선순위가 높은 화면에 집중하다 보니, 일부 JSP 레거시 코드가 전환되지 않은 채 남아 두 기술 스택이 공존하는 상태로 프로젝트가 종료됐다. 처음부터 전환 우선순위 기준(트래픽·변경 빈도·유지보수 비용)을 명시적으로 설정하고 점진적 전환 계획을 문서화했다면 더 일관된 코드베이스를 남길 수 있었다.
- DB 집계 쿼리 수정은 백엔드 개발자분의 안내 아래 제한적으로 참여한 수준이었다. 왜 그 집계 기준이 문제인지, 백엔드 아키텍처 전체에서 어떤 의미를 갖는지를 당시에는 완전히 이해하지 못한 채 결과만 확인하는 수준이었다. 이 아쉬움이 이후 백엔드 구조를 스스로 이해하고 싶다는 강한 동기가 됐다.

### 배운 점
- **화면 너머의 시스템 파이프라인에 처음으로 눈을 뜨게 된 씨앗**: 커리어 첫 프로젝트에서 프론트엔드 개발자로 시작해 폴링 주기를 최적화하는 과정에서 도메인 특성과 서버 부하 트레이드오프를 처음으로 분석했고, 대시보드 정합성 문제를 해결하기 위해 처음으로 백엔드 DB 집계 레이어에 발을 들였다. 스스로 주도한 것은 아니었고, 백엔드 개발자분의 안내를 받아 참여한 수준이었지만 — 이 경험이 "내가 만든 화면이 어디서 데이터를 받아서 어떻게 동작하는지"에 대한 궁금증을 처음으로 심어주었다. 이 씨앗이 이후 MES 프로젝트에서 백엔드 개발에 더 적극적으로 참여하게 만든 동력이 됐고, 궁극적으로 풀스택 개발자로의 전향을 결심하게 만든 커리어의 출발점이었다.
