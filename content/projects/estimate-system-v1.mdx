> **프로젝트 개요**
>
> Vue 3.4 + TypeScript + Node.js/Express 풀스택으로 스마트팜 박람회·영업 현장에서 즉각적인 예상 견적 제공과 상담일지 기록을 단독으로 설계·구축한 MVP 시스템입니다. DB 스키마 설계부터 Vue TypeScript 모듈 분리, Nodemailer 이메일 인프라, Multer 파일 업로드, Capacitor Android 앱 패키징, Nginx 배포까지 전 스택 파이프라인을 단독으로 완성했습니다.
>
> **프로젝트 성격**: 박람회 현장이라는 비즈니스 제약(네트워크 불안정, 고객 앞 즉각 응대, Android 태블릿 운용)을 아키텍처 설계에 직접 반영하고, 운영 중 발견한 하드코딩 한계를 2차 견적 시스템 개발의 출발점으로 삼은 단독 풀스택 MVP

## 문제 상황 & 해결

### 1. html2canvas 기반 이메일 견적서 — 용량 초과 장애 및 렌더링 지연 해결

**현상**

견적 결과 화면을 html2canvas로 캡처하여 base64 이미지로 인코딩한 뒤 이메일에 첨부하는 구조였다. 초기 구현에서 전체 견적 화면을 그대로 캡처하여 발송하자 Gmail의 첨부 용량 제한을 초과하여 발송이 거부되는 케이스가 반복됐다. 복잡한 레이아웃 전체를 캡처하다 보니 렌더링에 10초 이상이 소요되어 고객 앞에서 기다리는 상황이 발생했다.

| 구분 | 개선 전 (base64 전체 화면 캡처) | 개선 후 (바이너리 변환 + 단순화 캡처) |
|---|---|---|
| 이메일 발송 성공 여부 | 용량 초과로 발송 거부 반복 | 발송 거부 케이스 미발생 |
| 렌더링 + 발송 소요 시간 | 10초 이상 소요 | 5초 이하로 단축 |
| 캡처 대상 DOM 복잡도 | 그래프·아이콘·전체 UI 요소 포함 | 핵심 견적 항목 요약 레이아웃만 포함 |
| 현장 응대 연속성 | 고객 앞 대기 상황 반복 발생 | 즉시 발송 흐름 현장 안착 |

**원인**:
- base64 인코딩은 바이너리 대비 약 33% 용량이 증가한다. 그래프·아이콘·전체 레이아웃을 포함한 화면을 캡처하면 이미지 크기가 제한을 초과하는 구조였다
- html2canvas는 DOM 렌더링 복잡도에 비례해 처리 시간이 늘어난다. 불필요한 UI 요소가 캡처 대상에 포함되면 지연이 심해지는 구조였다

**해결 과정**:

*백엔드*: 이메일 발송 API에서 base64 문자열을 이메일 본문에 인라인으로 삽입하지 않고, 바이너리 버퍼로 변환한 뒤 Nodemailer의 첨부 파일 옵션에 파일로 첨부하는 방식으로 변경했다. 인코딩 오버헤드 없이 실제 이미지 크기 그대로 첨부 파일로 처리되며, 이메일 클라이언트의 이미지 렌더링 호환성도 함께 개선됐다.

*프론트엔드*: 견적 결과 화면 중 이메일 전송 대상 영역을 별도 컴포넌트로 분리했다. 견적 금액·스마트팜 타입·면적·주요 항목 요약 등 핵심 항목만 포함한 단순화된 레이아웃으로 재구성하여 캡처 대상 DOM 크기를 줄이고 렌더링 부하를 낮췄다.

**결과**:
- 이메일 발송 실패 문제가 해소되어 이후 박람회 기간 중 발송 거부 케이스가 미발생했다
- 렌더링 + 발송 소요 시간이 10초 이상에서 5초 이하로 단축됐다
- 고객 앞에서 즉시 견적서를 이메일로 발송하는 흐름이 현장에서 안정적으로 자리잡혔다

### 2. 상담자 수기 기록의 디지털 보존 — Multer 기반 이미지 격리 저장

**현상**

상담 과정에서 상담자는 태블릿 화면에 고객의 토지 형태·보유 건물 정보·특이사항 등을 직접 수기로 작성했다. 이 수기 메모는 상담 종료 후 태블릿에서 사라질 위험이 있었고, 상담 헤더 레코드의 컬럼에 이미지를 직접 저장하면 이미지 갱신 시마다 상담 레코드 수정이 필요한 강결합 구조가 됐다.

| 저장 방식 | 이미지 갱신 영향 | 데이터 무결성 | 사후 추적 가능 여부 |
|---|---|---|---|
| 상담 레코드 컬럼에 직접 저장 | 이미지 갱신 시 상담 레코드 수정 필요 | 갱신 생명주기 불일치로 무결성 손상 위험 | 이미지와 상담 데이터 결합으로 조회 복잡 |
| 이미지 테이블 분리 + 격리 저장 (선택) | 이미지 갱신이 상담 레코드에 영향 없음 | 상담·이미지 생명주기 완전 독립 | 상담 ID 기준 URL 참조로 즉각 교차 추적 가능 |

**원인**:
- 수기 메모를 디지털로 보존하는 저장 구조가 초기 설계에 없었다
- 상담 데이터(날짜·담당자·유형)와 이미지 자산의 갱신 생명주기가 다르므로, 동일 레코드에 함께 관리하면 수정 시마다 상담 기록 자체가 변경되는 데이터 무결성 문제가 발생하는 구조였다

**해결 과정**:

*백엔드*: Multer 미들웨어로 multipart/form-data 요청을 처리하고, 캡처된 이미지를 상담 ID 기반 파일명으로 서버 업로드 전용 디렉토리에 저장했다. Express static 미들웨어로 해당 디렉토리를 정적 파일 경로로 서빙하여 이미지를 URL로 접근할 수 있게 구성했다.

*데이터 모델*: 상담 수기 기록 이미지를 별도 테이블로 분리하고 상담 헤더 FK를 참조하는 1:1 관계로 설계했다. 이미지 경로와 상담 헤더 외래키를 해당 테이블에 저장하여, 이미지 갱신이 상담 레코드에 전혀 영향을 주지 않는 독립 구조를 확보했다.

**결과**:
- 상담자가 현장에서 직접 작성한 수기 메모가 상담 ID 단위로 격리·저장됐다
- 상담 목록 조회 시 이미지 URL을 이미지 테이블에서 바로 참조할 수 있어, 사후 고객 후속 관리 시 상담 당시의 맥락을 이미지 형태로 즉각 교차 추적할 수 있게 됐다
- 이미지 파일 갱신이 상담 레코드 수정과 완전히 분리되어 데이터 무결성이 유지됐다

## 기술적 판단 & 설계 포인트

### 1. 견적 계산 로직의 프론트엔드 집중화 — 박람회 오프라인 환경 대응

박람회·현장 영업 환경은 네트워크가 불안정하거나 데이터 통신이 제한되는 경우가 빈번하다. 견적 계산을 서버 API로 처리하면 통신 오류 시 핵심 기능이 멈추고, 매번 왕복 지연이 발생해 고객 앞에서의 즉각적인 응대가 어렵다.

**결정**: 평수·타입·단수 기반 전체 견적 계산, 자재 수량 계산, 단가 계산, 연간 운영비 계산을 백엔드 없이 클라이언트 내부에서 완결되는 견적 계산 코어 모듈로 구현했다. 상수·단가 데이터는 견적 도메인 상수 모듈로 분리하여 계산 로직과 데이터 모델의 역할을 명확히 나눴다.

| 방식 | 네트워크 장애 시 | 응대 속도 | 백엔드 의존도 |
|---|---|---|---|
| 서버 API 계산 | 견적 기능 전체 중단 | 왕복 지연으로 즉각 응대 불가 | 모든 계산에 API 호출 필요 |
| 프론트엔드 코어 모듈 집중화 (선택) | 네트워크 없이도 즉시 계산 가능 | 입력 즉시 전 항목 실시간 갱신 | 저장·이메일 발송 시에만 관여 |

> 네트워크 연결 없이도 입력 즉시 전체 항목별 견적과 연간 운영비가 계산되어 화면에 표시되며, 백엔드는 최종 결과 저장과 이메일 발송에만 관여하는 구조가 됐다.

### 2. PostgreSQL 관계형 스키마 설계 — 고객·견적·상담 이력 추적 구조 확보

고객 한 명이 박람회 방문 중 여러 번 견적을 조회하거나 상담을 여러 건 남길 수 있다. 이를 단일 테이블로 관리하면 이력 추적과 데이터 정합성이 무너진다. 고객·견적·상담·수기 이미지의 갱신 생명주기가 각각 다르므로 독립 엔티티로 분리했다.

| 테이블 | 역할 | 관계 |
|---|---|---|
| 고객 마스터 | 방문 고객 기본 정보 | — |
| 견적 이력 | 타입·면적·항목별 견적 결과 이력 | 고객 마스터 FK (1:N) |
| 상담 헤더 | 상담 날짜·담당자·유형 정보 | — |
| 상담 수기 기록 이미지 | 태블릿 직접 작성 메모 이미지 경로 | 상담 헤더 FK (1:1) |

> 고객 ID 하나로 전체 견적·상담 이력을 조회할 수 있는 구조가 됐고, 수기 이미지와 상담 데이터를 분리하여 이미지 갱신 시 상담 레코드를 수정하지 않아도 되는 독립 구조를 확보했다.

### 3. Nodemailer + Gmail SMTP — MVP 단계 비용 최적화 이메일 인프라 구축

별도 메일 서버나 유료 발송 SaaS를 도입하면 운영 비용과 설정 공수가 발생한다. 영업 현장용 MVP에서는 안정성과 비용을 동시에 고려해야 했다.

**결정**: Gmail 계정의 앱 비밀번호(App Password)를 환경 변수로 관리하고 Nodemailer로 SMTP 발송 서비스를 구성했다.

| 옵션 | 비용 | MVP 적합성 | 확장 가능성 |
|---|---|---|---|
| 유료 발송 SaaS | 발송량 기반 과금, 초기 설정 공수 발생 | 발송량이 낮은 MVP 단계에서 비용 대비 효과 낮음 | 고 |
| Nodemailer + Gmail SMTP (선택) | 무료, 즉시 사용 가능 | MVP 발송량에 완전 적합 | 환경 변수 교체로 SMTP 제공자 전환 가능 |

> 앱 비밀번호 방식으로 2단계 인증을 우회하지 않으면서도 프로그래매틱 발송이 가능하다. 추가 비용 없이 이메일 견적서 발송 기능을 안정적으로 운영했으며, 발송량이 늘어날 경우 환경 변수만 교체하면 SMTP 제공자를 전환할 수 있는 확장 가능한 구조다.

### 4. Capacitor 6 — 단일 코드베이스 Android 하이브리드 앱 패키징

박람회 현장에서의 사용 방식은 두 가지였다. 고객에게는 QR코드를 통해 웹 링크로 접근하게 하고, 상담자는 회사 보유 Android 태블릿으로 고객 정보를 입력하고 수기 메모를 작성하는 구조였다. 회사 보유 기기가 Android 위주였기에 iOS 대응은 처음부터 고려 대상이 아니었다.

**결정**: Vue 3 SPA 코드베이스를 유지하면서 Capacitor 6를 도입해 Android 앱으로 패키징했다. 하나의 Vue 코드베이스로 웹과 Android 태블릿 앱 두 가지 환경을 동시에 지원하는 구조가 됐다.

| 구성 방식 | 태블릿 전체 화면 운용 | 네이티브 기능 접근 | 코드베이스 관리 |
|---|---|---|---|
| 웹 앱 직접 사용 | 전체 화면 활용 제약 | 네이티브 기능 접근 제한 | 단일 |

## 회고

### 잘한 점
- **현장 제약 기반 아키텍처 설계**: 박람회라는 특수 환경(네트워크 불안정, 즉각 응대, Android 태블릿)을 기술 설계에 직접 반영하여 견적 계산을 프론트엔드에 집중화한 판단이 현장 운영 안정성의 핵심이었다
- **전 스택 파이프라인 단독 완성**: DB 스키마 설계부터 Vue TypeScript 모듈 분리, Nodemailer 이메일 인프라, Multer 파일 업로드, Capacitor Android 패키징, Nginx 배포까지 전 구간을 단독으로 설계하고 박람회 일정에 맞춰 완성했다
- **html2canvas 장애 이중 대응**: 백엔드의 base64→바이너리 변환과 프론트엔드의 캡처 레이아웃 단순화를 동시에 적용하여 발송 거부 케이스를 완전히 제거하고 현장 응대 흐름을 안착시켰다

### 아쉬운 점
- 견적 단가와 프리셋 데이터가 견적 도메인 상수 모듈에 하드코딩되어 있어, 영업 상황에 따라 단가·항목이 변경될 때마다 코드 수정 및 재배포가 필요했다. 이 구조적 한계를 운영 중 직접 체감하면서, 영업팀이 화면에서 프리셋 단가와 계산 규칙을 직접 관리할 수 있는 동적 구조를 도입한 2차 견적 시스템 개발로 이어지는 계기가 됐다
- 박람회 기간이 촉박하여 자동화 테스트를 작성하지 못하고 수동 E2E 방식으로 전체 기능을 검증했다. 평수·타입·단수에 따른 분기가 많고 항목별 계산이 복잡한 견적 계산 코어 모듈 특성상, 핵심 계산 함수에 대한 단위 테스트 도입의 필요성을 이후 수정 시마다 수동으로 전체 케이스를 재검증해야 하는 비용을 통해 절실히 체감했다
- 백엔드 단일 파일에 라우터·DB 쿼리·비즈니스 로직이 혼재되어 있어, 기능 추가 시 가독성이 떨어졌다. 도메인별 라우터·서비스 레이어 분리의 필요성을 인식하고, 이를 2차 시스템 설계 단계에서 적극 반영했다

### 배운 점
- **비즈니스 제약이 설계 기준**: 박람회 현장이라는 운용 환경의 구체적인 제약 조건이 견적 계산 프론트엔드 집중화라는 핵심 아키텍처 결정의 근거가 됐다. 기술 선택의 기준은 기능 구현 가능성보다 비즈니스 환경 적합성이 먼저라는 것을 이 프로젝트에서 체득했다
- **MVP 범위 판단**: Nodemailer + Gmail SMTP 선택처럼, MVP 단계에서 '지금 필요한 안정성'과 '나중에 교체 가능한 구조' 사이에서 균형 잡힌 판단을 내리는 능력이 단독 개발에서 특히 중요함을 배웠다
- **운영 한계가 다음 시스템의 출발점**: 하드코딩 단가 프리셋이라는 운영 한계를 직접 경험하면서 동적 단가 관리를 도입한 2차 견적 시스템으로 발전시키는 사이클을 경험했다. 1차 시스템에서 체감한 구조적 문제가 2차 시스템의 설계 요건이 되는 개발 과정은, 반복적 개선을 통해 시스템이 성숙해가는 방식을 실전에서 경험한 사례였다
