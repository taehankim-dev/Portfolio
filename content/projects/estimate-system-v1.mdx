> 스프레드시트 기반 수작업 견적으로 인한 계산 오류와 시간 소요를 해결하기 위한 프로젝트입니다. 영업 현장에서 즉시 예상 견적을 제공할 수 있는 웹 기반 시스템을 단독 개발했으며, Capacitor를 활용해 하이브리드 앱으로 확장했습니다.

## 문제 상황 & 해결 경험

**문제**: 스프레드시트 기반 수작업 견적으로 인한 계산 오류 및 시간 소요, 배포 후 이메일 전송 실패 및 페이지 로딩 지연

**역할**: 풀스택 개발 / 하이브리드 앱 전환 / 단독 개발

**결과**:
- 렌더링 속도 10초 이상 → 5초 이하로 개선
- 영업 현장에서 실사용
- 견적 산출 오류 감소

---

## 문제 상황 & 해결

### 1. 이메일 이미지 용량 초과

**현상**: 예상 견적 결과를 이메일로 전달하는 과정에서 이미지 용량 초과로 발송 거부

**원인**:
- 견적 결과 화면을 그대로 캡처하여 이메일 첨부
- 이메일 서비스의 이미지 용량 제한 초과

**해결**:
- 레이아웃 단순화: 불필요한 요소 제거
- 핵심 정보 위주 재구성: 평수, 총 견적 금액, 주요 자재 항목만 포함
- 이미지 압축: 캡처 시 해상도 조정

**결과**:
- 이메일 발송 실패 문제 해소
- 영업 현장에서 견적 전달 안정성 확보

### 2. 페이지 로딩 속도 지연

**현상**: 견적 계산 결과 페이지의 초기 로딩 속도가 느려 현장 상담 흐름이 끊김

**원인**:
- 단일 컴포넌트에서 자재·인건비·이익률 계산을 모두 순차적으로 실행
- `loading = true` 상태로 모든 계산이 끝날 때까지 화면 전체가 블로킹 — 사용자에게 빈 화면이 노출
- 마지막 계산이 완료되어야만 `v-if` 조건이 해제되어 렌더링 시작

**해결**:

섹션별 컴포넌트로 분리하여 각 영역이 자신의 데이터가 준비되는 즉시 독립적으로 렌더링되도록 구조를 변경했다. 전체 계산이 끝날 때까지 기다리는 단일 대기 상태를 제거한 것이 핵심이다.

```vue
<!-- Before: 전체 계산 완료 전까지 화면 전체 블로킹 -->
<template>
  <div v-if="!loading">
    <!-- 모든 항목이 한꺼번에 표시 -->
  </div>
</template>

<script setup>
const loading = ref(true);

onMounted(async () => {
  // 순차 실행 — 가장 느린 계산이 전체 렌더링을 지연시킴
  await calculateMaterials();
  await calculateLabor();
  await calculateProfit();
  loading.value = false; // 여기까지 와야 화면이 뜸
});
</script>
```

```vue
<!-- After: 섹션별 독립 렌더링 — 준비된 영역부터 순차 표시 -->
<template>
  <div>
    <!-- 각 섹션이 자체 로딩 상태를 갖고 독립적으로 렌더링 -->
    <MaterialSection />
    <LaborSection />
    <ProfitSection />
  </div>
</template>

<script setup lang="ts">
// 부모 컴포넌트는 섹션 배치만 담당
// 계산 로직과 로딩 상태는 각 섹션 컴포넌트 내부에서 관리
// → 자재 계산이 끝나면 MaterialSection이 먼저 표시, 나머지는 순차 렌더링
</script>
```

**결과**:
- 초기 렌더링까지 걸리는 시간: **10초 이상 → 5초 이하**
- "아무것도 안 보이는" 대기 상태 제거 — 현장 상담 중 사용자 신뢰도 향상

## 테스트 용이성 설계

### 섹션 컴포넌트 분리가 테스트 격리를 달성한다

단일 컴포넌트 구조에서는 자재·인건비·이익률 계산이 하나의 `onMounted`에 묶여 있어, 자재 계산 로직 하나를 테스트하려 해도 전체 컴포넌트를 마운트해야 했다. 섹션별로 분리한 이후에는 각 컴포넌트의 계산 로직을 독립적으로 검증할 수 있다.

```typescript
// MaterialSection의 계산 로직을 컴포저블로 추출하면
// Vue 컴포넌트 마운트 없이 단위 테스트 가능
function useMaterialCalculation(materials: MaterialItem[], area: number) {
  const total = computed(() =>
    materials.reduce((sum, m) => sum + m.unitPrice * m.quantity * area, 0)
  );
  return { total };
}

// 테스트: Vue 환경 없이 계산 로직만 검증
describe('useMaterialCalculation', () => {
  it('단가 × 수량 × 면적의 합계를 반환한다', () => {
    const materials = [
      { unitPrice: 10000, quantity: 2 },
      { unitPrice: 5000, quantity: 3 },
    ];
    const { total } = useMaterialCalculation(materials, 10);

    // (10,000 × 2 × 10) + (5,000 × 3 × 10) = 200,000 + 150,000
    expect(total.value).toBe(350000);
  });
});
```

### 설계 원칙: 렌더링과 계산을 분리하면 테스트 대상도 명확해진다

컴포넌트가 "화면을 그리는 책임"과 "값을 계산하는 책임"을 함께 가지면 테스트 시 DOM 환경이 필수가 된다. 계산 로직을 컴포저블로 분리하면 순수 TypeScript 함수로서 단위 테스트할 수 있고, 이 프로젝트에서 확립한 이 패턴은 이후 2차 견적 시스템의 `EstimateCalculator` 설계에서도 동일하게 적용됐다.

## 결과

### 구조적 성과
- **렌더링 구조 개선**: 단일 블로킹 로딩 → 섹션별 독립 렌더링으로 전환, 초기 표시까지 10초 이상 → 5초 이하 달성
- **외부 제약의 구조적 흡수**: 이메일 서비스의 이미지 용량 제한을 우회하려 하지 않고, 핵심 정보만 담은 레이아웃으로 재설계하여 제약을 시스템 안으로 흡수 — 발송 성공률 안정화
- **실사용 검증**: 영업팀 박람회·상담 현장에서 실사용되어 스프레드시트 기반 수작업 견적 완전 대체 — 계산 오류 원인 구조적 제거
