## 문제 상황 & 해결

### 1. 협업사 인프라 연동 전 수제 독립 테스트 프론트 페이지를 통한 하드웨어 정합성 선제 검증

**현상**

협업사의 모바일 앱과 인프라 연동 개발이 완료되기 전, 내가 구현한 제어 API와 실물 하드웨어가 정상적으로 맞물려 동작하는지 독립적으로 검증할 수단이 없었다. 외부 의존 없이 백엔드 라우팅과 물리 기기 응답을 사전에 확인하지 않으면, 최종 연동 단계에서 문제가 터졌을 때 원인이 내 제어 서버인지 협업사 인프라인지 판별조차 어려운 구조였다.

| 구분 | 협업사 연동 단계까지 검증 보류 | 수제 독립 테스트 환경 구성 |
|---|---|---|
| 결함 식별 시점 | 최종 연동 단계에서 불량 노출 | 연동 전 백엔드·하드웨어 결함 조기 식별 |
| 원인 판별 난이도 | 내 제어 서버인지 협업사인지 불분명 | 독립 환경에서 책임 경계 명확히 분리 |
| 데드라인 리스크 | 연동 단계 돌발 장애 → 일정 지연 위험 | 사전 검증으로 연동 단계 장애 발생 최소화 |

**원인**:
- 협업사 인프라와 내 제어 서버가 결합되기 전까지 제어 API를 직접 트리거할 수단이 없었다. 외부 시스템 연동을 기다리는 수동적 대기 방식으로는 제어 서버의 내부 동작 정합성을 사전에 보장할 방법이 없었다.

**해결 과정**:

모든 액추에이터 제어 API를 각각 1:1로 트리거하는 버튼을 한 화면에 배치한 임시 독립 테스트 프론트 페이지를 수제 제작했다. 개발 노트북과 라즈베리파이 단말을 동일한 로컬 무선 네트워크에 연결하고, 브라우저에서 버튼을 하나씩 수동으로 눌러가며 각 액추에이터가 지연 없이 정상 반응하는지 E2E 교차 검증을 완수했다.

| 검증 대상 액추에이터 | 검증 방식 | 확인 기준 |
|---|---|---|
| 워터펌프 (물 분사) | 제어 API 1:1 버튼 수동 클릭 | 물 분사 즉각 반응 여부 |
| 도어 모터 (앞문·뒷문·상단) | 방향별 버튼 클릭 (정방향·역방향·정지) | 개폐 동작 및 자동 중지 타이머 정상 여부 |
| 환기팬 | 온·오프 버튼 클릭 | 팬 기동·정지 즉각 반응 여부 |
| 식물 생장 LED | 온·오프 버튼 클릭 | LED 점등·소등 즉각 반응 여부 |

**결과**:
- 협업사 인프라와 결합하기 전에 백엔드 라우팅 결함과 하드웨어 오작동 유무를 조기에 명확히 식별할 수 있었다.
- 최종 연동 단계에서 예상치 못한 기기 오작동 없이 전체 가동률을 안전하게 확보했다.

### 2. 박람회 현장 네트워크 단절 — 메모리 기반 오프라인 명령 큐로 제어 명령 유실 방지

**현상**

박람회장은 인터넷 인프라가 극도로 불안정한 환경이다. 와이파이 또는 이더넷 연결이 일시 단절되면 협업사 모바일 앱의 SSE 스트림 연결이 끊어지고, 그 사이에 전송된 제어 명령이 유실되어 하드웨어가 아무 반응도 하지 않는 치명적 장애가 발생할 수 있었다. 전시 중 하드웨어가 명령을 받지 못해 동작을 멈추는 상황은 전시 자체를 망치는 수준의 리스크였다.

| 단절 시나리오 | 대비 없는 경우 | 오프라인 명령 큐 적용 후 |
|---|---|---|
| SSE 스트림 연결 끊김 | 단절 구간 제어 명령 전부 유실 | 단절 구간 명령 큐에 적재, 재연결 후 순차 처리 |
| 클라이언트 재연결 | 수동 개입 없이는 재연결 불가 | 자동 재연결 루프로 단절-재연결 사이클 자율 동작 |
| 하드웨어 상태 동기화 | 명령 누락으로 잘못된 상태 지속 | 재연결 직후 순차 동기화로 올바른 상태 복원 |

**원인**:
- SSE 스트림 기반 통신 구조는 연결이 끊기는 순간 그 사이의 이벤트를 자동으로 보장하지 않는다. 현장 인프라 불안정성을 설계 단계에서 선제적으로 방어하지 않으면, 전시 가동률은 현장 네트워크 품질에 완전히 종속될 수밖에 없는 구조였다.

**해결 과정**:

백엔드 내부에 메모리 기반 오프라인 명령 큐를 구축했다. 스트림 연결이 끊어진 동안 수신됐어야 할 명령들을 큐에 적재하고, 재연결 즉시 순차 처리하여 하드웨어가 올바른 상태로 동기화되도록 구현했다. SSE 클라이언트에는 연결 실패 시 자동 재연결을 시도하는 루프를 이식하여, '단절-적재-재연결-순차 동기화' 자율 사이클이 외부 개입 없이 자동으로 동작하도록 했다.

**결과**:
- 전시 기간 동안 일시적 네트워크 단절이 발생하더라도 제어 명령이 유실되지 않고 재연결 후 정상 반영됐다.
- 전시 기간 내내 명령 누락으로 인한 하드웨어 오작동이 발생하지 않아 무장애 가동을 달성했다.

## 기술적 판단 & 설계 포인트

### 1. GPIO 직접 제어 → 아두이노 시리얼 경로로의 현실적 아키텍처 선회 — 배선을 뜯으며 하드웨어 불량을 판정하고 데드라인을 사수한 판단

협업사로부터 전달받은 핀맵 명세를 기반으로 라즈베리파이 GPIO 핀으로 모터와 릴레이를 직접 제어하는 로직을 구현했다. 그러나 실제 구동 테스트에서 명세대로 완성된 코드에도 하드웨어가 전혀 반응하지 않는 위기 상황에 직면했다.

소프트웨어 결함인지 하드웨어 물리 결함인지 판별하기 위해, 연결 점퍼선과 배선을 하나씩 직접 교체해가며 교차 검증을 진행했다. 협업사가 동봉한 아두이노 보드로 교체 연결했음에도 특정 모터가 여전히 동작하지 않는 것을 확인했다. 소프트웨어 경로를 바꿔도 동일한 증상이 재현된다는 사실에서 문제의 본질이 코드가 아닌 '하드웨어 자체의 물리적 단선 및 모터 불량'임을 최종 판정하고 협업사에 긴급 수리를 요청했다.

**결정**: 수리 완료 후 박람회 개막까지 남은 시간이 턱없이 부족했다. 라즈베리파이 GPIO 직접 제어 코드를 처음부터 재검증하기에는 리스크가 너무 컸다. 아두이노 보드단에 이미 내장되어 안정성이 검증된 시리얼 명령 프로토콜(USB Serial, Baud Rate 9600)을 그대로 활용하여 FastAPI 제어 서버와 직접 중계 연동하는 경로를 택했다.

```
[협업사 인프라 / 모바일 앱]
    │  단방향 SSE 스트림 (제어 신호 푸시)
    ↓
[라즈베리파이 — 독립 프로세스로 내장 구동]
 FastAPI 제어 서버 + SSE 스트림 수신 클라이언트
    │  USB 시리얼 통신 (Baud Rate 9600)
    ↓
[아두이노 보드 — 물리 신호 직접 구동]
 워터펌프 / 도어 모터 / LED / 환기팬 액추에이터
```

| 항목 | GPIO 직접 제어 재검증 경로 | 아두이노 시리얼 경로 (선택) |
|---|---|---|
| 검증 상태 | 수리 후 재검증 필요 — 불확실 | 이미 현장 검증 완료 — 안정성 확인된 경로 |
| 데드라인 리스크 | 추가 디버깅 소요 시간 → 일정 지연 위험 | 즉시 적용 가능 → 납기 준수 |
| 기술 부채 | 미검증 경로 재시도로 리스크 적재 | 검증된 경로 활용으로 리스크 최소화 |

> 기존 계획에 집착하지 않고 이미 증명된 경로로 신속하게 이동한 이 판단이 데드라인 준수를 가능하게 했다. 이미 현장 검증된 경로를 활용한 덕분에 추가 하드웨어 디버깅 없이 전시 일정에 맞춰 안정적인 시스템 가동을 완료했다.

### 2. 단방향 SSE 수신 아키텍처 — 협업사 규격 최하단을 수용한 다운스트림 명령 중계

전체 연결 구조에서 내가 구현한 제어 서버는 Unity와 직접 통신하지 않는다. `[Unity ↔ 협업사 인프라 서버 ↔ 협업사 모바일 앱]`으로 이어지는 협업사 인프라의 최하단에서, 모바일 앱이 발신하는 SSE 제어 신호를 수신하여 물리 하드웨어 명령으로 변환하는 역할에 철저히 집중했다. 역방향 하드웨어 피드백 전송은 설계 범위에 포함하지 않아 구현 복잡도를 현실적으로 관리했다.

**결정**: 협업사가 제공한 초기 규격 가이드를 기반으로 aiohttp 비동기 SSE 클라이언트를 구현하고, 수신된 이벤트를 FastAPI 하드웨어 제어 파이프라인으로 즉각 변환하는 단방향 중계 구조를 설계했다.

SSE 스트림으로 수신되는 제어 프로토콜은 액추에이터 유형(`type`)과 동작 모드(`mode_type`)를 포함하는 JSON 구조였다. 클라이언트는 수신 즉시 해당 필드를 파싱하여 아두이노 시리얼 명령 문자열로 변환한 뒤 USB 시리얼 채널로 전송한다.

| 처리 단계 | 역할 | 구현 기술 |
|---|---|---|
| SSE 스트림 수신 | 협업사 앱 발신 제어 이벤트 비동기 수신 | aiohttp 비동기 클라이언트 |
| 이벤트 파싱 | JSON 페이로드에서 액추에이터·모드 추출 | asyncio 이벤트 루프 |
| 시리얼 명령 변환·전송 | 제어 신호 → 아두이노 USB 시리얼 명령 | PySerial (Baud Rate 9600) |

> 협업사 모바일 앱에서 전송된 제어 신호가 SSE 스트림을 통해 라즈베리파이 제어 서버에 실시간으로 도달하고, 즉시 아두이노 시리얼 명령으로 변환되어 물리 하드웨어를 구동하는 단방향 파이프라인이 완성됐다.

### 3. FastAPI 액추에이터별 독립 제어 엔드포인트 설계 — 단일 제어 API의 의존성 폭발 방지

액추에이터별 특성이 상이하다. 도어 모터는 정방향·역방향·정지라는 3가지 방향 명령과 자동 중지 타이머가 필요하고, 워터펌프·LED·환기팬은 On/Off 이진 제어로 충분하다. 단일 통합 제어 엔드포인트로 이 모든 동작을 처리하면, 타입별 분기 로직이 하나의 함수에 집중되어 특정 액추에이터 로직 수정이 전체 제어 파이프라인에 영향을 미치는 결합도 위험이 생긴다.

**결정**: 액추에이터 유형별로 독립 제어 엔드포인트를 설계하여 각 라우터가 단일 하드웨어 도메인에만 집중하도록 했다.

| 액추에이터 | 제어 방식 | 특이사항 |
|---|---|---|
| 워터펌프 | On/Off 이진 제어 | — |
| 식물 생장 LED | On/Off 이진 제어 | — |
| 환기팬 | On/Off 이진 제어 | — |
| 앞문·뒷문 (DC 모터) | 정방향·역방향·정지 3방향 제어 | 자동 중지 타이머 로직 내장 |
| 상단 도어 (서보 모터) | 개방·폐쇄 방향 제어 | 자동 중지 타이머 로직 내장 |

> 특정 액추에이터의 제어 로직이 변경되어도 해당 엔드포인트만 수정하면 되고, 다른 액추에이터 제어 흐름에는 영향을 미치지 않는다. 코드 전 구간에 레이어별 Try-Catch 예외 처리 파이프라인을 적용하여 시리얼 통신 오류·SSE 스트림 단절·액추에이터 응답 실패가 전체 프로세스를 다운시키지 않도록 격리했다.

### 4. Linux systemd 상시 가동 체계 — 박람회 무인 운영 인프라

박람회 전시 환경에서 제어 서버는 24시간 운영되어야 하며, 전원 차단·예외 발생·재부팅 상황에서도 사람이 개입하지 않고 자동 복구되어야 한다. 수동 재시작이 필요한 구조는 전시장에서 현실적으로 운영 불가능하다.

**결정**: FastAPI 제어 서버를 라즈베리파이 호스트 OS의 systemd 배경 서비스로 등록했다. 프로세스가 예외로 종료되거나 전원이 차단된 후 재부팅되더라도 자동으로 서비스가 복구되는 자가 치유 무인 운영 체계를 구성했다.

| 장애 시나리오 | systemd 미적용 시 | systemd 적용 후 |
|---|---|---|
| 예외 발생으로 프로세스 종료 | 수동 재시작 필요 | 자동 재시작 정책으로 즉각 복구 |
| 전원 차단 후 재부팅 | 서비스 미가동 상태로 방치 | 부팅 완료 후 자동 서비스 기동 |
| 전시장 상주 인원 부재 | 원격 접속 또는 직접 방문 필요 | 개입 없이 24시간 자율 운영 |

> 전시 기간 내내 제어 서버는 라즈베리파이 부팅과 함께 자동 기동되었고, 운영 중 예외 발생 시에도 자동 복구되어 무인 운영 환경을 유지했다.

## 회고

### 잘한 점
- **배선 교차 검증을 통한 하드웨어 물리 불량 판정**: 하드웨어가 반응하지 않을 때 코드를 계속 의심하는 대신 점퍼선을 하나씩 직접 교체해가며 교차 검증을 진행하고, 소프트웨어 경로를 바꿔도 동일 증상이 재현된다는 사실에서 물리 불량임을 최종 판정한 것은 임베디드 도메인 경험 없이 내린 실리적 현장 판단이었다.
- **검증된 경로로의 신속한 아키텍처 선회**: 수리 후 데드라인이 촉박해진 상황에서 불확실한 GPIO 재검증 경로에 집착하지 않고, 이미 현장에서 안정성이 확인된 아두이노 시리얼 경로로 즉시 이동한 것이 전시 일정 준수를 가능하게 했다. 기존 계획보다 결과를 우선에 두는 결정이었다.
- **협업사 연동 전 선제 검증 환경 구축**: 외부 의존을 기다리는 대신 수제 테스트 프론트 페이지를 직접 제작하여 리스크를 스스로 통제한 것은 단독 개발자로서의 자율적 품질 관리였다.

### 아쉬운 점
- 현재 구조는 협업사 앱에서 제어 서버로의 단방향 명령 수신만 구현했다. 온습도 센서 수치나 아두이노 시리얼 응답을 협업사 인프라로 역방향 전송하는 피드백 파이프라인은 설계 범위에 포함하지 않았다. 실시간 센서 데이터를 앱 화면에 표시하거나 제어 완료 확인을 앱으로 회신하는 구조로 이어지지 못한 것은 향후 고도화 여지로 남아 있다.
- AI Agent(Claude Sonnet) 페어 프로그래밍으로 임베디드 도메인을 빠르게 돌파할 수 있었지만, 예상치 못한 하드웨어 예외 상황에서 독립적으로 디버깅 방향을 잡는 역량은 여전히 부족했다. 프로젝트 이후에도 임베디드 도메인 지식의 더 깊은 내재화 필요성을 체감하고 있다.

### 배운 점
- **기술을 화려하게 쌓는 것보다 주어진 물리적 제약과 시간적 한계 안에서 살아 움직이는 시스템을 만들기 위해 가장 현실적인 경로를 빠르게 찾아내는 실리적 판단력이 개발자의 핵심 역량이다**: 하드웨어가 반응하지 않을 때 코드 수정보다 배선 교차 검증이 먼저였고, 데드라인이 촉박할 때 불확실한 경로 재검증보다 이미 검증된 경로로의 선회가 먼저였다. 코드 완성도가 아닌 현장 판단력이 결과를 만든다는 것을, 임베디드 영역에서 직접 체득했다.
- **AI Agent와의 기민한 페어 프로그래밍으로 도메인 지식을 단기간에 확장하고 리스크를 스스로 통제할 수 있다**: 경험 제로 베이스에서 Claude Sonnet을 나침반으로 삼아 임베디드 제어 코드를 독학하며 박람회 데드라인을 지킨 경험은, AI Agent가 "대신 해주는 도구"가 아닌 "함께 생각하는 시니어"로 작동할 때 단독 개발자의 생산성과 책임감이 어디까지 확장될 수 있는지를 보여줬다. 이 방식의 핵심은 AI가 생성한 코드를 맹목적으로 수용하는 것이 아니라, 직접 배선을 뜯으며 결과를 교차 검증하는 주체적 책임감이었다.
