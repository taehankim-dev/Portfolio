## 문제 상황 & 해결

### 1. 하드웨어 궁합 문제 — 원인 진단 후 제어 방식 전환

**현상**: 라즈베리파이 단독 제어에서 모터가 안정적으로 동작하지 않음

**원인 진단**:
처음에는 Python 제어 코드의 타이밍 문제로 가정하고 코드를 수정했지만 개선되지 않았다. 문제를 소프트웨어로 가정하고 계속 코드를 수정하는 것은 근본 원인이 아닌 곳에 시간을 쓰는 것이었다. 모터 드라이버와 배선 구조를 확인한 결과, 라즈베리파이의 GPIO 출력 전류가 해당 모터 구동 요구사항을 충족하지 못하는 하드웨어 궁합 문제임을 확인했다.

**선택지 비교**:

| 방안 | 판단 |
|---|---|
| Python 코드 계속 수정 | 근본 원인(하드웨어)이 아닌 곳에 시간 투자 → 일정 내 해결 불가능 | ❌ |
| 모터 드라이버 교체 | 부품 조달 시간 필요 → 전시 일정 불가 | ❌ |
| **아두이노 기반으로 전환** | 제작사 제공 아두이노 자료 활용 → 검증된 하드웨어 경로 사용 | ✅ |

**결과**: 라즈베리파이가 고수준 제어 로직(FastAPI API 서버)을 담당하고, 아두이노가 저수준 하드웨어 신호를 담당하는 역할 분리 구조로 안정적인 동작 확보

### 2. FastAPI 제어 API — 전시 현장 원격 제어

전시 현장에서 담당자가 PC나 모바일로 미니어처를 제어할 수 있도록 라즈베리파이 위에 FastAPI 서버를 올렸다. 하드웨어 제어 명령을 HTTP API로 추상화하여 물리적 접근 없이 동작 제어가 가능하게 했다.

```python
from fastapi import FastAPI
import serial  # 아두이노와 시리얼 통신

app = FastAPI()
arduino = serial.Serial('/dev/ttyUSB0', 9600)

@app.post("/control/{device}")
async def control_device(device: str, action: str):
    """
    device: 'water_pump' | 'door' | 'led' | 'fan'
    action: 'on' | 'off'
    """
    command = f"{device}:{action}\n"
    arduino.write(command.encode())  # 아두이노로 명령 전달

    return {"device": device, "action": action, "status": "sent"}

@app.get("/status")
async def get_status():
    # 아두이노로부터 현재 상태 읽기
    arduino.write(b"status\n")
    response = arduino.readline().decode().strip()
    return {"status": response}
```

이 구조에서 라즈베리파이–아두이노 시리얼 통신 로직은 FastAPI 라우터와 분리되어 있어, 향후 하드웨어가 변경되더라도 API 인터페이스는 유지한 채 시리얼 통신 부분만 수정하면 된다.

## 결과

### 확인한 것들
- **원인 진단이 먼저**: 모터 불안정 문제를 소프트웨어로 가정하고 코드를 수정하는 것은 근본 원인이 아닌 곳에 시간을 쓰는 것이었다. 하드웨어 경로를 직접 확인하고 나서야 올바른 해결책으로 전환할 수 있었다 — "문제의 원인을 먼저 진단한 뒤 해결책을 선택한다"는 원칙을 하드웨어 환경에서도 재확인
- **추상화의 실용적 가치**: 라즈베리파이 위에 FastAPI를 올려 하드웨어 제어를 HTTP API로 추상화한 결정은, 단순히 원격 제어를 위한 것이기도 했지만 동시에 하드웨어 변경 시 API 인터페이스를 유지할 수 있는 구조를 만들었다
- **'동작 가능한 최소 구조' 우선**: 일정 압박 상황에서 이상적인 아키텍처보다 전시 환경에서 실제로 동작하는 구조를 우선하는 판단이 옳았다 — 제약 조건이 명확한 상황에서의 우선순위 결정 경험
