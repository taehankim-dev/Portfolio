## 아키텍처 결정

### 문제: 웹 API와 MQTT 수집이 동일 백엔드에서 경합

**선택지 비교**:

| 방안 | 장점 | 단점 | 선택 |
|------|------|------|------|
| 큐 기반 비동기 처리 | 기존 구조 유지, 코드 수정 최소 | 메모리 사용량 증가, 복잡도 상승, 근본 해결 아님 | ❌ |
| 스레드풀 크기 조정 | 즉시 적용 가능 | 센서 증가·트래픽 급증 시 동일 문제 재발 | ❌ |
| **백엔드 분리** | 책임 분리, 독립 스케일링, 장애 격리 | 인프라 복잡도 증가, 데이터 동기화 필요 | ✅ |

**결정 이유**:
- **Fault Tolerance**: Web API 서비스에 장애가 발생하더라도 MQTT Collector는 계속 센서 데이터를 수집 및 저장 → 농장 환경 데이터 손실 방지. 반대로 MQTT Collector 재기동 중에도 웹 제어 API는 영향 없이 응답 가능
- **수평 확장 독립성**: 농가 수·센서 수 증가 시 MQTT Collector만 scale-out하면 되고, Web API는 무중단 유지. 단일 서비스였다면 두 워크로드가 동일 리소스를 공유하므로 어느 한쪽의 부하 급증이 전체 응답성에 영향을 줌
- **장기 유지보수성**: 단기 인프라 복잡도 증가를 감수하더라도, 운영 중 발생하는 각 문제를 서비스 경계 안에서 독립적으로 해결할 수 있는 구조가 장기적으로 유리

## 문제 상황

### 1. 웹 제어 API 응답 지연

**현상**: MQTT 데이터 수집이 진행되는 동안 웹 제어 API 요청이 지연되는 문제

**원인**:
- MQTT 데이터 수집 로직이 동기적으로 처리되어 웹 제어 API가 후순위로 밀림
- 단일 백엔드에서 데이터 수집과 웹 API를 동시에 처리하면서 리소스 경쟁 발생

### 2. 알림 중복 발송

**현상**: 백엔드 재배포 시 이미 발송된 알림이 중복으로 재발송되는 문제

**원인**:
- 알림 발송 상태를 메모리에서만 관리
- 재기동 시 상태가 초기화되면서 이전 알림 기록이 소실

**영향**:
- 불필요한 SMS 비용 증가
- 사용자에게 중복 알림으로 인한 혼란 발생

### 그 외 운영 이슈

실시간 데이터 표시 문제(MQTT retain 설정으로 과거 데이터 우선 수신), HMI 센서 설정 값 노출 문제 등을 타임스탬프 검증과 입력 단위 검증으로 해결했습니다.

## 선택지와 판단

### 문제 1: 웹 API 응답 지연 해결 방안

| 방안 | 장점 | 단점 | 선택 |
|------|------|------|------|
| 비동기 처리 강화 | 코드 변경 최소화 | 근본적인 리소스 경쟁 미해결 | ❌ |
| 스레드풀 크기 증가 | 간단한 설정 변경 | 센서 증가 시 동일 문제 재발, 서버 리소스 한계 | ❌ |
| **백엔드 분리** | 장애 격리, 독립 스케일링, 워크로드 성격 분리 | 운영 복잡도 증가 | ✅ |

**선택 이유**:
- MQTT 데이터 수집(지속적·고빈도 I/O)과 웹 제어 API(사용자 요청 기반·저지연 요구)는 성격이 다른 워크로드 — 동일 프로세스에서 처리하면 한쪽의 부하가 다른 쪽의 응답성을 저하시킴
- **장애 격리**: Web API 장애 시 MQTT 수집 지속 가능, MQTT Collector 재기동 중에도 웹 제어 무중단 — 서비스 전체를 단일 장애점(SPOF)으로 만들지 않음
- **독립 스케일링**: 농가 수 증가 시 MQTT Collector만 scale-out, Web API 리소스는 유지 — Docker Compose 환경에서 서비스 단위 분리가 자연스럽게 이 구조를 지원

### 문제 2: 알림 상태 관리 방안

| 방안 | 장점 | 단점 | 선택 |
|------|------|------|------|
| Redis 캐시 | 빠른 조회 | 추가 인프라 필요 | ❌ |

**선택 이유**:
- 알림 발송 이력을 장기간 보관하여 문제 분석에 활용 가능

### 2. MQTT 데이터 수집 — 큐 버퍼링과 배치 저장 설계

MQTT 메시지는 센서마다 짧은 주기로 지속적으로 유입된다. 메시지가 들어올 때마다 DB에 개별 INSERT를 실행하면 커넥션을 과도하게 점유하고, Web API 서버와 커넥션 풀을 공유하는 상황에서는 제어 API 응답 지연으로 이어진다. 이를 해결하기 위해 **수신과 저장을 분리**하는 구조를 선택했다.

- **retain 메시지(과거 데이터)**: `ConcurrentLinkedQueue`에 담고 즉시 반환 → 수신 스레드 비점유
- **실시간 데이터**: 수신 즉시 DB에 저장 후 임계값 체크 → 알림 발송
- **`isOldData()`를 순수 함수로 분리**: 외부 의존성 없이 타임스탬프 값만으로 판단 → JUnit에서 경계값 테스트 용이

```java
@Service
public class MqttMessageRouter {

    // 비차단(non-blocking) 삽입을 지원하는 스레드 안전 큐
    // retain 메시지를 수신 스레드 점유 없이 보관
    private final ConcurrentLinkedQueue<MqttPayload> retainBuffer = new ConcurrentLinkedQueue<>();
    private final SensorDataRepository sensorDataRepository;
    private final AlertService alertService;

    @MqttMessageHandler(topic = "v1/farm/+/device/+/data")
    public void handleSensorData(MqttPayload payload) {

        if (isOldData(payload.getTimestamp())) {
            // retain 메시지: 큐에 담고 즉시 반환 → 수신 스레드 비점유
            // 실제 저장은 1분 주기 배치 작업이 담당
            retainBuffer.offer(payload);
            return;
        }

        // 실시간 데이터: 즉시 저장 후 임계값 체크
        sensorDataRepository.save(SensorData.from(payload));

        if (thresholdExceeded(payload)) {
            alertService.sendAlert(payload.getFarmId());
        }
    }

    // 순수 함수: LocalDateTime 값만으로 판단 → 단위 테스트에서 Mock 없이 검증 가능
    boolean isOldData(LocalDateTime timestamp) {
        return timestamp.isBefore(LocalDateTime.now().minusMinutes(5));
    }

    public ConcurrentLinkedQueue<MqttPayload> getRetainBuffer() {
        return retainBuffer;
    }
}
```

큐에 쌓인 retain 메시지는 1분 주기 스케줄러가 `JdbcTemplate.batchUpdate()`로 한 번에 저장한다. 수십 건의 개별 INSERT 대신 DB 통신 1회로 처리하므로 커넥션 점유 시간이 크게 줄어든다.

```java
@Component
public class SensorDataBatchProcessor {

    private final MqttMessageRouter router;
    private final JdbcTemplate jdbcTemplate;

    // 1분마다 큐를 비워 배치 저장 — DB 통신 횟수를 수집 건수의 1/N으로 감소
    @Scheduled(fixedDelay = 60_000)
    public void flushRetainBuffer() {
        List<MqttPayload> batch = new ArrayList<>();
        MqttPayload payload;
        while ((payload = router.getRetainBuffer().poll()) != null) {
            batch.add(payload);
        }
        if (batch.isEmpty()) return;

        jdbcTemplate.batchUpdate(
            "INSERT INTO sensor_data (farm_id, device_id, value, timestamp) VALUES (?, ?, ?, ?)",
            batch,
            batch.size(),
            (ps, p) -> {
                ps.setLong(1, p.getFarmId());
                ps.setString(2, p.getDeviceId());
                ps.setDouble(3, p.getValue());
                ps.setTimestamp(4, Timestamp.valueOf(p.getTimestamp()));
            }
        );
    }
}
```

### 3. 알림 상태 DB 저장

재배포 시에도 알림 상태를 유지하고 중복 발송을 방지하기 위해 최근 30분 이력을 DB에서 조회합니다.

```java
@Service
public class AlertService {
  
  public void sendAlert(Long farmId, String alertType) {
    // 최근 알림 발송 이력 확인
    Optional<AlertLog> recentAlert = alertLogRepository
      .findRecentAlert(farmId, alertType, LocalDateTime.now().minusMinutes(30));
    
    if (recentAlert.isPresent()) {
      log.info("Alert already sent within 30 minutes, skipping");
      return;
    }
    
    // SMS 발송
    smsService.send(farmId, alertType);
    
    // 발송 이력 저장
    alertLogRepository.save(AlertLog.builder()
      .farmId(farmId)
      .alertType(alertType)
      .sentAt(LocalDateTime.now())
      .build());
  }
}
```

## 테스트 용이성 설계

서비스 분리와 순수 함수 추출은 단순히 성능 문제를 해결하기 위한 선택이 아니었다. 각 컴포넌트가 **단독으로 테스트 가능한 구조**를 갖추게 되었다는 점도 중요한 설계 이득이다.

### AlertService — 멱등성 로직의 단위 테스트 격리

`AlertService.sendAlert()`의 핵심은 "30분 이내 이미 발송했으면 건너뛴다"는 멱등성 로직이다. `AlertLogRepository`를 주입받는 구조 덕분에 실제 DB 없이 Mock만으로 이 분기를 독립 검증할 수 있다.

```java
@ExtendWith(MockitoExtension.class)
class AlertServiceTest {

    @Mock AlertLogRepository alertLogRepository;
    @Mock SmsService smsService;
    @InjectMocks AlertService alertService;

    @Test
    @DisplayName("30분 이내 동일 알림 이력이 있으면 SMS를 발송하지 않는다")
    void skip_alert_when_recent_log_exists() {
        // given: 30분 이내 발송 이력이 있는 상황
        given(alertLogRepository.findRecentAlert(eq(1L), eq("TEMP_HIGH"), any()))
            .willReturn(Optional.of(new AlertLog()));

        // when
        alertService.sendAlert(1L, "TEMP_HIGH");

        // then: SMS 발송 메서드가 호출되지 않아야 함
        then(smsService).should(never()).send(any(), any());
    }

    @Test
    @DisplayName("발송 이력이 없으면 SMS를 발송하고 이력을 저장한다")
    void send_alert_and_save_log_when_no_recent_log() {
        // given: 이력 없음
        given(alertLogRepository.findRecentAlert(any(), any(), any()))
            .willReturn(Optional.empty());

        // when
        alertService.sendAlert(1L, "TEMP_HIGH");

        // then
        then(smsService).should(times(1)).send(eq(1L), eq("TEMP_HIGH"));
        then(alertLogRepository).should(times(1)).save(any(AlertLog.class));
    }
}
```

### MqttMessageRouter — 타임스탬프 경계값 테스트

`isOldData()`를 외부 의존성 없는 순수 함수로 분리했기 때문에, 파라미터 전달만으로 5분 경계의 모든 케이스를 검증할 수 있다.

```java
class MqttMessageRouterTest {

    private final MqttMessageRouter router = new MqttMessageRouter(/* mocks */);

    @Test
    @DisplayName("현재 시각으로부터 5분 초과된 타임스탬프는 과거 데이터로 판단한다")
    void old_data_when_timestamp_exceeds_5_minutes() {
        LocalDateTime oldTimestamp = LocalDateTime.now().minusMinutes(6);
        assertThat(router.isOldData(oldTimestamp)).isTrue();
    }

    @Test
    @DisplayName("5분 이내 타임스탬프는 실시간 데이터로 판단한다")
    void realtime_data_when_timestamp_within_5_minutes() {
        LocalDateTime recentTimestamp = LocalDateTime.now().minusMinutes(4);
        assertThat(router.isOldData(recentTimestamp)).isFalse();
    }
}
```

### 설계 원칙: 서비스 분리가 테스트 격리를 달성한다

| 컴포넌트 | 테스트 격리 방법 | 검증 대상 |
|---|---|---|
| `AlertService` | `AlertLogRepository` Mock | 멱등성 분기 (발송 여부 결정 로직) |
| `SensorDataBatchProcessor` | `JdbcTemplate` Mock | 배치 호출 여부 및 큐 비움 동작 |
| `MqttMessageRouter.isOldData()` | 순수 함수 직접 호출 | 타임스탬프 경계값 |

단일 백엔드 구조였다면 MQTT 수신 → DB 저장 → 알림 발송이 하나의 흐름으로 얽혀 있어, 알림 멱등성 하나를 테스트하기 위해 MQTT 세션과 DB를 모두 준비해야 했을 것이다. 수신 핸들러와 배치 저장을 분리함으로써 각 책임을 독립적으로 테스트할 수 있는 구조가 자연스럽게 만들어졌다.

## 결과

### 구조적 성과
- **응답 격리**: MQTT Collector를 별도 서비스로 분리한 뒤 Web API가 센서 수집 스레드와 리소스를 공유하지 않게 됨 → HMI 제어 요청이 데이터 수집 부하에 영향받지 않는 예측 가능한 응답 구조 확보
- **멱등성 보장**: 알림 발송 이력을 `AlertLog` 테이블에 영속화하여 재배포·재기동 후에도 상태 재조회 가능 → 인메모리 상태 소실로 인한 중복 발송 구조적 제거
- **데이터 정합성**: MQTT `retain` 플래그로 수신되는 과거 메시지를 타임스탬프 비교로 걸러냄 → 사용자에게 항상 현재 시점의 센서 값 표시 보장

### 설계 관점 성과
- MQTT 데이터 수집·알림·웹 제어 영역에서 발생 가능한 운영 리스크를 사전에 식별하고, 각각 서비스 분리 / DB 영속화 / 유효성 검증이라는 구조적 해결책으로 대응
- 센서 구성과 데이터 포맷이 표준화되지 않은 현장 환경에서, 즉각적인 실운영보다 구조적 안정성 검증을 우선하는 개발 전략을 선택 — 실운영 확장의 전제 조건(센서 표준화, 운영 기준 합의)을 기술적으로 명확히 함

## 회고

### 잘한 점
- **책임 분리 원칙 적용**: MQTT 데이터 수집과 웹 API를 분리하여 각 서비스의 독립성 확보
- **문제 원인 정확한 파악**: 메모리 기반 상태 관리 → DB 저장으로 근본적인 문제 해결
- **운영 환경 고려**: Docker를 활용한 표준화된 배포 환경 구성
- **사용자 경험 고려**: 입력 단위 검증, 가이드 문구, 기본값 설계로 사용자 실수 방지

### 아쉬운 점
- 실시간 시스템에서는 동기/비동기 경계를 명확히 나누지 않으면 사용자 제어 API 안정성에 직접적인 영향을 준다는 점을 초기에 인식하지 못함
- 향후 앱 형태로 확장 시 사용자 경험 개선 가능성 존재
- 이후 학습을 통해, retain 메시지 처리를 `ConcurrentLinkedQueue` 버퍼 방식이 아닌 `@Async` + 전용 스레드풀 구조로도 구현할 수 있다는 것을 알게 되었다. 해당 방식은 배치 주기 없이 메시지 수신 즉시 비동기 저장이 가능하고, 스레드풀 설정으로 DB 부하를 제어할 수 있다는 장점이 있다. 다만 실제로 적용해본 적은 없으며, Self-invocation 문제를 피하기 위한 빈 분리 등 구조적 고려가 추가로 필요하다는 점도 함께 인지하고 있다.

### 배운 점
- **실시간 IoT 시스템의 운영 리스크**: 운영 리스크를 구조로 분리해 안정화하는 것의 중요성
- **동기/비동기 처리 전략**: 실시간 시스템에서 동기/비동기 경계를 명확히 나누지 않으면 사용자 제어 API 안정성에 직접적인 영향을 준다는 점 체감
- **표준화의 중요성**: 센서 표준화 및 운영 기준 합의가 선행되어야 실운영 단계로 자연스럽게 확장될 수 있음
- **책임 분리 원칙**: 이 프로젝트에서 확립한 "성격이 다른 워크로드는 서비스 레벨에서 분리" 원칙은 이후 다른 실시간 시스템 설계에서도 아키텍처 기준으로 적용되었습니다
