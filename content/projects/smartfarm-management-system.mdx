> **프로젝트 개요**
>
> 사내 PHP 레거시 스택과 HMI 제어기 조직 문화라는 제약 속에서, MQTT 기술을 홀로 독학하여 스마트팜 농가 고객과 사내 운영 인원 모두를 위한 전용 통합 관리 시스템을 바닥부터 설계·구현했습니다. MQTT 수집 전용 마이크로서비스와 웹 API 서버를 물리적으로 분리하고, 수집→1분 배치 저장→WebSocket 실시간 브로드캐스트 3단계 데이터 파이프라인을 구성했습니다. 메시지 처리 인터페이스 기반 팩토리 패턴으로 이기종 HMI 메시지 타입을 OCP 기반으로 분리하고, 실운영에서 마주칠 4가지 아키텍처 리스크를 설계 단계에서 선제적으로 식별하고 구조적으로 차단했습니다.
>
> **프로젝트 성격**: PHP 레거시 문화라는 조직적 제약 속에서 MQTT를 독학으로 안착시키고, 단순 구현을 넘어 실서비스 운영 시 마주칠 4가지 결합도 리스크를 데이터 파이프라인 설계와 아키텍처 분리로 선제 방어한 주도적 MSA 풀스택 프로젝트

## 문제 상황 & 해결

### 1. MQTT Retain 메시지로 인한 과거 센서 데이터 오표시 현상 제거

**현상**

MQTT 브로커의 Retain 설정이 활성화된 상태에서, 브로커는 토픽별 마지막 발행 메시지를 보존했다가 신규 구독자가 연결될 때 즉시 재전송한다. 웹 화면 최초 진입 시 수 초~수 분 전의 과거 누적 데이터가 최신 센서값인 것처럼 대시보드에 표출되는 현상이 발생했다.

| 구분 | Retain 대비 없는 경우 | 타임스탬프 검증 적용 후 |
|---|---|---|
| 화면 최초 진입 시 표시값 | 과거 보존 수치가 최신값으로 오표시 | 임계값 초과 과거 데이터 브로드캐스트 생략 |
| 이력 데이터 연속성 | 과거 Retain 데이터도 DB에 중복 저장 | 이력은 비동기 배치 저장으로 정상 유지 |
| 모니터링 신뢰도 | 실시간 수치 왜곡으로 운영 판단 오류 위험 | 발행 시각 기반 검증으로 실시간성 보장 |

**원인**:
- MQTT 프로토콜의 Retain 메커니즘은 신규 구독자에게 마지막 메시지를 즉시 전달하는 설계로, 데이터 수신 시점과 발행 시점 사이의 시간 차를 별도로 검증하지 않으면 과거 값이 현재 값처럼 처리된다

**해결 과정**:

수신된 HMI 메시지에 포함된 장치 발행 타임스탬프와 서버 수신 현재 시각을 비교하는 임계값 검증 로직을 센서 데이터 프로세서에 추가했다.

| 조건 | 처리 방식 | 브로드캐스트 여부 |
|---|---|---|
| 발행 시각과 수신 시각의 차이 ≤ 임계값 | 현재 데이터로 판단 → 즉시 브로드캐스트 + DB 저장 | 수행 |
| 발행 시각과 수신 시각의 차이 > 임계값 | 과거 Retain 데이터로 판단 → DB 비동기 배치 저장만 수행 | 생략 |

**결과**:
- 웹 화면 진입 직후 과거 Retain 데이터가 최신값처럼 표시되는 현상이 제거됐다
- 과거 데이터는 이력으로 조용히 저장되므로 센서 데이터 이력의 연속성도 정상 유지된다

### 2. HMI 원시 수치 포맷 직접 노출 방어 및 도메인 추상화 레이어 설계

**현상**

HMI 기기 통신 규격에 맞춰 설계된 원시 수치 포맷(릴레이 동작 모드 코드값, 온도 설정 정수 직접 입력 등)이 여과 없이 관리 화면에 그대로 노출됐다. HMI 매뉴얼 없이는 어떤 단위인지, 어느 범위가 유효한지 파악할 수 없었으며, 잘못된 수치가 제어 명령으로 그대로 발행되면 장치 오동작 위험이 있었다.

| 구분 | HMI 원시 수치 직접 노출 | 도메인 추상화 레이어 적용 |
|---|---|---|
| 운영자 이해도 | HMI 매뉴얼 없이 의미 파악 불가 | 운영자 언어 설명 문구로 직관적 파악 가능 |
| 잘못된 입력 가능성 | 범위 초과 수치 입력 후 그대로 발행 | 이중 검증으로 범위 외 수치 입력 단계에서 차단 |
| 오동작 리스크 | 잘못된 제어 명령 발행 구조적으로 허용 | 허용 범위 내 수치만 제어 명령으로 발행 |

**원인**:
- HMI 통신 규격에 맞춘 설계가 운영자 도메인 언어와 일치하지 않는 구조로, 기기 전문 지식을 운영자가 내재화하지 않으면 올바른 값을 입력할 수 없는 UX였다

**해결 과정**:

3가지 방어 장치를 설계·적용했다.

| 방어 계층 | 구현 내용 | 목적 |
|---|---|---|
| 입력 단위 검증 (이중) | Spring Validation 서버 검증 + 프론트엔드 클라이언트 검증 | 허용 범위 벗어난 수치 입력 단계에서 차단 |
| 운영자 언어 가이드 문구 | 각 설정 항목에 HMI 매뉴얼 없이도 이해 가능한 툴팁 배치 | HMI 전문 지식 없이도 항목 의미 파악 가능 |
| 안전 기본값 사전 설정 | 농작물 유형별 표준 운영 기준치를 기본값으로 사전 설정 | 입력 없이도 안전한 기준 상태로 유지 |

**결과**:
- HMI 장치 전문 지식이 없는 현장 운영자도 설정 화면을 오류 없이 조작할 수 있게 됐다
- 범위 외 수치로 인한 잘못된 제어 명령 발행 리스크가 구조적으로 차단됐다

### 3. 장치-농장 조인 쿼리의 N+1 성능 저하 해소

**현상**

관리자 대시보드에서 특정 농장의 장치 목록을 조회할 때, JPA가 장치 엔티티 리스트를 먼저 가져온 뒤 각 장치가 속한 농장 마스터 정보를 건마다 별도 SELECT 쿼리로 추가 조회하는 N+1 문제가 발생했다. 장치 수가 늘어날수록 쿼리 수가 선형으로 증가하여 대시보드 응답 시간이 눈에 띄게 지연됐다.

| 구분 | JPA 기본 지연 로딩 (N+1 발생) | JPQL Fetch Join 명시적 선언 |
|---|---|---|
| 실행 쿼리 수 (장치 N개) | 1 + N회 (장치 목록 1 + 농장 마스터 N) | 1회 (단일 조인 쿼리) |
| 농장 규모 확장 시 응답 성능 | 장치 수에 비례하여 선형 저하 | 농장 규모와 무관하게 고정 성능 유지 |
| 장애 격리 | 쿼리 수 폭증으로 DB 부하 집중 | 쿼리 수 고정으로 부하 예측 가능 |

**원인**:
- JPA 연관 엔티티 조회의 기본 지연 로딩 전략은 실제 접근 시점에 별도 쿼리를 발행하므로, 리스트 조회 후 각 엔티티의 연관 객체에 접근하는 패턴에서 N+1 문제가 구조적으로 발생한다

**해결 과정**:

JPQL Fetch Join으로 장치 엔티티와 농장 마스터를 단일 쿼리로 조인 조회하도록 리포지토리 레이어를 수정했다.

```java
@Query("SELECT d FROM Device d JOIN FETCH d.farm WHERE d.farm.id = :farmId")
List<Device> findAllByFarmWithFetch(@Param("farmId") Long farmId);
```

**결과**:
- 장치 N개 조회 시 N+1번 발생하던 쿼리가 단일 조인 쿼리 1번으로 축소됐다
- 대시보드 초기 로딩 응답 속도가 개선되어 농장 규모 확장과 무관하게 일정한 응답 성능을 확보했다

### 4. 백엔드 재기동 시 알림 발송 상태 초기화 및 SMS 중복 발송 차단

**현상**

초기 구조에서는 SMS 알림 발송 여부를 메모리 내 상태로 관리했다. 임계값 초과 이후 서버를 재기동하면 발송 상태가 초기화되어, 동일 센서 이상 상황에 대해 이미 발송된 알림이 재발송되는 문제가 발생했다. 불필요한 Aligo SMS 비용과 수신자 알림 피로도 양면에서 운영 리스크였다.

| 구분 | 인메모리 상태 관리 | DB 영속화 구조 (선택) |
|---|---|---|
| 서버 재기동 후 상태 | 발송 이력 초기화 → 동일 이상에 재발송 | DB 이력 기반 복원 → 중복 발송 차단 |
| 쿨다운 판단 기준 | 메모리 내 타임스탬프 (재기동 시 소실) | DB에 저장된 쿨다운 만료 시각 교차 검증 |
| 런타임 환경 의존성 | 프로세스 생명주기에 상태 종속 | 프로세스 재기동과 완전히 독립 |

**원인**:
- 알림 발송 상태를 런타임 메모리에만 보관하면 서버 재기동·장애 복구 시 상태가 소실되어, 동일 조건에 대해 알림이 중복 발송되는 것을 막을 수 없는 구조였다

**해결 과정**:

SMS 발송 이력을 DB 테이블에 명시적으로 영속화하는 구조로 전환했다. 알림 발송 시점에 발송 시각·대상 센서·쿨다운 만료 시각을 DB에 기록하고, 임계값 초과가 감지될 때마다 DB에서 최근 발송 이력을 조회하여 쿨다운 기간 내 재발송 여부를 동적으로 교차 검증하도록 설계했다.

**결과**:
- 서버 재기동 여부와 무관하게 DB 이력을 기반으로 상태를 복원하여 알림 중복 발송이 방지됐다
- SMS 발송 비용 절감 및 알림 신뢰도 향상 효과를 확보했다

## 기술적 판단 & 설계 포인트

### 1. 웹 API 서버 & MQTT 수집 서비스 물리 분리 — 고빈도 수집 부하로부터 제어 응답 격리

레거시 사이트에서 MQTT 데이터가 폭발적으로 수집될 때 웹 제어 API 응답이 후순위로 밀리는 성능 병목이 실운영 중 반복됐다. 원인은 수집·저장·브로드캐스트라는 고부하 작업과 제어·조회·설정이라는 저지연 작업이 동일 프로세스 내에서 경쟁하는 단일 서비스 구조였다.

**결정**: MQTT 수집 전담 마이크로서비스와 웹 API 서버를 물리적으로 분리하여, 각 서비스가 독립적인 책임 영역에만 집중하는 MSA 구조를 설계했다.

```
[Frontend — Next.js 15.5]
 REST API 요청 (농장·장치 조회, 제어, 설정)    WebSocket/STOMP 실시간 구독
        │ HTTP/REST                                        │
        ↓                                                   │
[백엔드 웹 API 서버]        ─── 내부 제어 명령 위임 ───→   │
 농장·사용자·장치·SMS 관리                               ↓
                                          [MQTT 수집 전용 마이크로서비스]
                    │                      HMI 메시지 수신·분기·처리
                    │ SQL                  DB 배치 저장 / SMS 임계값 모니터링
                    │                      WebSocket 실시간 브로드캐스트
                    └──────────────────────────────┘
                                    ↓ SQL
                              [PostgreSQL]
                                                    ↕ MQTT
                                              [MQTT 브로커]
                                                    ↕ 구독·발행
                                          [HMI 엣지 장치 — 온습도·CO2·pH·EC 센서]
```

| 항목 | 단일 통합 서비스 | 마이크로서비스 물리 분리 (선택) |
|---|---|---|
| 고빈도 수집 시 제어 API 응답 | 수집 부하에 의해 제어 응답 후순위로 밀림 | 수집 서비스 독립 처리 — 제어 API 응답 안정성 유지 |
| 장애 격리 | 수집 이상 시 전체 API 서버 장애 전파 | 서비스 경계에서 장애 격리 가능 |
| 독립 확장성 | 수집 부하 증가 시 전체 서버 스케일업 필요 | 수집 서비스만 독립 확장 가능 |

> 웹 API 서버는 순수하게 제어·조회·설정 역할에만 집중하고, 수집·저장·브로드캐스트의 고부하 작업은 MQTT 수집 전용 서비스가 독립적으로 처리한다. 레거시에서 반복되던 병목 구조가 구조적으로 해소됐다.

### 2. MessageProcessor 팩토리 패턴 — 이기종 HMI 메시지 타입별 OCP 기반 확장 구조

HMI 엣지 장치는 메시지 타입에 따라 서로 다른 토픽 구조와 페이로드 포맷으로 데이터를 발행한다.

| MQTT 토픽 패턴 | 발행 데이터 유형 |
|---|---|
| `v1/farm/+/device/+/data` | 센서 측정값 (온도·습도·CO2·pH·EC) |
| `v1/farm/+/device/+/data/relays` | 릴레이 전용 상태 데이터 |
| `v1/farm/+/device/+/control/result` | 제어 명령 처리 결과 |
| `v1/farm/+/device/+/status` | 장치 온라인/오프라인 상태 |

이를 단일 핸들러에서 분기 처리하면 메시지 타입이 추가될 때마다 기존 분기 로직 전체를 수정해야 하는 개방-폐쇄 원칙(OCP) 위반 구조가 된다.

**결정**: 메시지 처리 전담 인터페이스(MessageProcessor Interface)를 정의하고, 각 타입별 전담 프로세서가 이를 구현하는 팩토리 패턴을 설계했다.

```
[MQTT 수신 클라이언트]
        │ 수신 콜백
        ↓
[메시지 타입 분기 디스패처 — 토픽 파싱 → 메시지 타입 식별]
        ↓
┌──────────────────────────────────────────────┐
│  타입별 전담 메시지 프로세서                    │
│  • 센서 데이터 프로세서 (수집·저장·임계값)      │
│  • 릴레이 상태 프로세서 (릴레이 이벤트 기록)   │
│  • 제어 결과 프로세서 (제어 명령 결과 로깅)    │
│  • 장치 상태 프로세서 (온·오프라인 상태 관리)  │
└──────────────────────────────────────────────┘
```

| 항목 | 단일 핸들러 분기 방식 | MessageProcessor 팩토리 패턴 (선택) |
|---|---|---|
| 신규 메시지 타입 추가 시 | 기존 분기 로직 수정 필요 — OCP 위반 | 신규 전담 프로세서만 추가 — 기존 코드 변경 없음 |
| 타입별 로직 결합도 | 모든 타입 로직이 하나의 핸들러에 집중 | 각 타입 프로세서가 단일 책임으로 분리 |
| 유지보수성 | 타입 증가에 따라 핸들러 복잡도 선형 증가 | 타입 증가와 무관하게 기존 프로세서 단순 유지 |

> 새로운 HMI 메시지 타입이 추가되더라도 기존 프로세서는 수정 없이 새 전담 프로세서만 추가하면 되는 개방-폐쇄 원칙 기반 확장 구조를 확보했다.

### 3. 수집 → 1분 배치 저장 → WebSocket 실시간 브로드캐스트 3단계 파이프라인

온도·습도·CO2 등 센서는 초 단위로 메시지를 발행한다. 수신할 때마다 DB에 INSERT 쿼리를 실행하면 분당 수십~수백 건의 쿼리가 발생하여 DB 부하가 급증한다. 반면 WebSocket 실시간성은 포기할 수 없다.

**결정**: 수집·저장·브로드캐스트를 독립된 3단계 파이프라인으로 분리하여, DB 성능 오버헤드 최소화와 실시간성 보장을 동시에 달성했다.

```
[MQTT 메시지 수신]
        │
        ↓
[센서 데이터 프로세서]
    ├── [1] DB 배치 저장 서비스
    │       └── 1분 단위 버퍼에 축적 → 일괄 INSERT (이력 데이터 기록 비용 최소화)
    ├── [2] 임계값 모니터링 서비스
    │       └── 센서 값이 설정 임계 범위 초과 시 → Aligo SMS 알림 발송
    └── [3] WebSocket 브로드캐스트 서비스
            └── /topic/dashboard/farm/{farmId} 토픽으로 즉시 Push
                → Frontend @stomp/stompjs 구독 → Chart.js 실시간 갱신
```

| 항목 | 수신마다 DB 저장 방식 | 1분 배치 저장 방식 (선택) |
|---|---|---|
| DB INSERT 쿼리 수 | 수신 건수만큼 발생 (초 단위 폭증) | 1분 단위 묶음 처리 (쿼리 수 대폭 감소) |
| 실시간 브로드캐스트 | 저장과 동일 경로 — DB 부하 영향을 받을 수 있음 | 브로드캐스트 경로와 저장 경로 완전 분리 |
| 이력 데이터 연속성 | 유지 | 유지 (배치 버퍼 축적 후 일괄 저장) |

> DB 성능 오버헤드와 실시간성은 트레이드오프 관계가 아니다. 저장 경로와 브로드캐스트 경로를 분리하면 두 요구사항을 독립적으로 충족할 수 있다.

### 4. 안전 기본값 프리셋 발급 아키텍처 — USER 제어권 보장과 오조작 리스크 동시 방어

농가 고객(USER 역할)에게 SMS 임계값 설정과 실시간 장치 원격 제어 권한을 전면 개방하는 것이 안전한가에 대한 의문이 설계 초기에 제기됐다. 잘못된 수치 한 번이 농작물 환경 전체에 영향을 미칠 수 있기 때문이었다. 그러나 지역별 세부 기후 기준값이 농가마다 상이하므로 농가 고객에게 제어권을 보장해야 한다는 비즈니스 의사결정을 수용했다.

**결정**: USER에게 제어권을 전면 개방하되, 계정 발급 시 백엔드가 농작물 유형별 안전 기본값 프리셋을 미리 설정한 상태로 계정을 생성하는 이중 안전장치 아키텍처를 도입했다.

| 역할 | 접근 범위 | 계정 발급 주체 |
|---|---|---|
| ADMIN | 사용자·농장 생성·삭제, 전체 데이터 조회 (사내 개발·관리 인원) | 수동 발급 (최상위 권한) |
| MANAGER | 장치 제어, SMS 임계값·수신자 설정 (사내 현업 운영 인원) | ADMIN 수동 발급 |
| USER | 자신의 농장 대시보드·센서 조회, 장치 제어, SMS 임계값 설정 (농가 고객) | ADMIN/MANAGER 수동 발급 |

> USER에게 제어 권한을 보장하면서도, 발급 시점에 안전 기본값 프리셋이 이미 설정된 상태로 계정이 생성되므로 입력 없이도 안전한 기준 상태가 유지된다. 잘못된 설정으로 인한 장치 오동작 리스크를 아키텍처 레이어에서 선제 방어했다.

### 5. 폐쇄형 회원 관리 — MQTT 내부 식별 코드 주입의 구조적 필연성

웹 화면을 통한 일반 회원가입(자체 계정 생성)을 차단하고 사내 인원만이 농가 계정을 수동 발급할 수 있도록 설계했다. 이유는 단순한 보안 정책이 아닌 구조적 필연이었다.

**결정 근거**: HMI 기기와 백엔드 서버 간의 MQTT 통신은 사내 전용 내부 식별 코드(농장·장치 매핑 정보)가 계정 생성 시점에 반드시 함께 주입되어야만 한다. 이 식별 코드 없이 생성된 계정은 어떤 HMI 기기와도 연동되지 않는다.

| 항목 | 개방형 회원가입 허용 | 폐쇄형 수동 발급 (선택) |
|---|---|---|
| 계정-기기 연동 무결성 | 식별 코드 미주입 계정 생성 → 기기 연동 불가 계정 양산 | 발급 시 식별 코드 동시 주입 → 연동 무결성 보장 |
| 보안 리스크 | 임의 계정이 HMI 데이터 접근 시도 가능 | 사내 인원 발급으로 접근 주체 통제 |
| 운영 복잡도 | 연동 불가 계정 정리 운영 부담 | 발급 시점에 연동 완결 — 사후 정리 불필요 |

> 개방형 회원가입을 허용하는 순간 계정과 기기 간의 연동 무결성이 완전히 파괴된다. 폐쇄형 수동 발급은 보안 정책이 아닌 MQTT 아키텍처 구조가 요구하는 필연적 설계다.

## 회고

### 잘한 점
- **MSA 물리 분리 아키텍처 선제 설계**: 레거시에서 반복되던 단일 서비스 병목 패턴을 직접 목격하고, 이 프로젝트에서는 처음부터 수집과 제어를 물리 분리하는 구조로 설계했다. 레거시의 실패 패턴을 신규 시스템 설계의 아키텍처 판단 기준으로 직접 전환한 결정이었다
- **4대 실운영 리스크 선제 식별 및 구조적 차단**: Retain 데이터 오표시·원시 수치 노출·N+1 성능 저하·SMS 중복 발송이라는 4가지 운영 리스크를 실운영 전 단계에서 식별하고 구조적으로 방어했다. 기능이 동작하는 것에 그치지 않고 실서비스 운영 시나리오를 설계 단계에서 시뮬레이션한 것이 핵심 판단이었다
- **폐쇄형 회원 관리의 엔지니어링적 근거 설계**: MQTT 내부 식별 코드 주입이라는 시스템 구조적 필연성을 파악하고, 이를 보안 정책이 아닌 아키텍처 요구사항으로 설계에 반영했다. 비즈니스 요구와 기술 구조의 교차점을 정확히 짚어낸 설계였다

### 아쉬운 점
- HMI 기기에 제어 코드를 심는 초기 설치와 오류 수정이 모두 오프라인 현장 출장에 의존해야 하는 물리적 제약이 실운영 확산의 가장 큰 병목이다. 설치 이후 기기 오류가 발생해도 원격 수정이 불가능하여 재출장을 감수해야 하는 구조에서, OTA(Over-The-Air) 방식의 원격 업데이트나 CI/CD 파이프라인 연동 인프라를 처음부터 아키텍처에 포함했다면 운영 비용을 구조적으로 낮출 수 있었다. HMI 기기 레이어의 원격 관리 가능성을 선제적으로 설계해 두는 것이 다음 유사 프로젝트에서 반드시 반영해야 할 교훈이다
- WebSocket 재연결 후 단절 구간의 센서 이력을 보완 조회하는 갭 리커버리 로직이 구현되지 않았다. 재연결 즉시 REST로 누락 구간의 이력 데이터를 보완 조회하는 구조를 추가했다면 모니터링 연속성이 더욱 강해졌을 것이다

### 배운 점
- **레거시 기술  문화라는 조직적 환경에서도, 도메인 외부에서 검증된 기술을 독학으로 안착시키는 것이 주도적 엔지니어의 역할이다**: MQTT를 사내에서 제대로 다뤄본 사람이 없는 환경에서 처음부터 끝까지 독학으로 구현을 완수했다. 조직이 익숙한 것과 시스템에 올바른 것이 다를 때, 스스로 기술 기반을 확장하고 설계 결정에 책임을 지는 것이 주도적 엔지니어로 성장하는 경로임을 이 프로젝트에서 직접 체득했다
- **단순 구현과 실운영 방어는 다른 차원의 엔지니어링이다**: 기능이 동작하는 것은 실운영 기반을 확보하는 것과 다르다. Retain 오표시·원시 수치 노출·N+1 저하·SMS 중복 발송이라는 4가지 리스크는 코드가 기능적으로 정상 동작하는 상태에서도 발생할 수 있는 운영 레이어의 문제였다. 실서비스 운영 시나리오를 설계 단계에서 시뮬레이션하고 선제 방어하는 것이 엔지니어링의 완성이라는 인식을, 이 프로젝트를 통해 처음으로 명확하게 확립했다
