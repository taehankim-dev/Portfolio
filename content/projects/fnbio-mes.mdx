> **프로젝트 개요**
>
> Spring Boot 3.2.4 + Vue 3.4 기반으로 에프엔바이오 식품 제조 현장의 원부자재 입출고·칭량·생산 실적 전 공정을 전산화한 풀스택 제조실행시스템(MES)입니다. 2인 협업 체계에서 동료가 현장 태블릿 UI와 LOT 추적 모듈에 집중하는 동안, 시스템 비즈니스 무결성을 결정짓는 FIFO 연산·BOM 수율 계산·트랜잭션 제어·IsPrint 도메인 제약 등 백엔드 코어 로직 전 구간과 PC 기반 통합 관리 화면 전반을 단독으로 설계·구현했습니다.
>
> **프로젝트 성격**: 이전 프로젝트에서 체득한 기술 병목(Apache POI 서버 부하, JPA/MyBatis 역할 혼용)을 설계 단계에서 선제 배제하고, 제조 현장의 LOT 추적성과 품질 문서 무결성을 시스템 레벨에서 구조적으로 확보한 백엔드 중심 풀스택 MES

## 문제 상황 & 해결

### 1. 동일 원료 복수 LOT 출고 시 칭량 기록서 데이터 불일치 — 클라이언트 사이드 다중 LOT 그룹화 파싱

**현상**

하나의 생산 실적에서 동일 원료가 재고 부족으로 두 개 이상의 LOT에서 나뉘어 출고되는 경우가 있다. API는 출고 건별(LOT 단위)로 데이터를 반환하기 때문에, 동일 원료가 두 행으로 응답되고 이를 그대로 렌더링하면 칭량 기록서 원료 테이블에 같은 원료명이 중복으로 출력됐다.

| 렌더링 상태 | 원료명 | LOT 번호 |
|---|---|---|
| 중복 렌더링 (수정 전) | 원료 A | LOT-001 |
| 중복 렌더링 (수정 전) | 원료 A | LOT-002 |
| 품질 양식 기준 (목표) | 원료 A | LOT-001, LOT-002 |

칭량 기록서 제조·품질 관리 기준 양식에서는 동일 원료명은 단 한 행으로 표시하고, 해당 행 내부에 사용된 복수의 LOT 번호가 나열되어 묶여 있어야 한다.

**원인**:
- API 설계상 출고 이력은 입고(LOT) 단위로 반환되는 것이 정상이다. 각 LOT의 개별 사용량도 기록서에 필요하므로 백엔드 집계보다 프론트엔드에서 '원료 단위 행 + LOT 리스트' 이중 구조로 가공하는 방식이 더 적합했다.
- 화면 렌더링 레이어에서 원료 단위 재집계 처리가 빠져 있었던 것이 근본 원인이었다.

**해결 과정**:

*그룹화 알고리즘*: API 응답 배열을 이중 순회하며 원료 코드를 기준으로 동일 원료 항목을 탐색한다. 동일 원료·동일 LOT이면 사용량을 누적 합산하고, 동일 원료·다른 LOT이면 해당 LOT 번호를 별도 리스트에 추가한다. 처리 완료된 항목은 중복 처리 방지 플래그로 마킹하여 재방문을 차단한다. 결과 배열에는 원료별 단일 행만 남고, 각 행의 LOT 리스트 필드에 사용된 전체 LOT 번호가 담기는 구조로 변환된다.

*타입 시스템*: 칭량 기록서 도메인에 맞춰 TypeScript 타입을 별도로 설계했다. 칭량 기록 정보 타입과 칭량 원료 정보 타입을 분리 선언하고, 원료 정보 타입의 LOT 리스트 필드를 배열 타입으로 명시하여 그룹화 결과가 타입 안전하게 바인딩되도록 구성했다.

**결과**:
- 칭량 기록서 원료 테이블이 '원료 단위 1행 + 복수 LOT' 형태로 정확히 렌더링됐다.
- 동일 원료 중복 표시 문제가 해소되어 출력된 기록서가 제조·품질 관리 기준 양식에 부합하게 됐다.
- LOT 리스트 타입이 명시되어 이후 기능 확장 시 타입 오류를 컴파일 단계에서 잡을 수 있는 구조가 됐다.

### 2. 브라우저 인쇄 레이아웃 오염 — DOM 복사·격리 기반 순수 JavaScript 인쇄 제어

**현상**

칭량 기록서 상세 팝업에서 `window.print()`를 호출하면, 인쇄 대상이 팝업 내부 양식으로 한정되지 않고 배경 화면의 AG-Grid 테이블·사이드 메뉴·헤더 등 전체 화면 요소가 함께 인쇄됐다. 품질 기준 양식과 다른 레이아웃으로 출력되어 실무 사용이 불가능한 상태였다.

| 방식 | 문제점 |
|---|---|
| CSS `@media print` 요소 숨김 | 팝업 z-index·포지셔닝 충돌로 부작용 발생, UI 변경마다 선택자 유지보수 필요 |
| DOM 복사·격리 | 인쇄 대상 외 요소를 DOM 레벨에서 완전 배제, 단일 함수 관리 포인트 |

**원인**:
- `window.print()`는 현재 DOM 전체를 인쇄 대상으로 삼는다. CSS `@media print` 방식으로 불필요한 요소를 숨기면, 팝업 레이어 구조에서 z-index·포지셔닝 충돌로 의도하지 않은 요소가 인쇄 영역에 잔류하는 부작용이 발생했다.
- 선택자 기반 관리 방식은 UI 변경이 생길 때마다 인쇄 로직을 함께 수정해야 하는 유지보수 비용이 구조적으로 발생했다.

**해결 과정**:

*DOM 복사·격리*: 인쇄 대상 칭량 기록서 양식 DOM을 `cloneNode(true)`로 독립 복사한다. 루트 앱 컨테이너를 `display: none`으로 임시 숨김 처리하고, 복사된 DOM만 담은 인쇄 전용 컨테이너를 `body`에 직접 마운트한다. `window.print()`를 호출하고, 인쇄 완료 후 `setTimeout`으로 인쇄 컨테이너를 제거한 뒤 앱 컨테이너를 `display: block`으로 복원한다. 이 일련의 흐름을 단일 격리 함수로 캡슐화하여 호출 측이 내부 구현 세부사항에 의존하지 않도록 구성했다.

*IsPrint 연동*: 인쇄 완료 후 백엔드 API를 호출하여 해당 기록서의 출력 완료 플래그를 갱신하고, 프론트엔드에서 모든 입력 필드를 즉시 비활성화 처리한다. 인쇄 행위와 잠금 처리가 하나의 흐름으로 연결되어 출력 후 수정 차단이 자동으로 완결된다.

**결과**:
- 인쇄 시 칭량 기록서 양식만 출력되고 나머지 화면 요소가 DOM 레벨에서 완전히 배제됐다.
- CSS 선택자 방식 대비 관리 포인트가 단일 함수로 집중되어, 레이아웃 변경 시 인쇄 로직을 따로 수정할 필요가 없는 구조가 됐다.
- 인쇄 완료와 IsPrint 잠금이 단일 흐름으로 연결되어 출력된 기록서의 수정 차단이 자동으로 완결됐다.

## 기술적 판단 & 설계 포인트

### 1. JPA + MyBatis 하이브리드 전략 — 연산 유형별 역할 분리

MES는 특성이 다른 두 종류의 DB 연산이 공존한다. 마스터 데이터 등록·엔티티 상태 갱신처럼 단일 테이블 중심의 생명주기 조작이 있고, LOT 이력 추적·입출고 현황·생산 통계 집계처럼 다중 테이블 조인과 복잡한 동적 조건이 필요한 연산이 있다. 모든 쿼리를 JPA 하나로 처리하면 복잡한 통계 화면에서 N+1 문제나 Fetch Join 한계에 부딪히고, MyBatis 하나로 처리하면 간단한 엔티티 상태 갱신조차 XML Mapper에서 수동 관리해야 하는 비효율이 생긴다.

| 영역 | 선택 기술 | 이유 |
|---|---|---|
| 단순 엔티티 CRUD / 상태 갱신 | JPA Repository | @Transactional 내 안전한 엔티티 생명주기 관리, 반복성 낮은 CRUD에 최적 |
| 다중 조인 현황·통계·LOT 역방향 이력 추적 | MyBatis XML Mapper | ResultMap 기반 복잡 매핑, 동적 SQL 조건 제어, N+1 구조적 회피 |

JPA로 트랜잭션 내 엔티티 생명주기를 안전하게 관리하면서, 복잡한 집계·현황 화면에서는 MyBatis의 ResultMap과 동적 SQL로 쿼리를 직접 제어했다. 성능 병목이 발생하기 쉬운 통계·이력 추적 화면에서 N+1 문제를 구조적으로 회피하면서, JPA 영속성 컨텍스트가 주는 트랜잭션 안전성도 동시에 확보했다.

### 2. 출력 플래그(IsPrint) 기반 품질 문서 변조 방지 — 백엔드 도메인 제약 + 프론트엔드 UI 잠금 이중 구조

칭량 기록서는 식품 제조 현장의 제조·품질 관리 기준 문서다. 출력 후 내용이 수정되면 실제 제조 기록과 시스템 기록이 불일치하게 되며, 품질 감사 시 치명적인 문제가 된다. 단순히 UI에서 버튼을 숨기는 방식은 API 직접 호출로 우회할 수 있어 실질적인 방어가 되지 않는다고 판단했다.

| 방어 레이어 | 구현 방식 | 방어 대상 |
|---|---|---|
| 백엔드 도메인 제약 (1차) | 출력 완료 상태 확인 → 비즈니스 예외 발생 → DB 저장 원천 차단 | API 직접 호출 및 악의적 우회 |
| 프론트엔드 UI 잠금 (2차) | 출력 후 전체 입력 필드 즉시 비활성화 | 일반 사용자의 화면 내 수정 시도 |

출력 행위를 '되돌릴 수 없는 상태 전환'으로 모델링했다. 백엔드 비즈니스 서비스에서 저장 요청이 들어올 때 출력 완료 여부를 먼저 확인하고, 출력 완료 상태이면 비즈니스 예외를 발생시켜 DB 저장을 원천 차단한다. 프론트엔드에서는 출력 확인 버튼 클릭 시 플래그를 즉시 갱신하고 모든 입력 필드를 비활성화하여, 사용자가 화면에서 수정 시도 자체를 할 수 없도록 한다.

> 백엔드 예외 처리로 악의적 API 직접 호출도 차단되고, 프론트엔드 UI 잠금으로 일반 사용자의 실수도 방지된다. 출력된 기록서와 DB 기록이 항상 동일한 내용임을 보장하는 구조다.

### 3. PostgreSQL 부분 인덱스(Partial Index) — MES 데이터 누적 특성을 반영한 FIFO 성능 설계

MES는 운영 기간이 늘어날수록 입출고 이력 데이터가 급격히 누적된다. FIFO 자동 출고 로직의 핵심 연산은 '현재 재고가 남아 있는 입고 이력'을 입고일 오름차순으로 조회하는 것인데, 시간이 지날수록 전체 이력에서 재고가 소진된 행이 대다수를 차지하게 된다. 단순 B-Tree 인덱스로는 이 소진 행들을 대량으로 불필요하게 스캔하는 구조가 된다.

| 인덱스 유형 | 적용 컬럼 | 최적화 목적 |
|---|---|---|
| B-Tree 인덱스 | LOT 번호 컬럼 | LOT 추적 화면 조회 및 LOT 중복 방지 쿼리 |
| 날짜 범위 인덱스 | 입고일 컬럼 | 기간 필터 기반 현황·통계 화면 쿼리 |
| 부분 인덱스 (핵심) | 원부자재 번호 + 재고 수량 (WHERE stk_qty > 0) | FIFO 출고 시 유효 재고 행만 인덱스에 포함 |

부분 인덱스는 `재고 수량 > 0` 조건을 붙여 현재 유효한 재고 행만 인덱스에 포함시킨다. 재고가 소진되면 해당 행은 자동으로 인덱스에서 제외되므로, 데이터가 누적되어도 FIFO 조회 대상이 실시간 유효 재고 집합으로 고정된다. 일반 인덱스 대비 인덱스 크기 자체가 작고, 재고 소진 처리가 인덱스 갱신 비용에 미치는 영향도 없어 쓰기 성능 저하도 억제된다.

### 4. 세션+쿠키 인증 + 30분 타임아웃 — 내부 폐쇄망 시스템 특성에 맞는 현실적 판단

JWT의 무상태(stateless) 구조가 확장성 측면에서 이론적 장점이 있다는 것은 인지하고 있었다. 그러나 이 시스템은 사내 폐쇄망 내에서 소수의 현장 관리자가 사용하는 내부 전용 시스템으로, 서버 증설 없이 단일 인스턴스로 안정적으로 운영되는 환경이었다. JWT의 장점이 발현되는 조건(다중 서버 분산, 마이크로서비스 간 인증 위임 등)이 이 프로젝트에 해당하지 않았고, 토큰 탈취 대응(블랙리스트 관리) 같은 추가 구현 부담만 생기는 구조였다.

| 구성 요소 | 설계 내용 |
|---|---|
| 세션 유효 시간 | 30분 타임아웃 — 미사용 세션 자동 만료 |
| 활동 감지 연장 | 사용자 활동 중 세션 만료 시각 자동 갱신 — 정상 업무 흐름 방해 없음 |
| 비밀번호 저장 | BCrypt 단방향 해시 — DB 유출 시 평문 노출 방지 |

검증된 세션+쿠키 조합으로 구현 복잡도를 낮추면서, 30분 미활동 자동 로그아웃으로 비인가 접근 위험을 구조적으로 억제했다. 활동 감지 기반 세션 연장 메커니즘으로 실제 업무 중인 사용자의 강제 로그아웃을 방지하면서, 미사용 세션만 정확히 만료시키는 세션 생명주기 제어를 구현했다.

### 5. Apache POI 배제 및 exceljs 클라이언트 전환 — 이전 프로젝트 기술 병목 경험의 선제 반영

이전 프로젝트에서 서버 사이드 엑셀 처리 시 JVM 힙 메모리를 대량 점유하고 응답 지연이 길어지는 병목을 직접 확인했다. 단순 엑셀 업로드 기능 하나가 서버 전체 응답성에 영향을 주는 구조라는 것을 파악하고 있었기에, 이 MES의 설계 단계에서 동일한 병목이 발생하지 않도록 처음부터 방향을 달리 잡았다.

이 MES에서 엑셀 기능이 필요한 맥락을 분석했다. 원부자재·BOM 마스터 데이터의 엑셀 업로드는 초기 데이터 세팅 시기에만 일시적으로 집중되고, 이후에는 낱건 입력이 대부분이었다. 소수의 현장 관리자 환경에서 업로드 빈도도 낮았다.

**결정**: 엑셀 파싱과 다운로드 로직을 모두 프론트엔드의 exceljs 라이브러리로 전환하여 서버 부하를 구조적으로 제거했다. 엑셀 업로드 시 파싱을 브라우저 자원으로 처리하고, 다운로드 역시 클라이언트 단에서 파일을 생성하여 서버 왕복 없이 즉시 내려받는 구조를 구성했다. 서버가 엑셀 처리 연산에 전혀 관여하지 않게 됨으로써, 이전 프로젝트에서 확인한 병목 지점이 이 시스템에서는 구조 설계 단계에서 원천 배제됐다.

## 회고

### 잘한 점
- **코어 비즈니스 무결성 설계**: FIFO 연산·BOM 수율·IsPrint 제약을 단순 기능 구현이 아닌 비즈니스 도메인 제약 조건으로 모델링하여, 시스템이 '현장 규칙을 강제하는 구조'가 되도록 설계했다.
- **이전 경험의 선제 반영**: 이전 프로젝트에서 확인한 Apache POI 병목과 JPA/MyBatis 역할 혼용의 비효율을 이 프로젝트 설계 단계에서 구체적으로 반영했다. 경험에서 얻은 기술 자산이 다음 프로젝트의 아키텍처 결정에 직접 적용된 사례였다.
- **이중 방어 설계**: IsPrint 제약을 백엔드 예외와 프론트엔드 UI 잠금 양쪽에서 독립적으로 구현하여, 어느 한 레이어가 우회되어도 다른 레이어에서 차단하는 구조를 완성했다.
- **성능 설계의 선제성**: MES의 데이터 누적 특성을 사전에 분석하고 PostgreSQL 부분 인덱스를 설계 단계에서 반영하여, 장기 운영 환경에서의 성능 저하를 구조적으로 예방했다.

### 아쉬운 점
- 비즈니스 로직 검증을 로그 출력과 화면 단 수동 E2E 방식으로 진행했다. FIFO 연산이나 수율 계산처럼 계산 규칙이 명확한 도메인 로직은 자동화 단위 테스트가 특히 효과적으로 적용될 수 있는 영역이다. IsPrint 이중 제약처럼 복수 레이어를 포함하는 시나리오도 자동화 통합 테스트로 검증했다면, 수동 테스트에서 놓치기 쉬운 엣지 케이스를 더 안정적으로 커버할 수 있었을 것이다.
- WebSocket(STOMP) 온습도 센서 데이터 브로드캐스트를 구현했으나, 센서 연결이 끊어지거나 이상값이 감지되는 경우의 알림 처리는 프로젝트 기간 내에 구현하지 못했다. 실시간 데이터 파이프라인을 더 완결된 형태로 구성하는 작업이 아쉬운 부분으로 남았다.

### 배운 점
- **도메인 제약은 레이어 독립적이어야 한다.**: UI에서만 막는 것과 백엔드 도메인 레이어에서 막는 것의 실질적 차이를 IsPrint 설계를 통해 체득했다. API 직접 호출 우회가 가능한 환경에서 단일 레이어 방어는 보안 제약이 아니라 UX 제약에 불과하다.
- **부분 인덱스(Partial Index)의 실용성**: 도메인 특성(소진 행 누적)을 분석하여 인덱스 조건을 설계하면, 범용 인덱스보다 실질적으로 더 작고 효율적인 인덱스를 구성할 수 있다는 것을 이 프로젝트에서 처음으로 직접 설계하며 체득했다.
- **기술 경험은 다음 설계의 기준점**: 이전 프로젝트의 병목(Apache POI 서버 부하, JPA/MyBatis 혼용)을 설계 단계에서 선제 배제할 수 있었던 것은, 단순히 기술을 알았던 것이 아니라 해당 기술의 한계를 실제 운영 환경에서 직접 겪었기 때문이다. 개발 경험이 쌓일수록 설계 결정의 근거가 이론이 아닌 실증에 기반하게 된다는 것을 이 프로젝트에서 다시 확인했다.
