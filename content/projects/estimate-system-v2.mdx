> **프로젝트 개요**
>
> 1차 견적 시스템 운영 과정에서 드러난 하드코딩 구조의 유지보수 한계를 극복하기 위해 기획된 2차 풀스택 견적 시스템입니다. 자재 단가·수량 공식·이익률·프리셋 구성을 코드가 아닌 데이터베이스 레코드로 완전히 분리하여, 코드 수정·재배포 없이 영업팀이 관리자 화면에서 직접 견적 기준을 제어할 수 있는 동적 규칙 엔진 구조를 단독으로 설계·구현했습니다. Node.js/Express/TypeScript 기반 Routes-Services-Utils 3계층 백엔드와 Vue 3/Pinia 기반 프론트엔드 전 구간을 담당했으며, 15개 이상의 관계형 테이블 스키마를 설계하고 SQL Injection 방지·트랜잭션 루프·동적 SQL 구성을 외부 라이브러리 없이 서비스 레이어 내부에서 직접 구현했습니다.
>
> **프로젝트 성격**: 1차 시스템의 구조적 한계를 아키텍처 수준에서 해결하기 위해 데이터 주도 규칙 엔진과 레이어드 아키텍처를 설계하고, DB 보안·유연성을 외부 의존 없이 내재화한 2차 풀스택 견적 시스템

## 문제 상황 & 해결

### 1. 기준 면적 보정 규칙 에러 — 자재 특성 무시한 단순 비례 계산의 한계

**현상**

프리셋은 기준 면적(예: 100평)을 기준으로 자재별 수량을 정의한다. 그런데 고객이 150평을 입력하면 자재마다 수량 증가 패턴이 달라야 한다 — 일부 자재는 면적 비례로 수량이 늘어나지만, 다른 자재는 구조물 특성상 고정 수량이거나 면적 구간에 따라 단계적으로 증가하는 패턴을 가진다.

| 자재 유형 | 면적 증감 시 수량 패턴 | 초기 구현의 결과 |
|---|---|---|
| 면적 비례 자재 | 입력 면적에 선형 비례하여 증가 | 비례식으로 처리 가능 — 문제 없음 |
| 단계적 증가 자재 | 특정 면적 구간 초과 시 단계 증가 | 단순 비례식으로 수량 과소 또는 과대 산출 |
| 고정 수량 자재 | 면적 무관 고정 수량 | 비례 계산 적용 시 수량 왜곡 발생 |

초기 구현에서는 이 도메인 차이를 흡수하지 못해, 실제 설치에 필요한 수량과 다른 견적 결과가 산출됐다.

**원인**:
- 프리셋 기준값과 실제 입력값을 동일한 단순 비례식으로 처리하는 구조였다.
- 자재마다 면적 변화에 대한 수량 증감 패턴이 다름에도 이 도메인 차이가 코드에 반영되지 않아 계산이 어긋났다.

**해결 과정**:

자재별 면적 구간과 해당 구간에서의 수량 보정값·이익률 보정값을 정의하는 보정 규칙 테이블을 신설했다. 견적 계산 코어 유틸리티 훅은 실제 입력 면적을 기준으로 해당 자재의 보정 규칙 레코드를 조회하고, 적합한 보정값을 수량 계산에 반영하는 방식으로 흐름을 재설계했다.

| 구분 | 보정 규칙 테이블 신설 전 | 보정 규칙 테이블 신설 후 |
|---|---|---|
| 수량 계산 방식 | 모든 자재에 단일 비례식 일괄 적용 | 자재별 면적 구간 보정 레코드 참조 후 개별 적용 |
| 신규 보정 패턴 대응 | 코어 계산 로직 직접 수정 필요 | 관리자 화면에서 보정 레코드 추가로 완결 |
| 계산 정확도 | 자재 특성 미반영 수량 왜곡 발생 | 자재별 특성에 맞춘 정확한 수량 산출 |

**결과**:
- 기준 면적과 실제 입력 면적의 차이를 자재 특성에 따라 개별적으로 흡수하여 견적 산출 정확도를 확보했다.
- 영업팀이 관리자 화면에서 보정 규칙을 직접 조정할 수 있는 구조가 됐다.

### 2. 항목별 계산 타입 분리 — 도메인 차이를 코드 분기가 아닌 데이터 구조로 흡수

**현상**

자재와 노무비 항목은 계산 방식이 서로 다르다. 면적 비례로 수량이 결정되는 자재, 단수(층수)에 비례하는 노무비, 고정 금액으로 처리되는 기본 공사비처럼 각 항목의 계산 근거가 도메인 차원에서 다르다. 초기 구현에서는 모든 항목을 동일한 계산 흐름 내에서 if-else 분기로 처리하다 보니 조건문이 복잡하게 얽혔고, 신규 항목이 추가될 때마다 기존 분기 로직을 수정해야 했다.

| 계산 타입 | 대표 항목 유형 | 수량 결정 기준 |
|---|---|---|
| 면적 기반 | 재배 Rack, 배관 자재 | 입력 면적 × 단위 수량 (+ 면적 보정 규칙) |
| 단수 기반 | 설치 인건비, 층간 구조 자재 | 입력 단수(층수) × 기준 수량 |
| 고정 수량 | 기본 공사비, 제어 시스템 | 프리셋 정의 고정값 |

**원인**:
- 도메인의 본질적인 차이(계산 타입이 다름)를 코드 분기로 처리하려 한 것이 근본 원인이었다.
- 항목이 늘어날수록 분기가 증가하고, 어느 조건에서 어떤 계산이 적용되는지 추적하기 어려워졌다.

**해결 과정**:

자재 마스터 스키마에 계산 타입 필드(면적 기반, 단수 기반, 고정 수량 등)를 추가하고, 서비스 레이어의 계산 로직이 이 필드값을 읽어 적절한 계산 경로로 라우팅되도록 구조를 변경했다. 새로운 계산 타입이 생기면 자재 데이터에 타입을 지정하고, 서비스 레이어에 해당 타입에 대한 계산 경로만 추가하면 된다.

| 구분 | 계산 타입 분리 전 | 계산 타입 분리 후 |
|---|---|---|
| 계산 로직 위치 | 단일 계산 흐름 내 if-else 분기 집중 | 자재 마스터 `계산 타입` 필드 기반 서비스 라우팅 |
| 신규 항목 추가 | 기존 if-else 분기 직접 수정 필요 | 자재 데이터에 타입 지정 + 서비스에 경로 추가 |
| 코드 가독성 | 분기 증가로 계산 흐름 추적 어려움 | 타입별 독립 계산 경로로 가독성 향상 |

**결과**:
- 계산 로직의 분기 복잡도가 줄어 코드 가독성이 향상됐다.
- 신규 항목 추가 시 기존 로직을 건드리지 않는 구조가 됐다.

## 기술적 판단 & 설계 포인트

### 1. 코드가 아닌 데이터가 제어하는 규칙 엔진 아키텍처

1차 시스템의 핵심 문제는 자재 단가·수량 공식·이익률 기준이 소스 코드에 직접 박혀 있었다는 것이다. 영업팀이 단가를 조정하거나 신규 견적 유형을 추가하려면 반드시 개발자가 개입해야 했고, 매 변경마다 빌드·배포 사이클이 돌아갔다.

**결정**: 모든 계산 기준을 데이터베이스 레코드로 분리하는 '데이터 주도 규칙 엔진' 구조를 채택했다.

| 구성 요소 | 역할 | 변경 방법 |
|---|---|---|
| 자재 마스터 | 항목별 계산 타입 필드 관리 | 관리자 화면에서 타입 필드 수정 |
| 면적 보정 규칙 테이블 | 면적 구간별 수량·이익률 보정값 정의 | 관리자 화면에서 보정 레코드 추가 |
| 프리셋 구성 항목 테이블 | 자재·노무비 항목 조합을 데이터로 정의 | 관리자 화면에서 항목 추가·수정 |
| 항목별 동적 보정 규칙 테이블 | 프리셋 단위의 세부 조정값 제어 | 관리자 화면에서 레코드 조작 |

> 신규 견적 유형 추가, 단가 변경, 항목 구성 조정이 관리자 화면에서 즉시 반영된다. 배포 없이 비즈니스 규칙을 수정할 수 있는 구조가 됐다.

### 2. Routes - Services - Utils 레이어 구조 — 1차 단일 파일의 한계 극복

1차 백엔드는 라우터·DB 쿼리·비즈니스 로직이 단일 파일에 혼재되어, 기능 추가 시 가독성이 급격히 저하됐다. 2차에서는 도메인별로 라우터 파일과 서비스 파일을 쌍으로 분리하고, Routes - Services - Utils 3계층 구조를 엄격히 준수했다.

| 레이어 | 책임 | 1차 시스템 대비 변화 |
|---|---|---|
| Routes (라우터) | HTTP 요청 수신, 필수 필드 검증, 일관된 응답 포맷 반환 | 도메인별 독립 파일로 분리 |
| Services (서비스) | 비즈니스 로직 처리, DB 쿼리 실행, 트랜잭션 관리 | 1차 단일 파일 구조 → 도메인별 독립 서비스 파일 |
| Utils (유틸리티) | 성공·에러 응답 포맷 헬퍼, 공통 검증 함수 | 공통 로직 중앙화로 코드 중복 제거 |

> 자재·프리셋·견적·운영비 등 도메인별 서비스 파일이 독립적으로 관리되어, 한 도메인 수정이 다른 도메인에 영향을 주지 않는 구조가 됐다.

### 3. 외부 도구 없이 DB 보안·유연성을 아키텍처 내부에서 해결

데이터베이스 보안과 쿼리 유연성을 확보하는 방법으로 전용 외부 라이브러리에 위임하는 방법이 있다. 하지만 외부 라이브러리에 의존하면 내부 동작에 대한 제어권이 줄어들고, 보안에 직결되는 영역에서 "라이브러리가 알아서 처리해 주겠지"라는 불투명한 신뢰가 생긴다. 동작 원리가 명확하게 보이는 정직한 구현이 유지보수 측면에서 더 낫다고 판단하여 PostgreSQL 드라이버와 TypeScript 기본 로직만으로 직접 구현하는 방향을 선택했다.

| 구현 결정 | 방법 | 해결한 요구 |
|---|---|---|
| SQL Injection 방지 | $1, $2 파라미터 바인딩을 전 쿼리에 일관 적용 | 입력값이 SQL에 직접 삽입되는 경로 원천 차단 |
| 트랜잭션 루프 구조 | 배열 순회하며 개별 바인딩 파라미터와 함께 순차 INSERT | 배열 마스터 데이터 무결성·보안 동시 확보 |
| 동적 SQL 구성 | WHERE 절 조건 배열 누적 + 바인딩 인덱스 동적 증가 | 어떤 조건 조합에도 파라미터 바인딩 일관 유지 |

> DB 쿼리 흐름에 대한 제어권이 완전히 서비스 레이어 내부에 있고, 외부 의존성에 따른 취약점 노출 지점이 없다.

### 4. DB 트랜잭션으로 견적 저장 무결성 보장

견적 결과를 저장할 때는 견적 기본 정보, 구간 정보, 결과 금액, 항목별 스냅샷, 운영비 스냅샷 등 여러 테이블에 데이터가 동시에 기록된다. 중간에 오류가 발생하면 일부 테이블에만 데이터가 저장된 불완전한 상태가 될 수 있다.

**결정**: 견적 전체 저장 API에서 BEGIN - 다중 INSERT - COMMIT 흐름으로 트랜잭션을 명시적으로 관리했다. 어느 단계에서든 오류가 발생하면 ROLLBACK으로 전 단계를 원상 복구한다.

| 처리 단계 | 내용 | 오류 발생 시 |
|---|---|---|
| BEGIN | 트랜잭션 시작 | - |
| 1단계 INSERT | 견적 기본 정보 저장 | ROLLBACK — 전 단계 원상 복구 |
| 2단계 INSERT | 구간 정보 저장 | ROLLBACK — 전 단계 원상 복구 |
| 3단계 INSERT | 결과 금액 저장 | ROLLBACK — 전 단계 원상 복구 |
| 4단계 INSERT (루프) | 항목별 스냅샷 순차 저장 | ROLLBACK — 전 단계 원상 복구 |
| 5단계 INSERT (루프) | 운영비 스냅샷 순차 저장 | ROLLBACK — 전 단계 원상 복구 |
| COMMIT | 전체 원자적 반영 | - |

> 견적 이력 조회 시 항상 완전한 데이터 세트만 존재하게 됐다. 네트워크 지연이나 DB 오류 발생 시에도 부분 저장 상태로 인한 데이터 오염을 원천 차단했다.

### 5. Pinia 전역 상태 관리 — 복잡한 견적 입력 흐름 제어

견적 입력 과정은 견적 유형 선택 → 구간 정보 입력 → 프리셋·항목 선택 → 계산 결과 확인으로 여러 단계에 걸쳐 상태가 누적된다. 컴포넌트 간 props 전달로 관리하면 구조가 지나치게 복잡해진다.

**결정**: Pinia 스토어에 견적의 전체 라이프사이클 상태를 집중 관리하고, pinia-plugin-persistedstate로 브라우저 스토리지에 자동 영속화했다. 견적 계산 코어 유틸리티 훅은 스토어의 상태를 읽어 구간별 면적·단수 기반 실시간 연산을 수행하고, 결과를 스토어에 다시 반영하는 단방향 흐름으로 설계했다.

| 상태 범주 | 스토어 관리 항목 | 영속화 방식 |
|---|---|---|
| 견적 기본 정보 | 견적 유형, 구간 수, 고객 정보 | pinia-plugin-persistedstate |
| 선택된 항목 | 프리셋, 자재 항목, 연간 운영비 | pinia-plugin-persistedstate |
| 실시간 계산 결과 | 구간별 면적·단수 기반 연산, 전체 합산 금액 | 계산 코어 훅에서 스토어에 직접 반영 |

> 어느 페이지에서든 스토어를 통해 견적 상태를 일관되게 읽고 쓸 수 있게 됐고, 새로고침 후에도 입력 중이던 상태가 그대로 유지된다.

## 결과

### 정량적 성과
- 코드 수정·재배포 없이 자재·프리셋·계산 규칙을 관리자 화면에서 직접 조정할 수 있는 동적 규칙 엔진 구조 완성
- 견적 저장 시 5개 이상 테이블 원자적 처리(BEGIN-COMMIT-ROLLBACK) — 부분 저장 상태로 인한 데이터 오염 사례 제로
- 동적 검색 필터(키워드·날짜·견적 유형 가변 조합) 전 구간 파라미터 바인딩 일관 적용

### 정성적 성과
- 관리자 화면에서 영업팀이 자재 단가, 수량 규칙, 프리셋 구성을 직접 변경할 수 있는 구조 구현
- 인쇄 전용 레이아웃 PDF 출력(html2canvas + jsPDF)과 마스터 데이터 엑셀 다운로드(ExcelJS)로 현장 제출용 산출물과 내부 데이터 관리 두 가지 비즈니스 요구 동시 충족
- 기술적으로 완성된 시스템이었으나 조직 차원의 견적 기준 미표준화로 실사용으로 이어지지 못함 — 도메인 합의가 기술 설계보다 선행되어야 한다는 DDD 본질을 실무에서 체감한 경험

## 회고

### 잘한 점
- **데이터 주도 규칙 엔진**: 1차 시스템에서 직접 경험한 하드코딩 한계를 아키텍처 수준에서 해결했다. 계산 규칙을 코드가 아닌 DB 레코드로 분리하여 배포 없이 비즈니스 규칙을 제어하는 구조를 완성한 것은, 선행 프로젝트 경험을 아키텍처 설계 개선으로 직접 전환한 성장 사이클이었다.
- **DB 보안·유연성 내재화**: SQL Injection 방지(파라미터 바인딩 일관 적용), 트랜잭션 루프 구조, 동적 SQL 구성을 외부 라이브러리 의존 없이 서비스 레이어 내부에서 직접 구현하여, DB 쿼리 흐름에 대한 완전한 가시성을 확보했다.
- **레이어드 아키텍처 전환**: 1차의 단일 파일 구조를 Routes-Services-Utils 3계층으로 전환하여 도메인별 독립성을 확보하고, 한 도메인 수정이 다른 도메인에 영향을 주지 않는 구조를 만들었다.

### 아쉬운 점
- 기술적으로 완성된 동적 규칙 엔진이었지만, 실제 영업팀 운영에는 투입되지 못했다. 근본 원인은 기술 외부에 있었다 — 견적 계산 규칙이 지역·담당자·견적 유형마다 달랐고, 조직 내에서 "어떤 기준이 표준인가"에 대한 합의가 없는 상태였다. 유연한 엔진을 만들었지만, 그 엔진에 올바른 데이터를 채워 넣으려면 "무엇이 올바른 규칙인가"라는 도메인 차원의 표준화가 먼저 선행되어야 했다 — 아무리 유연한 시스템도 비즈니스 표준이 없으면 빈 껍데기가 된다.
- 개발·검증 단계에서 로그 기반 수동 E2E 방식으로 동작을 확인했다. 15개 이상의 테이블이 관계된 복잡한 도메인에서는 핵심 계산 코어 함수에 대한 단위 테스트가 있었다면 보정 규칙 버그를 훨씬 빠르게 잡아낼 수 있었을 것이다.
- 비즈니스 규칙을 DB 레코드로 분리한 것은 아키텍처 측면에서 올바른 결정이었지만, 그 대가로 실사용자 입장의 초기 설정 진입장벽이 높아진 부분을 설계 단계에서 충분히 고려하지 못했다. 15개 이상의 테이블에 걸쳐 자재 마스터·보정 규칙·프리셋 구성 항목·운영비 등을 도메인별로 나눠 입력해야 하는 관리자 화면 흐름은, 시스템 전체 구조를 꿰뚫고 있는 개발자 관점에서는 자연스러운 컨텍스트 전환이었지만 현장 영업 담당자에게는 목적이 불분명한 화면 이동으로 느껴질 수 있었다. 아키텍처의 유연함과 사용자가 감당하는 설정 복잡도 사이의 간극을 설계 초기에 더 명시적으로 다뤘어야 했다.
- 60평형 스마트팜 기준으로 재배 Rack·양액공급기·LED 등 자재 항목과 노무비를 합산하면 최소 50건 이상의 세부 항목을 낱건으로 수동 등록해야 하는 입력 오버헤드가 존재했다. 마스터 데이터 일괄 적용 기능은 백엔드 서비스 레이어 내부에 구현했으나, 세부 항목 구성 자체를 최초 1회 등록할 때의 반복 입력 부담은 해소되지 않았다. 기존에 등록된 유사 프리셋을 복제하여 단가나 항목 구성만 편집하는 프리셋 복제·템플릿 상속 기능을 설계 주안점에 포함했어야 한다는 것을 실제 데이터 입력 과정에서 뒤늦게 인식했다.

### 배운 점
- **도메인 합의가 기술 설계보다 선행되어야 한다**: 이 경험은 도메인 주도 설계(DDD)의 본질을 실무에서 직접 체감한 계기가 됐다. DDD가 말하는 "도메인 전문가와의 언어 공유(Ubiquitous Language)"는 단순한 소통 방법론이 아니라, 시스템이 실제로 작동하기 위한 전제 조건이었다. 기술 아키텍처가 아무리 유연하고 정교하더라도, 도메인 자체가 표준화되지 않으면 그 유연함은 발휘될 기회조차 갖지 못한다. 코드를 짜기 전에 도메인 전문가와의 합의를 구조화된 형태로 확인하는 것이 단순 구현을 넘어서 시스템을 실제로 동작하게 만드는 개발자의 역할임을 이 프로젝트에서 배웠다.
- **선행 프로젝트 경험의 아키텍처 전환**: 1차에서 직접 경험한 하드코딩 구조의 유지보수 한계가 2차 시스템의 데이터 주도 규칙 엔진 설계로 이어졌다. 이전 프로젝트의 한계를 다음 프로젝트의 아키텍처 개선 기준점으로 삼는 성장 사이클을 이 프로젝트에서 처음으로 완전하게 경험했다.
